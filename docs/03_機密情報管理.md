# 3. 機密情報管理（Vault・Secret 自動登録・漏洩防止）

## 機密情報管理の目的
本リポジトリは GitHub 上でコードを公開・共有することを前提としており、API トークン・SSH 秘密鍵・データベースパスワード・Discord Webhook URL などの機密情報を「絶対にリポジトリに平文で残さず」「必要な実行主体には自動的に届ける」運用が必要である。
本章では、機密情報のライフサイクルを 3 層で扱う。

| 層 | ツール | 役割 |
|---|---|---|
| ① 保管 | Ansible Vault | サーバー構成・コンテナ展開で必要な機密を AES256 で暗号化してリポジトリに含める |
| ② 配布 | Terraform GitHub Provider | 各種シークレットを GitHub Actions Secrets として一元自動登録 |
| ③ 漏洩防止 | pre-commit + detect-secrets / git-secrets | コミット前に未暗号化の機密が紛れ込むのを自動検出 |

### この章で扱うファイル
```text
infra
├── .pre-commit-config.yaml *           # ③ 漏洩防止フック定義
├── .secrets.baseline *                 # ③ detect-secrets が許容する既知パターン
├── iac-workspace
│   ├── ansible
│   │   ├── group_vars
│   │   │   └── all
│   │   │       └── vault.yml.example   # ① Vault のテンプレート（リポジトリ同梱の参考用）
│   │   └── roles
│   │       └── web
│   │           └── tasks/main.yml      # ① key.pem を Vault 経由で配置するタスク
│   ├── local_config                    # ローカル個人設定（平文は Git 管理外）
│   │   ├── ansible
│   │   │   ├── vars.yml                # 公開変数（IP/ドメイン/バケット名等）
│   │   │   └── credentials
│   │   │       ├── vault.yml           # ① 実体の暗号化機密ファイル（AES256）
│   │   │       ├── .vault_pass         # ① Vault 復号パスフレーズ
│   │   │       ├── id_rsa              # ① Ansible 用 SSH 秘密鍵
│   │   │       └── key.pem             # ① SSL 秘密鍵（暗号化）
│   │   └── terraform
│   │       ├── terraform.tfvars        # ② Terraform 用機密（GitHub PAT / Discord URL / R2鍵）
│   │       └── backend.hcl             # R2 バックエンドアクセス鍵
│   └── terraform
│       └── main.tf                     # ② GitHub Provider による Secret 自動登録の定義
└── .github
    └── workflows                       # ② 自動登録された Secret を消費する側
```

### 利用可能になるサービス・コマンド
- 機密ファイルの暗号化: `ansible-vault encrypt <file>`
- 暗号化機密の編集: `ansible-vault edit <file>`
- GitHub Secrets の一括登録: `terraform apply -var-file=../local_config/terraform/terraform.tfvars`
- 機密漏洩の事前チェック: `pre-commit run --all-files`

### 3.1 Ansible Vault による機密ファイル保管

#### 3.1.1 採用目的
- SSL 秘密鍵・データベースパスワード・Cloudflare API キーなどを **コードと一緒に GitHub で公開** したい
- 復号には事前共有のパスフレーズが必要で、運用者以外には平文を見せない
- ファイル単位とインライン変数の両方を AES256 で暗号化できる

#### 3.1.2 ファイル種別と配置

| ファイル | 配置 | 暗号化 | 用途 |
|---|---|---|---|
| `local_config/ansible/credentials/vault.yml` | ローカル個人設定 | 暗号化 (AES256) | DB パスワード・API キー・Discord Webhook 等の変数 |
| `local_config/ansible/credentials/.vault_pass` | ローカル個人設定 | 平文（Git 管理外） | Vault 復号パスフレーズ。run_playbook.sh が `ANSIBLE_VAULT_PASSWORD_FILE` 環境変数で参照 |
| `local_config/ansible/credentials/id_rsa` | ローカル個人設定 | 暗号化推奨 | Ansible が各ノードに SSH 接続する秘密鍵 |
| `local_config/ansible/credentials/key.pem` | ローカル個人設定 | 暗号化 (AES256) | Web ロールが配置する SSL 秘密鍵 |
| `ansible/group_vars/all/vault.yml.example` | リポジトリ | 平文（テンプレート） | 新規構築時の参考用ダミー値 |
| `local_config/ansible/vars.yml` | ローカル個人設定 | 平文 | IP・ドメイン・バケット名など公開可能な変数 |

`group_vars/all/vault.yml.example` と `local_config/ansible/credentials/vault.yml` の役割の違い:
- `.example` は新規参加者がコピーして使うテンプレートであり、ダミー値しか含まない。Git に同梱する。
- `credentials/vault.yml` は各個人の環境の実値であり、暗号化された状態でも Git 管理対象外（`.gitignore` で除外）。

#### 3.1.3 Vault 内に保管する主な変数
[local_config/ansible/credentials/vault.yml](../iac-workspace/local_config/ansible/credentials/vault.yml)（テンプレート）
```yaml
# Cloudflare API資格情報
cf_access_key: "DUMMY_CF_ACCESS_KEY"
cf_secret_key: "DUMMY_CF_SECRET_KEY"

# PostgreSQL データベース認証情報
postgres_user: "postgres_admin"
postgres_password: "DUMMY_POSTGRES_PASSWORD"
db_name: "shakedb"
db_user_reader: "db_reader"
db_password_reader: "DUMMY_READER_PASSWORD"
db_user_editor: "db_editor"
db_password_editor: "DUMMY_EDITOR_PASSWORD"

# Minecraft (RCON) 認証情報
mc_rcon_password: "DUMMY_MC_RCON_PASSWORD"

# Discord Bot (ubsleepy) 認証情報
ubsleepy_r2_access_key_id: "DUMMY_R2_ACCESS_KEY_ID"
ubsleepy_r2_secret_access_key: "DUMMY_R2_SECRET_ACCESS_KEY"
ubsleepy_discord_token: "DUMMY_DISCORD_TOKEN"
ubsleepy_backup_discord_webhook: "https://discordapp.com/api/webhooks/YOUR_DUMMY_WEBHOOK_URL"

# 月次リストアテスト用
r2_access_key_id: "DUMMY_R2_ACCESS_KEY_ID"
r2_secret_access_key: "DUMMY_R2_SECRET_ACCESS_KEY"
r2_endpoint: "https://YOUR_CLOUDFLARE_ACCOUNT_ID.r2.cloudflarestorage.com"
```

#### 3.1.4 暗号化の実行
変数ファイル全体を暗号化する。
```bash
ansible-vault encrypt iac-workspace/local_config/ansible/credentials/vault.yml
```

ファイル単位の機密（SSL 秘密鍵など）も同様に暗号化する。
```bash
ansible-vault encrypt iac-workspace/local_config/ansible/credentials/key.pem
```

#### 3.1.5 Playbook 実行時の復号
[run_playbook.sh](../iac-workspace/run_playbook.sh) は自動で `ANSIBLE_VAULT_PASSWORD_FILE=local_config/ansible/credentials/.vault_pass` を設定するため、対話的にパスフレーズを入力する必要はない。

手動実行する場合は `-e @<vault.yml>` で渡す。
```bash
ansible-playbook -i local_config/ansible/inventory.ini \
  -e @local_config/ansible/vars.yml \
  -e @local_config/ansible/credentials/vault.yml \
  ansible/site.yml
```

#### 3.1.6 Web ロールでの key.pem 配置例
[ansible/roles/web/tasks/main.yml](../iac-workspace/ansible/roles/web/tasks/main.yml)（抜粋）
```yaml
- name: SSL秘密鍵の配置
  ansible.builtin.copy:
    src: "{{ playbook_dir }}/../local_config/ansible/credentials/key.pem"
    dest: "{{ app_dir }}/certs/key.pem"
    owner: "{{ app_owner }}"
    group: "{{ app_group }}"
    mode: "0600"
```

`ansible.builtin.copy` は Vault で暗号化されたファイルを自動で復号して転送する。配置先パーミッションは 0600 に固定し、Web コンテナ（www-data ユーザ）のみが読める状態を維持する。

### 3.2 Terraform GitHub Provider による Secret 自動登録

#### 3.2.1 採用目的
本リポジトリの GitHub Actions ワークフロー（CI / CD / drift 検知）は、計 7 種類の Secret を必要とする。これらを GitHub の Web UI から手動登録するのは煩雑かつミスを生むため、Terraform の GitHub Provider で**コードとしてシークレットを定義し、`terraform apply` で一括登録**する。値の更新もコード変更 → apply で反映される。

#### 3.2.2 自動登録される Secret 一覧
[terraform/main.tf](../iac-workspace/terraform/main.tf) で定義しているすべての Secret は以下の通り。

| Secret 名 | 登録先リポジトリ | 値の出所 | 消費するワークフロー |
|---|---|---|---|
| `ANSIBLE_SSH_KEY` | shake-infra | `local_config/ansible/credentials/id_rsa`（平文ファイル） | cd_deploy.yml で各ノードへの SSH 接続に使用 |
| `ANSIBLE_VAULT_PASS` | shake-infra | `local_config/ansible/credentials/.vault_pass` | cd_deploy.yml で Vault 復号に使用 |
| `CLOUDFLARE_API_TOKEN` | shake-infra | `var.cloudflare_api_token` | terraform_drift.yml / ci_infra.yml の Plan で使用 |
| `CF_R2_ACCESS_KEY_ID` | shake-infra | `var.cf_r2_access_key_id` | terraform_drift.yml / ci_infra.yml の R2 バックエンド認証 |
| `CF_R2_SECRET_ACCESS_KEY` | shake-infra | `var.cf_r2_secret_access_key` | 同上 |
| `TF_BACKEND_CONFIG` | shake-infra | `local.tf_backend_config`（HCL 文字列） | terraform_drift.yml / ci_infra.yml で `backend-config` として書き出し |
| `DISCORD_WEBHOOK_URL` | shake-infra | `var.discord_webhook_url` | cd_deploy.yml / terraform_drift.yml の通知に使用 |
| `INFRA_REPO_DISPATCH_TOKEN` | アプリリポジトリ (ubsleepy 等) | `var.infra_repo_dispatch_token` | アプリ側 PR マージ時にインフラへ Repository Dispatch を送信 |

これら 8 種の Secret の流れにより、CI / CD / drift 検知のすべてが**手動 Web UI 操作を一切経由せずに**シークレット供給を完結させる。

#### 3.2.3 事前準備: Terraform 用 GitHub トークンの取得
Terraform が GitHub の Secrets を操作するため、初回のみ Fine-grained Personal Access Token (PAT) を発行する。

1. GitHub の Settings > Developer settings > Personal access tokens (Fine-grained tokens) へアクセス。
2. Generate new token をクリック。
3. Repository access で対象リポジトリ（shake-infra および ubsleepy 等のアプリリポジトリ）を選択。
4. Permissions > Repository permissions にて以下を設定:
    - `Administration` (Read/Write)
    - `Secrets` (Read/Write)
    - `Contents` (Read/Write)（INFRA_REPO_DISPATCH_TOKEN 用）
5. 生成された値をメモする。

#### 3.2.4 事前準備: Discord Webhook URL の取得
1. 通知を送信したい Discord チャンネルの設定 > 連携サービス > ウェブフック > 新しいウェブフックを作成。
2. ウェブフックの名前を設定し、「ウェブフックURLをコピー」をクリック。

#### 3.2.5 アプリリポジトリ用 Dispatch Token の発行
アプリ（ubsleepy 等）からインフラリポジトリへ API シグナルを送るための PAT を別途発行する。

1. GitHub の Settings > Developer settings > Personal access tokens (Fine-grained) で新規発行。
2. Repository access: shake-infra のみを選択。
3. Permissions > Repository permissions > `Contents` を Read and write に設定。
4. 生成された値をメモする。これが `infra_repo_dispatch_token` 変数の値となる。

#### 3.2.6 terraform.tfvars への記入と適用
[iac-workspace/local_config/terraform/terraform.tfvars](../iac-workspace/local_config/terraform/terraform.tfvars)
```hcl
# 3.2.3 で取得した Terraform 用 GitHub PAT
github_token              = "github_pat_xxxxxxxxxxxx"
# 3.2.4 で取得した Discord Webhook URL
discord_webhook_url       = "https://discord.com/api/webhooks/xxxx/xxxx"
# 3.2.5 で取得したアプリ → インフラ用 Dispatch PAT
infra_repo_dispatch_token = "github_pat_xxxxxxxxx"

# Cloudflare 関連（Terraform のリソース管理にも使用）
cloudflare_api_token      = "<Cloudflare API Token>"
cloudflare_account_id     = "<Cloudflare Account ID>"
cloudflare_zone_id        = "<Cloudflare Zone ID>"
server_ip                 = "<大学のグローバルIP>"

# R2 バックエンド・CI 用
cf_r2_access_key_id       = "<R2 Access Key ID>"
cf_r2_secret_access_key   = "<R2 Secret Access Key>"
```

このファイルは絶対にコミットしない（`.gitignore` で除外済）。適用は以下のコマンドで行う。

```bash
cd iac-workspace/terraform
terraform init
terraform apply -var-file=../local_config/terraform/terraform.tfvars
```

apply 後、GitHub の Settings > Secrets and variables > Actions に上記 Secret が一括反映される。値の更新も同コマンドの再実行で行う。

#### 3.2.7 Terraform GitHub Provider の定義例
[terraform/main.tf](../iac-workspace/terraform/main.tf)（抜粋）
```hcl
provider "github" {
  token = var.github_token
  owner = var.github_owner
}

data "github_repository" "infra" {
  name = "shake-infra"
}

resource "github_actions_secret" "ansible_ssh_key" {
  repository  = data.github_repository.infra.name
  secret_name = "ANSIBLE_SSH_KEY"
  value       = file("${path.module}/../local_config/ansible/credentials/id_rsa")
}

resource "github_actions_secret" "ansible_vault_pass" {
  repository  = data.github_repository.infra.name
  secret_name = "ANSIBLE_VAULT_PASS"
  value       = file("${path.module}/../local_config/ansible/credentials/.vault_pass")
}

# 他5種の Secret も同様に定義
```

`file()` 関数でローカルファイルを読み込む形にしているため、ローカルの id_rsa / .vault_pass を更新するだけで `terraform apply` 一発で GitHub Actions Secrets にも反映される。

### 3.3 漏洩防止: pre-commit + detect-secrets による事前検出

#### 3.3.1 採用目的
たとえ Vault や Terraform で機密を管理していても、開発中の不注意で平文の `vault.yml` や `terraform.tfvars` をコミットしてしまうリスクは残る。**コミット前に Git フックで自動チェック**することで、こうした「うっかり」を機械的にブロックする。

本リポジトリは `pre-commit` フレームワークを採用し、以下の3系統のチェックを一括設定している。

#### 3.3.2 構成ファイル
[.pre-commit-config.yaml](../.pre-commit-config.yaml)
```yaml
repos:
  - repo: https://github.com/ansible/ansible-lint
    rev: v26.4.0
    hooks:
      - id: ansible-lint
        files: iac-workspace/ansible/
        language_version: python3.13
        entry: env LC_ALL=C.UTF-8 ansible-lint

  - repo: https://github.com/antonbabenko/pre-commit-terraform
    rev: v1.96.1
    hooks:
      - id: terraform_fmt
      - id: terraform_validate

  - repo: https://github.com/Yelp/detect-secrets
    rev: v1.5.0
    hooks:
      - id: detect-secrets
        args: ['--baseline', '.secrets.baseline']
        exclude: ^docs/
```

| フック | 検出対象 |
|---|---|
| `ansible-lint` | Ansible のベストプラクティス違反・冪等性問題 |
| `terraform_fmt` | Terraform コードのフォーマット崩れを自動修正 |
| `terraform_validate` | Terraform コードの構文エラー |
| `detect-secrets` | AWS アクセスキー・JWT・base64 高エントロピー文字列等の機密パターン |

#### 3.3.3 .secrets.baseline の役割
`detect-secrets` は誤検知を防ぐため、リポジトリに**既知の安全な文字列**を許容リストとして登録する。これが [.secrets.baseline](../.secrets.baseline) の役割。

ダミー値（`DUMMY_CF_ACCESS_KEY` 等）や Vault で暗号化済みのファイルなど、検査対象外として正しいエントリのみが含まれている。新しい機密パターンを取り込みたい場合は以下で再生成する。
```bash
detect-secrets scan > .secrets.baseline
detect-secrets audit .secrets.baseline
```

#### 3.3.4 ローカル開発環境への導入
新規参加者は以下の3コマンドのみ実行すれば、コミット時に自動でチェックが走るようになる。

```bash
# pre-commit のインストール
pip install pre-commit

# フックをローカルの .git/hooks に登録
pre-commit install

# 手動で全ファイルに対して実行（初回確認用）
pre-commit run --all-files

# 各フックのバージョンを最新に更新
pre-commit autoupdate
```

#### 3.3.5 git-secrets による補助チェック（旧来手法）
本プロジェクト初期では git-secrets を採用していたが、現在は `pre-commit + detect-secrets` に統合した。旧設定が残っている場合のチェック項目を以下に記録しておく。

| パターン | 内容 |
|---|---|
| AWS アクセスキー / シークレットキー | `git-secrets --register-aws` 由来 |
| Ansible Vault ヘッダー非該当 (`^\$ANSIBLE_VAULT` がない vault.yml) | カスタムルールでコミット拒否 |
| `iac-workspace/local_config/terraform/terraform.tfvars` のコミット | 禁止ファイルとして登録 |

新規環境では pre-commit のみで十分であり、git-secrets の導入は不要。

### 3.4 シークレットライフサイクルの全体フロー

```text
[ ローカル管理者 ]                  [ Git リポジトリ ]                  [ GitHub Actions ]
        │                                  │                                    │
        │ ansible-vault encrypt            │                                    │
        ├─────────► vault.yml (暗号化済) ──► local_config/...                    │
        │                                  │  ※暗号文を含めて commit 可能       │
        │                                  │                                    │
        │ ansible-playbook with .vault_pass│                                    │
        │ ───────────────────────────────────────► 復号して各ノードへ展開        │
        │                                  │                                    │
        │ terraform apply                  │                                    │
        ├─────────► main.tf が読み取り ───► GitHub Provider                     │
        │           - id_rsa / .vault_pass / Cloudflare token 等                │
        │           を GitHub Actions Secrets として自動登録 ──────────────────► │
        │                                  │                                    │
        │ git commit                       │                                    │
        ├─────────► pre-commit ───► detect-secrets / ansible-lint / tf 検査     │
        │                                  │                                    │
        │                                  │                                    │ workflow が secrets.* で参照
        │                                  │                                    │ (cd_deploy / ci_infra / drift)
```

この3層の連携により、機密情報は「ローカルで暗号化保管 → 必要な実行系に自動分配 → コミット前に検出」というクローズドループで管理される。
