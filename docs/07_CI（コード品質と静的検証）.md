# 7. CI（コード品質と静的検証）

## CI の目的
Pull Request のマージ前に、Ansible / Terraform / Docker 設定の構文・スタイル・脆弱性を機械的に検証し、本番デプロイ時の予期せぬエラーや脆弱性混入を未然に防ぐ。CI はあくまで「検証のみ」を担当し、実機への変更（CD）は[08_CD（自動デプロイ）.md](08_CD（自動デプロイ）.md) で扱う。

### CI で実行する6つのジョブ

ワークフローは GitHub Actions 上に日本語の表示名で現れる（ワークフロー全体の一覧は [GitHubActionsワークフローリファレンス.md](GitHubActionsワークフローリファレンス.md) を参照）。

| 表示名（ジョブID） | 検証対象 | 失敗時の影響 |
|---|---|---|
| Ansible 静的解析・構文チェック（`ansible-ci`） | Ansible Playbook の構文と Lint | PR マージ不可（必須） |
| ローカル統合テスト（Docker）（`local-integration-tests`） | 使い捨てコンテナ/モックでロール・監視・フェイルオーバーを検証（§7.7） | PR マージ不可（必須） |
| Terraform 整形・検証（`terraform-ci`） | Terraform コードのフォーマットと validate | PR マージ不可（必須） |
| Terraform Plan（PRのみ）（`terraform-plan`） | PR 差分 plan を PR コメントに自動投稿。Dependabot PR ではスキップ（§7.8） | continue-on-error（参考情報） |
| コンテナ・IaC 脆弱性スキャン（`trivy-scan`） | Docker Compose / Ansible files の HIGH/CRITICAL 脆弱性 | PR マージ不可（必須） |
| Discord 通知（失敗時）（`notify-failure`） | いずれかのジョブ失敗時に Discord 通知（§7.8） | 通知のみ |

### この章で扱うファイル
```text
infra
├── .github/workflows
│   └── ci_infra.yml                            # CI ワークフロー定義
├── iac-workspace
│   └── ansible
│       └── .ansible-lint                       # Lint ルール設定
├── .ansible-lint                               # リポジトリ全体の Lint 設定（参照用）
└── .pre-commit-config.yaml                     # ローカル pre-commit との二重防御
```

### 利用可能になる機能
- PR 作成時に CI ワークフローが自動起動
- `terraform plan` の結果が PR コメントとして自動投稿
- 機密情報・脆弱性パターンを検出した PR は自動的にマージ拒否

### 7.1 CI ワークフローの全体像

#### 7.1.1 トリガー条件
[.github/workflows/ci_infra.yml](../.github/workflows/ci_infra.yml)
```yaml
on:
  pull_request:
    branches: [main]
    paths:
      - 'iac-workspace/**'
      - '.github/workflows/ci_infra.yml'
  push:
    branches: [main]
    paths:
      - 'iac-workspace/**'
      - '.github/workflows/ci_infra.yml'
```

PR と main ブランチへの直接 push の両方で発火。`paths` フィルタにより、ドキュメントのみの変更では CI を起こさず Actions 実行枠を節約する。

#### 7.1.2 実行環境
全ジョブが `ubuntu-latest` で動作し、Self-hosted Runner は使わない。理由は以下:
- 静的検証は外部接続不要のため public runner で問題ない
- CD（実機デプロイ）と CI を分離することで、本番ノード停止時でも CI は動作する
- GitHub Actions の無料枠を有効活用

### 7.2 ansible-ci ジョブ: 構文チェックと Lint

#### 7.2.1 採用目的
Ansible Playbook の記述には**書き方の自由度が高い**ため、属人化・冪等性違反・非推奨モジュールの使用などのリスクが発生しやすい。Ansible Lint で**公式のベストプラクティス（profile: production）**への準拠を機械的に検証する。

#### 7.2.2 ジョブ定義
```yaml
ansible-ci:
  name: Ansible Lint & Syntax Check
  runs-on: ubuntu-latest
  defaults:
    run:
      working-directory: iac-workspace/ansible
  steps:
    - name: Checkout Repository
      uses: actions/checkout@v4

    - name: Set up Python
      uses: actions/setup-python@v5
      with:
        python-version: '3.10'

    - name: Install Ansible and dependencies
      run: |
        python -m pip install --upgrade pip
        pip install ansible-core ansible-lint

    - name: Install Ansible Collections
      run: ansible-galaxy collection install -r requirements.yml

    - name: Run Ansible Syntax Check
      run: ansible-playbook -i inventory.ini site.yml --syntax-check

    - name: Run Ansible Lint
      working-directory: iac-workspace
      run: ansible-lint ansible/site.yml
```

#### 7.2.3 .ansible-lint の設定
[iac-workspace/.ansible-lint](../iac-workspace/.ansible-lint) または [.ansible-lint](../.ansible-lint)
```yaml
profile: production
exclude_paths:
  - .github/
warn_list:
  - experimental
```

`profile: production` は Ansible Lint が定義する最も厳格な品質基準。本番運用品質を CI レベルで強制する。

#### 7.2.4 inventory.ini ダミー利用
Syntax Check は実機への SSH 接続を伴わない静的検証だが、`ansible-playbook` 自体は inventory が存在しないと一部モジュールの解析でエラーになる。そのため [`ansible/inventory.ini`](../iac-workspace/ansible/inventory.ini)（リポジトリ同梱のダミー、または `local_config/ansible/inventory.ini` のテンプレート相当）を使って `--syntax-check` を実行している。実通信は発生しない。

### 7.3 terraform-ci ジョブ: フォーマットと validate

#### 7.3.1 採用目的
Terraform コードの記述スタイル統一と、構文・型・参照整合性のチェックを行う。バックエンドへの実接続は不要なため `terraform init -backend=false` で初期化する。

#### 7.3.2 ジョブ定義
```yaml
terraform-ci:
  name: Terraform Format & Validate
  runs-on: ubuntu-latest
  defaults:
    run:
      working-directory: iac-workspace/terraform
  steps:
    - name: Checkout Repository
      uses: actions/checkout@v4

    - name: Setup Terraform
      uses: hashicorp/setup-terraform@v3

    - name: Terraform Format Check
      run: terraform fmt -check

    - name: Terraform Init
      run: terraform init -backend=false

    - name: Terraform Validate
      run: terraform validate
```

#### 7.3.3 検査内容
| 工程 | 検査項目 |
|---|---|
| `terraform fmt -check` | インデント、空白、引用符などのスタイル違反を検出（失敗で CI が落ちる） |
| `terraform init -backend=false` | プロバイダのダウンロード（バックエンド接続は省略） |
| `terraform validate` | リソース定義の構文エラー、変数参照、型不整合の検出 |

### 7.4 terraform-plan ジョブ: PR コメント自動投稿

#### 7.4.1 採用目的
レビュアーが PR をマージする前に「このコード変更が実際にどのリソースを作成・更新・削除するか」を**コードレビュー画面で直接確認**できるようにする。Terraform の世界では特に DNS や Workers Route のような外向け影響が大きいため、apply 前のプレビューは重要。

#### 7.4.2 ジョブ定義（要点）
```yaml
terraform-plan:
  name: Terraform Plan (PR only)
  runs-on: ubuntu-latest
  if: github.event_name == 'pull_request'
  permissions:
    pull-requests: write
  defaults:
    run:
      working-directory: iac-workspace/terraform
  steps:
    - name: Checkout Repository
      uses: actions/checkout@v4

    - name: Setup Terraform
      uses: hashicorp/setup-terraform@v3

    - name: Write backend config
      run: echo "${{ secrets.TF_BACKEND_CONFIG }}" > ci_backend.hcl

    - name: Terraform Init
      run: terraform init -backend-config=ci_backend.hcl
      env:
        AWS_ACCESS_KEY_ID: ${{ secrets.CF_R2_ACCESS_KEY_ID }}
        AWS_SECRET_ACCESS_KEY: ${{ secrets.CF_R2_SECRET_ACCESS_KEY }}
        CLOUDFLARE_API_TOKEN: ${{ secrets.CLOUDFLARE_API_TOKEN }}

    - name: Terraform Plan
      id: plan
      run: terraform plan -no-color -input=false
      continue-on-error: true

    - name: Post Plan to PR
      uses: actions/github-script@v7
      with:
        script: |
          const plan = `${{ steps.plan.outputs.stdout }}`;
          const status = '${{ steps.plan.outcome }}' === 'success' ? '✅' : '⚠️';
          const body = `### ${status} Terraform Plan\n\`\`\`hcl\n${plan || '(出力なし — シークレット未設定またはエラー)'}\n\`\`\``;
          github.rest.issues.createComment({
            issue_number: context.issue.number,
            owner: context.repo.owner,
            repo: context.repo.repo,
            body: body
          });

    - name: Cleanup backend config
      if: always()
      run: rm -f ci_backend.hcl
```

#### 7.4.3 必要な Secret
Terraform Plan は実バックエンド（R2）と Cloudflare API に接続するため、以下の 4 つの Secret が必要。これらは [03_機密情報管理.md](03_機密情報管理.md) §3.2 の Terraform GitHub Provider で自動登録される。

| Secret | 用途 |
|---|---|
| `CLOUDFLARE_API_TOKEN` | Cloudflare プロバイダ用 API トークン |
| `CF_R2_ACCESS_KEY_ID` | R2 バックエンド用アクセスキー |
| `CF_R2_SECRET_ACCESS_KEY` | R2 バックエンド用シークレットキー |
| `TF_BACKEND_CONFIG` | `backend.hcl` の全内容（endpoint/bucket/key/region を含む） |

#### 7.4.4 continue-on-error の意義
`terraform plan` が失敗（exit code != 0）しても `continue-on-error: true` で次のステップに進む設計。理由は以下:
- Secret 未設定状態の Fork PR からも CI が機能するように
- 失敗内容を PR コメントに含めて開発者が見られるように

シークレット未設定の場合は plan ジョブが警告マーク（⚠️）でコメントを残し、PR のマージ自体はブロックしない。

#### 7.4.5 後始末
Backend 設定ファイル `ci_backend.hcl` は機密を含むため、`if: always()` で必ず削除する。CI 環境のキャッシュを介して機密が漏洩することを防ぐ。

### 7.5 trivy-scan ジョブ: コンテナ・IaC 脆弱性スキャン

#### 7.5.1 採用目的
Docker Compose ファイルや Ansible が参照するイメージタグ・設定ファイルから、**既知の HIGH/CRITICAL 脆弱性**を自動検出する。CVE データベースに基づくチェック。

#### 7.5.2 ジョブ定義
```yaml
trivy-scan:
  name: Container & IaC Vulnerability Scan
  runs-on: ubuntu-latest
  steps:
    - name: Checkout Repository
      uses: actions/checkout@v4

    - name: Run Trivy (IaC config scan)
      # aquasecurity/trivy-action v0.36.0 を使用。
      # 2026年3月のタグ改ざん事案に伴い、コミットSHAで固定。
      uses: aquasecurity/trivy-action@ed142fd07b38d3550b074464d607e058d9980d27
      with:
        scan-type: config
        scan-ref: iac-workspace/ansible/files/
        severity: HIGH,CRITICAL
        exit-code: '1'
```

#### 7.5.3 検査対象と運用
- スキャン対象: `iac-workspace/ansible/files/` 配下（各アプリの docker-compose.yml と関連設定）
- 検出レベル: HIGH または CRITICAL のみ（LOW/MEDIUM は無視）
- 検出時の挙動: `exit-code: '1'` により CI が失敗 → PR マージ拒否

新規 CVE が公開されると、過去にパスしていた PR も次回再実行で落ちる可能性がある。定期的な base イメージ更新が運用上必要。

### 7.6 ローカル開発との二重防御

CI（GitHub Actions 側）と pre-commit（ローカル Git 側）の同じツールで二重に検査する設計。
詳細は [03_機密情報管理.md](03_機密情報管理.md) §3.3。

| チェック | pre-commit ローカル | CI（GitHub Actions） |
|---|---|---|
| ansible-lint | ✓ | ✓ |
| terraform fmt | ✓（自動修正） | ✓（チェックのみ） |
| terraform validate | ✓ | ✓ |
| detect-secrets | ✓ | ─（CI では未組込、必要に応じて将来追加） |
| Trivy 脆弱性スキャン | ─ | ✓ |

これにより開発者は **コミット時点で大半のエラーをローカルで弾き** 、レビュー時には CI のみが補完的に動く構造となる。

### 7.7 ローカル統合テストとロール単体テスト（Molecule）

静的解析に加え、使い捨ての Docker コンテナとモックで「実際に動くか」を検証するテスト群を `iac-workspace/tests/` に置き、CI の「ローカル統合テスト（Docker）」ジョブが実行する。本番サーバー・Cloudflare・R2 には接続しないため、ローカルでも同じスクリプトで再現できる。

```bash
./iac-workspace/tests/run.sh             # pytest 一式（postgres / blackbox / failover）
./iac-workspace/tests/run_molecule.sh    # Molecule によるロール単体テスト（exporters）
```

| ファイル | 種別 | 内容 |
|---|---|---|
| tests/test_postgres_role.py | 収束/統合 | 捨て DB で scram 認証・パスワード同期の冪等性・Vault 変更時のドリフト是正を検証 |
| tests/test_blackbox_probe.py | 監視 | go-httpbin の 200/500 を blackbox が検知できることを実証し、prometheus 設定を promtool で構文検証 |
| tests/test_failover.py | 単体 | failover_webhook.py のメンテ切替ロジックを Cloudflare 非接続で検証 |
| ansible/roles/exporters/molecule/ | ロール単体 | コンテナ内でロールを収束 → 冪等（2 回目 changed 0）→ 検証（node-exporter 稼働） |

テスト用の Python 仮想環境はリポジトリ外（$HOME/.cache）に作成し、生成物は .gitignore で除外する。Molecule のシナリオ（converge.yml）がロール本体を解決できるよう、ansible-lint には ANSIBLE_ROLES_PATH を付与している。

### 7.8 CI 失敗の Discord 通知と Dependabot の扱い

#### 7.8.1 失敗時の Discord 通知
`notify-failure` ジョブが全ジョブを needs に取り、`if: failure()` で動作する。いずれかのジョブが失敗したときだけ Discord に通知し、成功時はスキップされる。通知文面のワークフロー名は `${{ github.workflow }}` で自動反映する。

#### 7.8.2 Dependabot PR での terraform-plan スキップ
GitHub はセキュリティ上、Dependabot の PR には Actions シークレットを渡さない。そのため R2 バックエンド認証が必須の `terraform-plan` は Dependabot PR では必ず失敗する。これを誤検知として避けるため、当該ジョブに条件を付けてスキップする。

iac-workspace/.github/workflows/ci_infra.yml（terraform-plan ジョブ）
```yaml
    if: github.event_name == 'pull_request' && github.actor != 'dependabot[bot]'
```
他の検証ジョブ（lint / ローカルテスト / Trivy / terraform-ci）は Dependabot PR でも実行される。
