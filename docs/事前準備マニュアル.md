# 事前準備マニュアル

## 本書の目的
[01_IaC環境構築と初期プロビジョニング.md](01_IaC環境構築と初期プロビジョニング.md) は「WSL2 のインストール」という構築開始後の作業から始まる。本書はその前段として、構築を始める前に手元の作業端末・ハードウェア・各種アカウント・外部サービスのトークンが揃っているかをチェックリスト形式で確認するための入口ドキュメントである。

本書のチェックがすべて埋まった状態であれば、第1章以降の手順を中断なく進められる。各項目の詳細な取得手順は該当章へのリンクを参照する。

## 1. 作業端末の準備

インフラを操作する作業用 PC（管理端末）の準備を確認する。

- [ ] Windows の場合、WSL2 が利用可能（または Linux 端末を用意済み）
- [ ] 安定したインターネット接続がある
- [ ] 各 Raspberry Pi と同一 LAN に到達できる（初期セットアップ時のみ必要。以降は Tailscale 経由）
- [ ] ブラウザで GitHub / Cloudflare / Tailscale / Discord の各管理画面にログインできる

作業端末にインストールするツール（Ansible / Terraform / SSH 鍵生成など）は構築開始後に [01章](01_IaC環境構築と初期プロビジョニング.md) の手順で導入するため、本段階では準備不要。

## 2. ハードウェアの準備

本基盤は 3 拠点・3 ノード構成を前提とする。ノードの役割詳細は [02章 §2.1](02_ネットワーク基盤.md) を参照。

| インベントリ名 | ハードウェア | 拠点 | 役割 |
|---|---|---|---|
| negitoroserver | Raspberry Pi Zero 2 W | 大学内ネットワーク | リバースプロキシ（ゲートウェイ） |
| shakeserver | Raspberry Pi 5 + Geekworm X120X UPS ボード | 自宅 | アプリ本体（Web/DB/Minecraft/Bot）と UPS 監視 |
| tarakoserver | Raspberry Pi 4 | 自宅 | 監視スタックと CI/CD ランナー |

確認項目:

- [ ] 上記 3 台（または相当スペックの代替機）を用意した
- [ ] 各機の OS（Raspberry Pi OS / Ubuntu 等）をインストールし、SSH ログインできる状態にした
- [ ] 各機の初期ユーザーと初期 IP アドレスを把握している（[01章 §1.8](01_IaC環境構築と初期プロビジョニング.md) の inventory.ini に記入する）
- [ ] shakeserver 用に UPS ボードを接続し、I2C を有効化した（UPS 監視を使う場合。[04章 §4.1.2](04_監視・ログ・通知基盤.md) 参照）
- [ ] 緊急用ローカル LAN セグメント `192.168.10.0/24` を把握している（[02章 §2.4](02_ネットワーク基盤.md) の UFW 戦略で常時開放される最終ライフライン）
- [ ] proxy ノードを設置する大学拠点の DNS / NTP サーバー情報を把握している（学内ポリシーで固定が必要な場合。[02章 §2.3.2](02_ネットワーク基盤.md)）

## 3. アカウントとトークンの準備

構築途中で都度発行すると手戻りが発生しやすいため、以下のアカウント開設とトークン取得を事前に済ませておくことを推奨する。各トークンは構築中に local_config 配下の設定ファイルへ記入する。

### 3.1 GitHub

- [ ] GitHub アカウントを保有している
- [ ] インフラ用リポジトリ（例: shake-infra）を作成済み
- [ ] アプリ用リポジトリ（例: ubsleepy, shake-web）を作成済み（CD を使う場合）
- [ ] Terraform 用 Fine-grained PAT を取得した
  - 権限: Administration (R/W)、Secrets (R/W)、Contents (R/W)
  - 用途: GitHub Actions Secrets の自動登録（[03章 §3.2.3](03_機密情報管理.md)）
- [ ] アプリ→インフラ連携用の Dispatch PAT を取得した
  - 権限: 対象を shake-infra に絞り、Contents (R/W)
  - 用途: Repository Dispatch（[03章 §3.2.5](03_機密情報管理.md) / [08章 §8.2](08_CD（自動デプロイ）.md)）

### 3.2 Cloudflare

- [ ] Cloudflare アカウントを保有している
- [ ] ドメインを取得し、Cloudflare のゾーンとして登録済み
- [ ] API トークンを取得した
  - 権限: ゾーン > DNS > 編集、アカウント > Workers R2 ストレージ > 編集
  - 用途: Terraform による DNS / R2 / Workers の管理（[01章 §1.6.1](01_IaC環境構築と初期プロビジョニング.md)）
- [ ] Zone ID と Account ID を控えた（[01章 §1.6.2](01_IaC環境構築と初期プロビジョニング.md)）
- [ ] R2 用の S3 互換資格情報（アクセスキー ID / シークレットアクセスキー）を取得した
  - 用途: バックアップ転送、Terraform Remote State バックエンド（[01章 §1.6.3](01_IaC環境構築と初期プロビジョニング.md) / [06章 §6.5](06_高可用性とシステム堅牢化.md)）

注意: Cloudflare の API トークンと R2 シークレットキーは発行画面を閉じると再表示できない。発行と同時に確実に控える。

### 3.3 Tailscale

- [ ] Tailscale アカウントを保有している
- [ ] Auth Key を発行した
  - 設定: Reusable オン（3 台で同一キーを使うため必須）、Pre-approved オン推奨
  - 用途: Ansible からの一括ノード登録（[02章 §2.2.2](02_ネットワーク基盤.md)）

### 3.4 Discord

- [ ] 通知先となる Discord サーバー / チャンネルを用意した
- [ ] 通知用の Webhook URL を取得した（[03章 §3.2.4](03_機密情報管理.md)）
  - 用途: 監視アラート、バックアップ結果、デプロイ結果、ライフサイクル通知
- [ ] Discord Bot（ubsleepy）を運用する場合、Discord Developer Portal で Bot トークンを取得した（[05章 §5.4](05_アプリのコンテナ化と運用.md)）

## 4. 機密情報の格納先の理解

取得したトークン類は、構築中に以下のローカルファイル（いずれも Git 管理外）へ記入していく。事前にこの対応関係を把握しておくと、第1章以降で迷わない。詳細は [03章](03_機密情報管理.md) を参照。

| 格納先ファイル | 主に記入する値 |
|---|---|
| local_config/ansible/credentials/vault.yml | DB パスワード、Cloudflare R2 鍵、Discord Bot トークン、Discord Webhook（[03章 §3.1.3](03_機密情報管理.md)） |
| local_config/ansible/vars.yml | ドメイン名、R2 バケット名、Account ID など公開可能な値 |
| local_config/ansible/credentials/.vault_pass | Ansible Vault の復号パスフレーズ |
| local_config/ansible/credentials/id_rsa | 各ノードへの SSH 秘密鍵 |
| local_config/terraform/terraform.tfvars | GitHub PAT、Cloudflare トークン、R2 鍵、Discord Webhook（[03章 §3.2.6](03_機密情報管理.md)） |
| local_config/terraform/backend.hcl | R2 Remote State バックエンドのアクセス鍵（[06章 §6.5.2](06_高可用性とシステム堅牢化.md)） |

リポジトリには各ファイルの `.example` テンプレートが同梱されている。コピーして実値を書き込み、`.gitignore` により平文がコミットされないことを確認する。
