# 9. 技術スタックと選定根拠

## 本章の目的
本リポジトリで採用した各技術・ツールについて、「なぜこれを選んだか」「代替案と比べてどこが優位か」を一覧化する。新規参加者や将来の自分が「この場面では別のツールでもよくないか？」と疑問に持ったときの参照先。

各ツールの具体的な使い方は、本章ではなく該当の機能章を参照する。

---

### 9.1 全体観: 採用ツール一覧

| 領域 | ツール | 主な代替 | 採用理由要約 |
|---|---|---|---|
| IaC 構成管理 | Ansible | Chef / Puppet / Salt | エージェントレス・YAML 記述・既存 Pi 群に追加導入不要 |

| バックアップ保管 | Cloudflare R2 | AWS S3 / Backblaze B2 | 同一ベンダーで管理一元化・データ転送無料 |

| セルフホストランナー | github_runner（Docker 化） | bare metal インストール | クリーンアップ容易・ホスト OS 汚染なし |

### 9.2 ネットワーク基盤

#### 9.2.1 Tailscale を選んだ理由
- **対抗候補との比較**:
  - WireGuard 直管理: 各ノードの公開鍵管理・キー再発行・ピア間設定が手動。3 拠点を超えると現実的でない
  - Nebula / ZeroTier: 同じ用途だが Tailscale の方が UX が成熟・MagicDNS や Tailscale SSH の周辺機能が充実
  - VPN プロバイダ商用品: 個人利用には過剰
- **決定打**:
  - 無料枠で 100 ノードまで（個人用途には十分すぎる）
  - Auth Key 1 個で `tailscale up --authkey=...` 一発登録可能 → Ansible で完全自動化できる
  - WireGuard ベースで実 NAT 越えを成功させる確率が高い
  - CGNAT 帯（100.64.0.0/10）からプライベート IP を自動付与 → UFW でレンジ指定が容易

#### 9.2.2 SSLH を選んだ理由
- **対抗候補との比較**:
  - HAProxy: TLS + 任意 TCP の多重化は可能だが、設定が複雑かつ Pi Zero 2 W の 512MB RAM では重い
  - Envoy: 同上、Envoy はクラウドネイティブ環境前提
- **決定打**:
  - 単一 443 ポートで HTTPS と Minecraft（バイナリプロトコル）を区別できる ALPN/SNI ベースのプロトコル判別器
  - 設定ファイルが 1 ファイル数行で済むシンプルさ
  - Debian パッケージとして apt で導入可能・追加ビルド不要

#### 9.2.3 Nginx Proxy Protocol を選んだ理由
- **対抗候補**: 単純な `proxy_pass` で X-Forwarded-For を追加する HTTP プロキシ
- **決定打**: SSLH の後段が TCP レベルプロキシなので HTTP ヘッダーを書けない。Proxy Protocol は HTTP/HTTPS/任意プロトコルすべてで動作するため、stream モジュールでも実 IP を保てる

### 9.3 機密情報管理

#### 9.3.1 Ansible Vault を選んだ理由
- **対抗候補との比較**:
  - HashiCorp Vault: サーバー常駐型でリソース消費が大きい、認可ロジックも別途必要
  - SOPS + age/PGP: Ansible との統合に手間（custom lookup や別タスクが必要）
  - 暗号化ボリューム（LUKS 等）: ファイル単位の細かい暗号化ができない
- **決定打**:
  - Ansible に最初から統合されている
  - 単一パスフレーズで暗号化／復号できる運用シンプルさ
  - 変数ファイル全体・特定値・任意ファイル（key.pem 等）の 3 通りに対応

#### 9.3.2 Terraform GitHub Provider を選んだ理由
- **対抗候補との比較**:
  - `gh` CLI スクリプト: 手書きシェルでは Secret の値が履歴に残る
  - 手動 Web UI 登録: コードレビューや CI が効かず、属人化・ミスを誘発
- **決定打**:
  - すべての Secrets を `terraform apply` 一発で再構築可能
  - ローカルファイル（id_rsa, .vault_pass）の値を `file()` 関数で参照すれば、ファイル更新 → apply で同期される
  - Provider 自体が GitHub 公式パートナー製で信頼性が高い

#### 9.3.3 pre-commit + detect-secrets を選んだ理由
- **対抗候補との比較**:
  - git-secrets: 過去に採用していたが、機密パターンが AWS 中心で本リポジトリの多様な機密に追従しづらかった
  - gitleaks: detect-secrets と同等だが、pre-commit フレームワークでの統合が detect-secrets の方が成熟
- **決定打**:
  - `.secrets.baseline` で既知の安全文字列を許容できる
  - 高エントロピー文字列検出（base64 トークン、JWT 等）が汎用的
  - pre-commit フレームワークで ansible-lint・terraform 関連フックと一括管理できる

### 9.4 監視・ログ

#### 9.4.1 Prometheus + Grafana を選んだ理由
- **対抗候補との比較**:
  - Telegraf + InfluxDB: 同等構成だが、Pull 型 vs Push 型で運用思想が異なる
  - Zabbix: GUI 依存が強く Pi 上で重い
  - Netdata: SaaS 利用前提が強く、設定の柔軟性に欠ける
- **決定打**:
  - **Pull 型**で監視サーバーが各ノードに接続する設計が、Tailscale UFW の運用と整合
  - Grafana に PromQL のサジェスト機能があり、運用者がクエリを書きやすい
  - exporter のエコシステムが圧倒的（node_exporter, cAdvisor, blackbox_exporter 等）
  - 全てのコンポーネントが OSS で無料

#### 9.4.2 Loki + Promtail を選んだ理由
- **対抗候補との比較**:
  - Elastic Stack（Elasticsearch+Logstash+Kibana）: Pi 上では重すぎる（JVM が前提）
  - Fluentd / Fluent Bit: 集約先に別途 DB が必要
  - Vector + ClickHouse: 高機能だがセットアップコストが高い
- **決定打**:
  - 「ラベルとインデックスを最小化したログ集約」という Prometheus と同じ哲学
  - Grafana に統合済 → 既存のダッシュボードと並べて表示可能
  - メトリクスとログを 1 つの可視化基盤で扱える運用簡素性

#### 9.4.3 Cloudflare Workers をフェイルオーバー先に選んだ理由
- **対抗候補との比較**:
  - AWS Lambda@Edge: アカウントとクレジットカード登録が必要・無料枠でも追加コスト懸念
  - Vercel / Netlify: 別サービスで認証が必要・Cloudflare DNS との統合に手間
  - 静的 HTML を別 VPS にホスト: VPS の固定費が発生
- **決定打**:
  - DNS が同じ Cloudflare 配下 → Workers Route で同一ゾーン内ルーティング切替が即時反映
  - 無料枠の 10 万リクエスト/日で個人サイトの障害時応答には十分
  - Terraform Cloudflare Provider で Workers と Route を宣言的に管理できる

### 9.5 ストレージとバックアップ

#### 9.5.1 Cloudflare R2 を選んだ理由
- **対抗候補との比較**:
  - AWS S3: 安価だがエグレス（取り出し）に課金される。リストアテスト時に痛い
  - Backblaze B2: 安価でエグレスも安いが、Cloudflare DNS との統合がない
  - 自宅 NAS: ハードウェア故障時にバックアップ自体が失われる SPOF
- **決定打**:
  - **エグレス無料**（リストア時にコストが発生しない）
  - S3 互換 API → `aws-cli` や `rclone`, `eeshugerman/postgres-backup-s3` などの既存ツールがそのまま動く
  - Cloudflare アカウント内で完結 → Terraform 一元管理

#### 9.5.2 eeshugerman/postgres-backup-s3 を選んだ理由
- **対抗候補との比較**:
  - 自前 cron + pg_dump スクリプト: 世代管理・通知ロジックを書く必要がある
  - pgBackRest: 機能豊富だが Pi 上では過剰
- **決定打**:
  - Docker Compose に 1 ブロック追加するだけで日次バックアップが完成
  - `SCHEDULE`, `S3_PREFIX`, `S3_ENDPOINT` 等の環境変数だけで設定完了
  - pg_dump カスタム形式（.dump.gz）で出力 → pg_restore で部分復元も可能

#### 9.5.3 itzg/mc-backup を選んだ理由
- itzg シリーズは Minecraft 公式コミュニティで事実上のデファクト
- `RCLONE_*` 環境変数で R2 への直接アップロードが可能
- save-all を実行してから tar.gz するロジックが内蔵

### 9.6 アプリケーション基盤

#### 9.6.1 uv を Python パッケージマネージャに選んだ理由
- **対抗候補との比較**:
  - pip: 標準だが lockfile がなく再現性に欠ける
  - poetry: 機能豊富だが pyproject.toml の規約に縛られる
  - pdm / hatch: 同等だが ecosystem の成熟度で uv に劣る
- **決定打**:
  - Rust 実装で**インストール速度が 10〜100 倍**速い → Pi の遅い CPU では効果絶大
  - venv 作成・依存導入が単一バイナリで完結
  - pip 互換のコマンドフラグ
  - 公式インストールスクリプトが配布されている

#### 9.6.2 itzg/minecraft-server を選んだ理由
- **対抗候補**: 公式 jar を手動配置 + systemd 化
- **決定打**:
  - すべてのサーバー設定が環境変数で制御可能 → Ansible templates との相性◎
  - CurseForge MOD の自動取得 (`CURSEFORGE_FILES`)
  - リソースパック URL/SHA1 の自動配信 (`RESOURCE_PACK`)
  - 公式 Java バージョンマトリクス対応（`java21` タグ等）
  - RCON 内蔵で IaC からゲーム内コマンドを叩ける

### 9.7 CI/CD

#### 9.7.1 ansible-lint を選んだ理由
- 公式コミュニティのデファクト
- `profile: production` で本番運用品質を機械的に強制可能
- pre-commit + GitHub Actions の両方に組み込める

#### 9.7.2 Trivy を選んだ理由
- **対抗候補との比較**:
  - Snyk: 商用無料枠あるがアカウント連携が必要
  - Anchore: 同等機能、コミュニティ Edition は機能制限あり
  - Docker Scan (Snyk OEM): 同上
- **決定打**:
  - OSS でアカウント不要
  - GitHub Actions に公式 action が用意されている
  - `scan-type: config` で IaC ファイル自体も検査できる（Docker イメージだけでなく Compose 設定等も）
  - HIGH/CRITICAL のフィルタが簡単

#### 9.7.3 Self-hosted Runner（Docker 化）を選んだ理由
- **対抗候補**:
  - GitHub-hosted runner: Tailscale 配下のノードへ到達不可
  - bare metal セットアップ: ホスト OS の汚染、アップデート時の手作業
- **決定打**:
  - Docker コンテナとして稼働 → 不要になればコンテナ削除で完全クリーンアップ
  - Ansible role で**冪等にデプロイ可能**（トークン入れ替えも apply 一発）
  - ホスト OS のパッケージマネージャに依存しない（actions-runner バイナリは GitHub から取得）

### 9.8 Ansible 性能最適化（ansible.cfg）

#### 9.8.1 採用設定
[iac-workspace/ansible.cfg](../iac-workspace/ansible.cfg)
```ini
[defaults]
interpreter_python = auto_silent
forks = 10
gathering = smart
fact_caching = jsonfile
fact_caching_connection = ~/.ansible/fact_cache
fact_caching_timeout = 86400
timeout = 30

[ssh_connection]
pipelining = True
ssh_args = -o ControlMaster=auto -o ControlPersist=60s -o ControlPath=/tmp/ansible-ssh-%r@%h-%p -o ConnectTimeout=30 -o ConnectionAttempts=3
```

#### 9.8.2 各設定の意図
| 設定 | 効果 |
|---|---|
| `interpreter_python = auto_silent` | Python の自動検出と警告抑制（ノードごとに OS バージョンが異なるため） |
| `forks = 10` | 3 ノードに対して余裕のある並列数。並列タスクのデッドロックを避ける |
| `gathering = smart` | 同一プレイブック内の 2 回目以降のファクト収集をスキップ（実行時間 -20〜30%） |
| `fact_caching = jsonfile` | ファクトをローカルにキャッシュ。次回実行でも収集コストを削減 |
| `fact_caching_timeout = 86400` | キャッシュ TTL は 1 日（カーネルアップデート等の頻度想定） |
| `pipelining = True` | タスクごとの SSH 接続オーバーヘッドを削減（最大の効果） |
| `ControlMaster=auto / ControlPersist=60s` | SSH 接続多重化により、ホストへの新規接続を 60 秒間再利用 |
| `ConnectTimeout=30 / ConnectionAttempts=3` | Tailscale の一時的なノード不在を許容する保険 |

これらの最適化により、フルデプロイ（site.yml 全体）の実行時間は最適化前と比べて約 40〜50% 短縮されている。

### 9.9 採用しなかった案

#### 9.9.1 Kubernetes を採用しなかった理由
- 3 ノード規模で Kubernetes は overkill
- Pi Zero 2 W では control plane を担えない
- 学習・運用コストが個人インフラの規模に対して過大

#### 9.9.2 Docker Swarm を採用しなかった理由
- ノード障害時の自動フェイルオーバーは Cloudflare Workers で代替
- ノード間のサービス移動は要件にない（各ノードの役割が固定）
- 過剰な抽象化を避け、各ノード上での Docker Compose 直管理を採用

#### 9.9.3 Cloudflare Tunnel を採用しなかった理由
- Tailscale で十分。Cloudflare Tunnel は HTTP/HTTPS に最適化されており Minecraft（任意 TCP）を通すには工夫が必要
- Tailscale 自体は Cloudflare に依存しないため、ベンダーロックインを 1 段階分散できる

### 9.10 まとめ: 設計思想

本リポジトリの技術スタックは以下 4 つの原則で選定されている。

1. **既存リソースの徹底活用**: Cloudflare アカウントを起点に DNS / R2 / Workers を一元管理
2. **OSS と無料枠の組合せ**: 月額固定コストをゼロに保つ
3. **エージェントレス・宣言的**: Ansible + Terraform + YAML/HCL の組合せで手動運用を完全排除
4. **Pi のリソース制約に配慮**: 軽量ツール（SSLH/Loki/uv）と最小権限ユーザーの徹底

新たなツール採用を検討する際は、これら 4 原則と本章の対抗候補比較を参考に判断する。
