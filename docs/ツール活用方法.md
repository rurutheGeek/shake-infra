## 1. Grafana

監視ノード（tarakoserver / home_node2）で稼働するダッシュボード可視化ツール。構築は [04章 §4.2.5](04_監視・ログ・通知基盤.md)。

アクセス: `http://<tarakoserver の Tailscale IP>:3000`（初期 ID/PW: admin/admin）

### 1.1 ダッシュボードの追加（インポート）
コミュニティ製ダッシュボードは ID 指定でインポートできる。

1. 左メニューの Dashboards > New > Import を開く
2. ダッシュボード ID を入力（下表）し Load
3. データソースに Prometheus を選択して Import

| ダッシュボード | ID | 用途 |
|---|---|---|
| Node Exporter Full | 1860 | OS リソース（CPU/メモリ/ディスク/ネットワーク） |
| cAdvisor | 14282（または 179） | Docker コンテナのリソース |

### 1.2 パネルのカスタマイズ
UPS など独自メトリクスは新規パネルで可視化する。

1. ダッシュボードを開き Add > Visualization
2. データソースに Prometheus を選択
3. クエリ欄に PromQL を入力。例:
   - `ups_battery_capacity_percent` — バッテリー残量
   - `ups_external_power_status` — 外部電源（AC）の有無
   - `ups_voltage_volts` — 電圧
4. 可視化タイプ（Time series / Gauge / Stat 等）を選択し Apply
5. ダッシュボード右上の保存アイコンで永続化

注意: Grafana の設定（ダッシュボード定義・データソース）はコンテナのボリュームに保存される。`docker compose down -v` でボリュームを消すと初期化されるため、重要なダッシュボードは JSON エクスポートで控えておく。

### 1.3 ログの検索（Loki）
メトリクスではなくログを調べる場合は Explore を使う。

1. 左メニューの Explore を開く
2. データソースに Loki を選択
3. LogQL でラベルを指定して検索。例:
   - `{job="varlogs"}` — `/var/log/` 配下のシステムログ
   - `{job="docker"}` — Docker コンテナのログ全般
   - `{compose_service="prometheus"}` — 特定コンテナのログ
   - `{host="tarakoserver"}` — ノードで絞り込み

利用可能なラベルの一覧は [04章 §4.2.4](04_監視・ログ・通知基盤.md) を参照。

### 1.4 アラートについて
本基盤のアラート発火は Grafana ではなく Prometheus + Alertmanager 側で定義している（→§2.2）。Grafana 側でアラートを追加することも可能だが、通知経路（Discord / フェイルオーバー）と重複しないよう、原則として Prometheus 側のルールに集約する。

## 2. Prometheus

メトリクス収集の中核。設定ファイルは Ansible 管理下にあり、ノード上で直接編集せずリポジトリのファイルを編集して再デプロイする。

アクセス: `http://<tarakoserver の Tailscale IP>:9090`（ターゲット状態は `/targets`）

### 2.1 PromQL クエリの記述
Prometheus UI の Graph タブ、または Grafana のクエリ欄で使用する基本例:

```promql
# CPU 使用率（idle 以外の割合）
100 - (avg by (instance) (rate(node_cpu_seconds_total{mode="idle"}[5m])) * 100)

# メモリ使用率
(1 - (node_memory_MemAvailable_bytes / node_memory_MemTotal_bytes)) * 100

# ルートパーティションのディスク使用率
100 - (node_filesystem_avail_bytes{mountpoint="/"} / node_filesystem_size_bytes{mountpoint="/"} * 100)

# ノードの死活
up{job="node_exporter"}
```

### 2.2 アラートルールの追加
アラートは [ansible/files/monitoring/prometheus_rules.yml](../iac-workspace/ansible/files/monitoring/prometheus_rules.yml) に定義する。既存ルールは [04章 §4.3.1](04_監視・ログ・通知基盤.md) を参照。

追加手順:
1. prometheus_rules.yml に新しい `alert:` ブロックを追記する
2. すべてのアラートは Alertmanager 経由で単一 Webhook（failover_webhook.py）に集約される（[04章 §4.3.2](04_監視・ログ・通知基盤.md)）。新規アラート名に対する個別の通知文面を変えたい場合は [ansible/files/monitoring/failover_webhook.py](../iac-workspace/ansible/files/monitoring/failover_webhook.py) の分岐も編集する
3. monitoring ロールを再適用して反映する

```bash
cd iac-workspace
ansible-playbook -i local_config/ansible/inventory.ini \
  -e @local_config/ansible/vars.yml \
  -e @local_config/ansible/credentials/vault.yml \
  ansible/site.yml --tags monitoring
```

注意: `ProxyDown` アラートは自動フェイルオーバーのトリガーを兼ねるため発火待機が 30 秒に設定されている（[06章 §6.1.5](06_高可用性とシステム堅牢化.md)）。この値を安易に変更するとフェイルオーバーの応答性に影響する。

### 2.3 スクレイプ対象の変更
監視対象ノードやポートを変える場合は [ansible/files/monitoring/prometheus.yml](../iac-workspace/ansible/files/monitoring/prometheus.yml) の `scrape_configs` を編集する。

- ターゲットは Tailscale IP（100.x.y.z）で固定する。理由は UFW がメトリクス取得を Tailscale 経由のみに許可しており、かつノード再起動で IP が変わらないため（[04章 §4.2.3](04_監視・ログ・通知基盤.md)）
- 新しいエクスポーターのポートを開ける場合は、対象ノードのロールに UFW 許可タスクを追加する（監視ノード Tailscale IP と緊急 LAN のみ許可）

編集後は §2.2 と同じく `--tags monitoring` で再適用する。反映確認は `/targets` で新ターゲットが UP になることを見る。

## 3. Ansible

構成管理の中核。実行は原則として統合管理スクリプト [run_playbook.sh](../iac-workspace/run_playbook.sh) 経由で行うが、開発・デバッグ時は直接 `ansible-playbook` を使う。

### 3.1 Playbook / Role の記述規則
- ロールは責務ごとにカプセル化する（一覧は [iac-workspace/README.md §4.3](../iac-workspace/README.md) と [ディレクトリ構造リファレンス.md](ディレクトリ構造リファレンス.md)）
- 静的ファイルは `ansible/files/<app>/`、Vault 変数を流し込む雛形は `ansible/templates/<app>/*.j2` に分離する（[05章 共通設計原則](05_アプリのコンテナ化と運用.md)）
- 冪等性を保つ。再実行で既存データを破壊しないよう、データ投入系は `stat` で存在確認してから `when:` でガードする（[05章 §5.2.2](05_アプリのコンテナ化と運用.md) / §5.4.2）
- `.ansible-lint` の `profile: production` に準拠する（[07章 §7.2.3](07_CI（コード品質と静的検証）.md)）
- Phase 4 は `strategy: free` で並列実行されるため、`pause` / `serial` は使えない。停止が必要な箇所は `fail` で代替する（[08章 §8.4.3](08_CD（自動デプロイ）.md)）

### 3.2 Vault 運用
機密変数ファイルの暗号化・編集（[03章 §3.1](03_機密情報管理.md)）。

```bash
# 暗号化
ansible-vault encrypt iac-workspace/local_config/ansible/credentials/vault.yml

# 中身の編集（自動で復号→再暗号化）
ansible-vault edit iac-workspace/local_config/ansible/credentials/vault.yml

# 一時的に平文表示
ansible-vault view iac-workspace/local_config/ansible/credentials/vault.yml

# 復号（平文に戻す。取り扱い注意）
ansible-vault decrypt iac-workspace/local_config/ansible/credentials/vault.yml
```

run_playbook.sh は `ANSIBLE_VAULT_PASSWORD_FILE` を自動設定するため対話入力は不要。手動実行時は `-e @<vault.yml>` で渡す。

### 3.3 個別反映とタグ
特定ロールだけを反映する場合は `--tags` を使う。

```bash
cd iac-workspace
ansible-playbook -i local_config/ansible/inventory.ini \
  -e @local_config/ansible/vars.yml \
  -e @local_config/ansible/credentials/vault.yml \
  ansible/site.yml --tags web
```

主なタグ: `common`, `proxy`, `exporters`, `monitoring`, `ups_exporter`, `postgres`, `minecraft`, `web`, `ubsleepy`, `ubsleepy_backup`, `github_runner`。

### 3.4 デバッグ手順
- 構文チェック: `ansible-playbook ... --syntax-check`（run_playbook.sh の「[1] テスト・検証フェーズ」でも実行可能）
- Lint: `ansible-lint ansible/site.yml`
- 模擬実行（Dry Run）: `--check`（一部 systemd タスクは `when: not ansible_check_mode` でスキップされる。[05章 §5.5.5](05_アプリのコンテナ化と運用.md)）
- 詳細ログ: `-v` / `-vvv`（接続レベルまで表示）
- 対象を絞る: `-l shakeserver`（特定ホストのみ）

Web ロール初回適用時は GitHub Deploy Key 未登録だと `fail` で停止し、ログに登録すべき公開鍵が表示される。指示に従い登録後に再実行する（[05章 §5.3.3](05_アプリのコンテナ化と運用.md)）。

## 4. Terraform

Cloudflare（DNS / R2 / Workers）と GitHub Actions Secrets を宣言的に管理する。作業ディレクトリは [iac-workspace/terraform](../iac-workspace/terraform)。

### 4.1 リソースの追加・変更
1. [terraform/main.tf](../iac-workspace/terraform/main.tf) にリソース定義を追記する
2. フォーマットと検証:
   ```bash
   cd iac-workspace/terraform
   terraform fmt
   terraform validate
   ```
3. 差分確認:
   ```bash
   terraform plan -var-file=../local_config/terraform/terraform.tfvars
   ```
4. 適用:
   ```bash
   terraform apply -var-file=../local_config/terraform/terraform.tfvars
   ```

PR 経由で変更する場合、CI が `terraform plan` の結果を PR コメントに自動投稿するため、apply 前に影響範囲を確認できる（[07章 §7.4](07_CI（コード品質と静的検証）.md)）。

### 4.2 GitHub Actions Secrets の更新
Secrets は Terraform GitHub Provider が `file()` でローカルファイルを参照して登録する。id_rsa や .vault_pass を更新した場合は、ファイルを差し替えて `terraform apply` を再実行するだけで GitHub 側にも同期される（[03章 §3.2.7](03_機密情報管理.md)）。

### 4.3 state 管理（Remote State）
状態ファイルは Cloudflare R2 にリモート保存される（[06章 §6.5](06_高可用性とシステム堅牢化.md)）。

- 認証情報はコードに書かず [local_config/terraform/backend.hcl](../iac-workspace/local_config/terraform/backend.hcl)（Git 管理外）に置く
- 初回のローカル→R2 移行:
  ```bash
  cd iac-workspace/terraform
  terraform init -backend-config=../local_config/terraform/backend.hcl -migrate-state
  ```
- 以降は R2 上の state が正となる。state の手動編集（`terraform state` サブコマンド）は最終手段とし、原則 plan/apply で収束させる

### 4.4 Drift（乖離）検知
Terraform 管理外でクラウド側に手動変更が入ると state と実態が乖離する。週次 Drift 検知ワークフローが毎週月曜 AM9:00(JST) に `terraform plan -detailed-exitcode` を実行し、差分があれば Discord に通知する（[06章 §6.5.4](06_高可用性とシステム堅牢化.md)）。緊急時は GitHub の Actions タブから手動実行できる。

### 4.5 フェイルオーバーの手動切り替え
Workers Route の有効/無効は `maintenance_mode` 変数で制御される。手動切り替えは専用ラッパーを使う（[06章 §6.1.4](06_高可用性とシステム堅牢化.md)）。

```bash
./iac-workspace/toggle_maintenance.sh on    # メンテ画面に切替
./iac-workspace/toggle_maintenance.sh off   # 通常運用に戻す
```

## 5. Tailscale

3 ノードをつなぐメッシュ VPN。構築は [02章 §2.2](02_ネットワーク基盤.md)。

### 5.1 ノードの追加
1. Tailscale Admin Console で Reusable な Auth Key を発行（既存キーが有効なら再利用可。[02章 §2.2.2](02_ネットワーク基盤.md)）
2. install_tailscale.yml の `tailscale_authkey` を設定し、対象ノードへ適用
   ```bash
   cd iac-workspace/ansible
   ansible-playbook -i local_config/ansible/inventory.ini playbooks/install_tailscale.yml -l <新ノード>
   ```
3. 実行後、inventory.ini の該当行が Tailscale IP に自動置換される（[02章 §2.2.5](02_ネットワーク基盤.md)）
4. 監視対象に加える場合は prometheus.yml にターゲット追記（→§2.3）、UFW 許可を該当ロールに追加する

### 5.2 ACL 設定
ノード間のアクセス制御は Tailscale Admin Console の Access Controls（ACL）で定義する。本基盤では UFW のソース IP 制限（Tailscale 帯 `100.64.0.0/10`）を主たる制御層としているため、ACL は最小権限の補助として扱う。ACL を厳格化する場合は、監視ノードからアプリノードのメトリクスポートへの到達性を壊さないよう注意する。

### 5.3 障害時の接続確認
```bash
# 各ノードの接続状態と Tailscale IP を確認
tailscale status

# 自ノードの Tailscale IP を確認
tailscale ip -4

# 特定ノードへの疎通
tailscale ping <相手ノードの Tailscale IP>
```

確認のポイント:
- ノードがオフラインの場合、まず物理電源と LAN、次に `systemctl status tailscaled` を確認する
- proxy ノード（大学拠点）は MagicDNS を無効化しているため、名前解決は学内 DNS に依存する（[02章 §2.3.2](02_ネットワーク基盤.md)）
- Tailscale 全体がダウンした場合の最終手段として、自宅 LAN（`192.168.10.0/24`）からの直接 SSH / HTTPS（8443 ポート）が常時開放されている（[02章 §2.3.1](02_ネットワーク基盤.md) / §2.4）

## 6. バックアップ・リストア

データ保管先は Cloudflare R2（バケット `shakeserver-backup`）。バックアップは自動取得され、復元能力は月次のリストアテストで検証する。

### 6.1 バックアップの仕組み（取得側）
| 対象 | 仕組み | 保存先 | 詳細 |
|---|---|---|---|
| PostgreSQL | eeshugerman/postgres-backup-s3 コンテナ（日次） | `s3://shakeserver-backup/db-backups/<db>_<timestamp>.dump.gz` | [05章 §5.1.3](05_アプリのコンテナ化と運用.md) |
| Minecraft ワールド | itzg/mc-backup コンテナ（24h ごと、rclone） | `s3://shakeserver-backup/minecraft-backups/` | [05章 §5.2.1](05_アプリのコンテナ化と運用.md) |
| Discord Bot データ | systemd timer + backup_ubsleepy_r2.sh（日次、最大 7 世代） | R2（ubsleepy 用バケット） | [05章 §5.5](05_アプリのコンテナ化と運用.md) |

PostgreSQL は `pg_dump --format=custom`（バイナリ）+ gzip で保存されるため、復元には `pg_restore` を使う（`psql < file` では復元できない）（[06章 §6.6.3](06_高可用性とシステム堅牢化.md)）。

### 6.2 リストアテスト（月次・検証）
本番に影響を与えずに「確実に復元できるか」を検証するワンショット Playbook [ansible/playbooks/test_restore.yml](../iac-workspace/ansible/playbooks/test_restore.yml) を月初に実行する（[06章 §6.6](06_高可用性とシステム堅牢化.md)）。

```bash
cd iac-workspace
ansible-playbook -i local_config/ansible/inventory.ini \
  -e @local_config/ansible/vars.yml \
  -e @local_config/ansible/credentials/vault.yml \
  ansible/playbooks/test_restore.yml
```

このテストは、R2 から最新の `.dump.gz` を取得し、使い捨ての `postgres:15-alpine` コンテナへ `pg_restore` でリストアし、テーブル一覧を確認した上で Discord に結果通知し、テストコンテナと作業ディレクトリを必ず後片付けする（[06章 §6.6.2](06_高可用性とシステム堅牢化.md)）。

必要な Vault 変数（[06章 §6.6.4](06_高可用性とシステム堅牢化.md)）: `r2_access_key_id` / `r2_secret_access_key` / `r2_endpoint` / `ubsleepy_backup_discord_webhook`。

### 6.3 本番への復元手順の考え方
実機への本番復元は通常運用では発生しないが、必要時は以下の流れになる。

1. 対象サービスのコンテナを停止する（必要に応じて [安全シャットダウン](06_高可用性とシステム堅牢化.md) を使用）
2. R2 から対象の `.dump.gz` をダウンロードして展開する（リストアテストと同じ取得ロジック）
3. PostgreSQL は稼働中コンテナに対し `pg_restore` を実行する
4. Minecraft は `world/` ディレクトリを差し替える前に該当コンテナを停止し、アーカイブを展開する（minecraft ロールの初回リストアと同じ要領。[05章 §5.2.2](05_アプリのコンテナ化と運用.md)）

本番復元は破壊的操作のため、事前に現状のバックアップを取得し、テストコンテナでの復元確認（§6.2）を済ませてから実施する。
