# IaC手順書 ドキュメント作成ガイドライン

本ディレクトリ（docs/）配下のIaC構築・運用手順書を作成・更新する際の統一基準を定めます。他の作業者が上から手順を追っていけば、全く同じ環境を構築できる再現性の高いマニュアルとなるように記述してください。近しい関連する内容は、複数のドキュメントに分散させず、同じドキュメント内にまとめるようにします。

## 1. 全体構成の基準

各手順書（ファイル）は、原則として以下の流れで構成します。

1. タイトル
   「# 章番号. 章のタイトル」を記述します。
2. 目的・導入理由
   「〇〇の目的」や「〇〇導入理由」といった見出しで、その章でどのような技術を導入し、なぜ必要なのか（目的・メリット）を冒頭で客観的に説明します。
3. この章で追加されるディレクトリ・ファイル構成
   「### ディレクトリ構成」などの見出しで、その章で新しく追加・変更されるファイル構成図を配置します。変更が加わる箇所には末尾に「*」を付与するなどして影響範囲を可視化します。
4. 利用可能になるコマンド・サービス
   その章の手順を完了することで実行可能になるコマンド（Ansible Playbookの実行例やスクリプトの実行方法など）や、アクセス可能になるサービス（URLリンクなど）を明記します。
5. 手順セクション
   「### [章番号].[連番] [ステップ名]」の形式で中見出しを作成し、時系列順に具体的な作業手順を記述します。

## 2. 記述とフォーマットの基準

1. 文体
   客観的な記述で統一します。
2. コードと設定ファイル
   コードブロックには必ず言語（yaml, bash, hcl など）を指定します。
   対象ファイルが明確になるよう、コードブロックの直前にファイルパス（例: iac-workspace/ansible/site.yml）を記述します。
3. 装飾の禁止
   プロジェクト規定に基づき、絵文字の使用、およびマークダウンによる文字の強調（太字、斜体など）は使用しません。
4. 具体例の抽象化
   ユーザーが独自のカスタマイズを行える領域（例: メンテナンス画面のデザイン、Webサイトの内部実装など）については、具体的な内容を記述せず、仕組みや枠組みの解説に留めます。
