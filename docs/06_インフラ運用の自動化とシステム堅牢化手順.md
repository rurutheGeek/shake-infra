本フェーズでは、現在のインフラ構成管理（IaC）リポジトリに対して商用レベルのベストプラクティスを導入し、運用負荷の軽減、コード品質の維持、および致命的な障害（ステート喪失）の防止を実現する。

### 6.1 Ansible Lintの導入と品質改善

【導入する目的】
PlaybookやRoleの記述には様々な書き方があるが、属人化を防ぎ、将来的なエラー（非推奨モジュールの使用や、冪等性が担保されないタスクなど）を未然に防ぐためには、コードの構文チェック（Lint）が不可欠である。Ansible Lintを導入することで、常に公式のベストプラクティスに沿った高品質なコードを維持することが可能になる。

1. 作業用PCのUbuntuターミナルで、Pythonパッケージマネージャ uv を用いて ansible-lint をインストールする。
```bash
uv pip install ansible-lint
```

2. プロジェクトのルートディレクトリ（[iac-workspace](../iac-workspace)）に、Lintの設定ファイル [.ansible-lint](../iac-workspace/.ansible-lint) を作成する。
```bash
cd ~/iac-workspace
```
.ansible-lint
```yaml
profile: production
exclude_paths:
  - .github/
warn_list:
  - experimental
```

3. 以下のコマンドを実行し、プロジェクト内のすべてのAnsibleファイルに対して構文チェックを実行する。エラーが表示された場合は、警告内容に従ってPlaybookやRoleの記述を修正する。
```bash
ansible-lint ansible/
```

### 6.2 自動セキュリティアップデートの導入（Unattended Upgrades）

【導入する目的】
Raspberry PiなどのLinuxサーバーをインターネットやVPNに接続して長期間運用する場合、日々発見されるOSやパッケージの脆弱性に対応する必要がある。手動で定期的に apt update && apt upgrade を実行するのは運用負荷が高く、更新漏れのリスクもある。unattended-upgrades パッケージを導入することで、セキュリティ関連のパッチのみを全自動で適用し、ゼロデイ攻撃等のリスクを最小化することができる。

1. 全ノードに適用される [roles/common/tasks/main.yml](../iac-workspace/ansible/roles/common/tasks/main.yml) に、自動アップデート用のパッケージインストールと設定ファイルの配置タスクを追記する。

[ansible/roles/common/tasks/main.yml](../iac-workspace/ansible/roles/common/tasks/main.yml) （追記）
```yaml
- name: 自動アップデートパッケージのインストール
  ansible.builtin.apt:
    name:
      - unattended-upgrades
      - apt-listchanges
    state: present

- name: 自動アップデートと深夜の再起動設定を適用
  ansible.builtin.copy:
    dest: /etc/apt/apt.conf.d/50unattended-upgrades
    content: |
      Unattended-Upgrade::Allowed-Origins {
          "${distro_id}:${distro_codename}-security";
          "${distro_id}ESMApps:${distro_codename}-apps-security";
          "${distro_id}ESM:${distro_codename}-infra-security";
      };
      Unattended-Upgrade::Automatic-Reboot "true";
      Unattended-Upgrade::Automatic-Reboot-Time "02:00";
    mode: "0644"
```
※この設定により、セキュリティパッチ適用後にカーネル更新などで再起動が必要な場合、午前2時に自動的に再起動されるようになる。

2. 変更したRoleをすべてのサーバーへ適用する。
```bash
ansible-playbook -i local_config/ansible/inventory.ini ansible/site.yml
```

### 6.3 Terraformステートのクラウド移行（Remote State）

【導入する目的】
これまでTerraformが管理するインフラの状態（CloudflareのDNS設定など）は、作業用PCのローカルファイル（terraform.tfstate）に保存されていた。もし作業用PCが故障したり、誤ってファイルを削除した場合、Terraformはインフラの現状を把握できなくなり、最悪の場合クラウド上のリソースを手動で作り直す必要がある。
このステートファイルを、すでに作成済みのCloudflare R2（S3互換オブジェクトストレージ）に保存（Remote State）することで、PCの故障リスクを排除し、安全にインフラ管理を継続することができる。

1. [terraform/main.tf](../iac-workspace/terraform/main.tf) を開き、最上部の terraform ブロック内に backend "s3" の設定を追記する。（YOUR_CLOUDFLARE_ACCOUNT_ID は自身の環境に合わせて変更する）

[terraform/main.tf](../iac-workspace/terraform/main.tf) （変更）
```tf
terraform {
  required_providers {
    cloudflare = {
      source  = "cloudflare/cloudflare"
      version = "~> 4.0"
    }
  }

  backend "s3" {
    bucket                      = "shakeserver-backup"
    key                         = "terraform/state.tfstate"
    region                      = "auto"
    endpoints                   = { s3 = "https://YOUR_CLOUDFLARE_ACCOUNT_ID.r2.cloudflarestorage.com" }
    skip_credentials_validation = true
    skip_region_validation      = true
    skip_requesting_account_id  = true
    skip_metadata_api_check     = true
    skip_s3_checksum            = true
  }
}
```

2. R2にアクセスするための認証情報（アクセスキーとシークレットキー）を記述した設定ファイル backend.hcl を local_config/terraform フォルダ内に作成する。
※このファイルには機密情報が含まれるため、絶対にGit等のバージョン管理システムにコミットしてはならない。

local_config/terraform/backend.hcl
```hcl
access_key = "YOUR_R2_ACCESS_KEY"
secret_key = "YOUR_R2_SECRET_KEY"
```

3. 以下のコマンドを実行し、ローカルのステートファイルをCloudflare R2上へマイグレーション（移行）する。途中で「本当に移行するか？」と聞かれたら yes と入力する。
```bash
cd ~/iac-workspace/terraform
terraform init -backend-config=../local_config/terraform/backend.hcl -migrate-state
```

### 6.4 電源ライフサイクル管理の IaC 自動化

【導入する目的】
複数台の Raspberry Pi やコンテナが稼働するインフラ環境において、メンテナンス時等に安全にシャットダウンまたは再起動を実行するための IaC 資材を整備する。手動で各ホストにSSHログインし順番にコンテナを落としてOSを終了する手順は、操作ミス（手順の誤りによるデータベース破損など）や運用の非効率化を招く。これを防ぐため、データベースや稼働コンテナを安全停止させたうえで電源を制御する自動化プレイブックおよび統合スクリプトを構築する。

1. ホストごとの役割や起動順序を考慮し、コンテナの正常終了およびホストOSのシャットダウンを記述したプレイブックを作成する。

[ansible/playbooks/shutdown_servers.yml](../iac-workspace/ansible/playbooks/shutdown_servers.yml)
```yaml
---
- name: 全サーバーの安全なシャットダウン（コンテナ安全停止からOSシャットダウンまで自動実行）
  hosts: all
  become: true
  tasks:
    - name: Minecraftコンテナを安全に停止
      community.docker.docker_container:
        name: shake_minecraft-pixelmon1.21.1
        state: stopped
      when: "'home_node1' in group_names"
      failed_when: false

    - name: PostgreSQL関連コンテナを安全に停止
      community.docker.docker_container:
        name: "{{ item }}"
        state: stopped
      loop:
        - shake_postgres
        - shake_adminer
        - shake_postgres-backup
      when: "'home_node1' in group_names"
      failed_when: false

    - name: 監視スタックコンテナを安全に停止
      community.docker.docker_container:
        name: "{{ item }}"
        state: stopped
      loop:
        - prometheus
        - grafana
        - blackbox_exporter
      when: "'home_node2' in group_names"
      failed_when: false

    - name: ホストOSをシャットダウン
      ansible.builtin.command: shutdown -h now
      async: 1
      poll: 0
      failed_when: false
```

2. 統合管理スクリプト [run_playbook.sh](../iac-workspace/run_playbook.sh) の「大項目 [3] サーバー電源・ライフサイクル操作」を通じて、対象ノードのシャットダウンまたは再起動を対話式に実行する。
- シャットダウン実行時には、事故防止のための最終確認プロンプトが表示され、肯定応答かつ Ansible Vault パスワードを入力することで初めて実行される。
- 再起動実行時には、アドホックに ansible の reboot モジュールを呼び出す。その際、Tailscale VPN ネットワークの切断による無限待機（ハングアップ）を防止するため、非同期オプション -B 2 -P 0 を付与し、再起動命令の送信完了後に即時に制御が返るよう最適化されている。

### 6.5 サーバー起動・停止の Discord 自動通知（Systemd ライフサイクル監視サービス）

【導入する目的】
ホストOSの予期せぬシャットダウン（停電や電源ケーブルの誤脱落などによる異常終了）を検知し、復旧起動時に自動的に Discord へ警告通知を送信する仕組みを導入する。Ansible プレイブックのトリガーに依存せず、ホストOS自体の Systemd 起動・停止プロセスと完全に同期させて動作させることで、100% 確実なライフサイクル監視を実現する。

local_config/ansible/vars.yml
```yaml
# Discord電源状態通知用のWebhookマッピング
discord_webhook_url: "{{ ubsleepy_backup_discord_webhook }}"
```

1. 起動時および停止時の状態遷移を自動判別し、 Discord Webhook 経由で Embed メッセージを送信するシェルスクリプトのテンプレートを作成する。

[ansible/templates/common/server-lifecycle-notify.sh.j2](../iac-workspace/ansible/templates/common/server-lifecycle-notify.sh.j2)
```bash
#!/bin/bash
WEBHOOK_URL="{{ discord_webhook_url }}"
HOSTNAME="$(hostname)"
MARK_FILE="/var/lib/lifecycle/normal_shutdown"

if [ -z "$WEBHOOK_URL" ] || [ "$WEBHOOK_URL" = "https://discordapp.com/api/webhooks/YOUR_DUMMY_WEBHOOK_URL" ]; then
    exit 0
fi

send_discord_message() {
    local color="$1"
    local title="$2"
    local description="$3"
    local payload=$(cat <<EOF
{
  "embeds": [
    {
      "title": "${title}",
      "description": "${description}",
      "color": ${color},
      "timestamp": "$(date -u +%Y-%m-%dT%H:%M:%SZ)"
    }
  ]
}
EOF
)
    curl -s -H "Content-Type: application/json" -X POST -d "$payload" "$WEBHOOK_URL" > /dev/null
}

case "$1" in
    start)
        if [ -f "$MARK_FILE" ]; then
            send_discord_message 3066993 "正常起動" "サーバー ${HOSTNAME} が正常に起動しました。"
            rm -f "$MARK_FILE"
        else
            send_discord_message 15158332 "予期せぬ起動" "サーバー ${HOSTNAME} が予期せぬシャットダウン（強制終了または停電など）から起動しました。前回の停止処理が正常に完了しなかった可能性があります。"
        fi
        ;;
    stop)
        mkdir -p "$(dirname "$MARK_FILE")"
        touch "$MARK_FILE"
        send_discord_message 15105570 "安全なシャットダウン・再起動" "サーバー ${HOSTNAME} が安全なシャットダウンまたは再起動シーケンスを開始しました。"
        ;;
esac
exit 0
```

2. ネットワークおよび VPN（Tailscale）が完全に開通した後に Discord 送信が試行されるよう制御した Systemd サービス定義ファイルを作成する。

[ansible/templates/common/server-lifecycle.service.j2](../iac-workspace/ansible/templates/common/server-lifecycle.service.j2)
```ini
[Unit]
Description=Server Lifecycle Discord Notification Service
After=network-online.target tailscaled.service
Wants=network-online.target tailscaled.service

[Service]
Type=oneshot
RemainAfterExit=yes
ExecStart=/usr/local/bin/server-lifecycle-notify.sh start
ExecStop=/usr/local/bin/server-lifecycle-notify.sh stop

[Unit]
[Install]
WantedBy=multi-user.target
```

3. 共通設定ロール common のタスクに、上記資材のデプロイとサービスの起動・自動起動有効化設定を追加し、インフラ展開時にすべてのサーバーに自動的に本監視機能がインストールされるようにする。

### 6.6 ホストOSのスワップ完全無効化による長寿命化（SWAP OFF）

【導入する目的】
Raspberry Pi サーバーにおいて、標準でオンになっている SWAP（メモリ不足時にストレージをメモリの代替として代用する機能）は、マイクロSDカードやSSDに対して微細なランダム書き込みを極めて頻繁に発生させる。これは物理セルの摩耗（ストレージ寿命 of 著しい低下）を招き、自宅インフラの信頼性を毀損する。また、スワップ発生時のディスクI/Oのボトルネックはシステム全体の応答性能を著しく低下させる。機器の寿命最大化と稼働性能の安定化を図るため、インフラ配下のすべてのホストでスワップ設定を恒久的に無効化する。

1. 共通ロール common のタスクにおいて、スワップ管理パッケージのアンインストール、メモリの即時解放、および再起動後に有効化されないようにするための設定除去を実施する。

[ansible/roles/common/tasks/main.yml](../iac-workspace/ansible/roles/common/tasks/main.yml) （追記）
```yaml
- name: Raspberry Pi 標準の dphys-swapfile サービスの停止と無効化
  ansible.builtin.systemd:
    name: dphys-swapfile
    enabled: false
    state: stopped
  failed_when: false

- name: 現在有効なすべてのスワップの即時無効化 (swapoff)
  ansible.builtin.command: swapoff -a
  changed_when: false

- name: fstab からのスワップ設定の完全排除
  ansible.builtin.replace:
    path: /etc/fstab
    regexp: '^([^#].*\sswap\s.*)$'
    replace: '# \1'

- name: 不要な dphys-swapfile パッケージの削除
  ansible.builtin.apt:
    name: dphys-swapfile
    state: absent
    purge: true
```
※この一連のタスクにより、現在稼働中のスワップ領域が即座に解放されるだけでなく、 /etc/fstab 内の定義が自動でコメントアウトされ、再起動後も一切スワップが生成・割り当てされない強固な IaC 制御が自動展開される。

### Terraform Drift 検知（週次自動チェック）

コードと実際のインフラ状態が乖離（drift）した場合に自動検知する仕組みを導入している。手動でインフラを変更した場合や障害復旧の過程でコードと実態が変わった場合に早期発見できる。

#### 仕組み

[.github/workflows/terraform_drift.yml](../.github/workflows/terraform_drift.yml) が毎週月曜 AM9:00 (JST) に `terraform plan` を実行する。

- `exit code 0`: 差分なし → 正常
- `exit code 2`: 差分あり → Discord に DRIFT DETECTED 通知
- `exit code 1`: エラー → Discord に ERROR 通知

#### 必要なシークレット

| シークレット名 | 内容 |
|---|---|
| `CLOUDFLARE_API_TOKEN` | Cloudflare プロバイダー用 |
| `CF_R2_ACCESS_KEY_ID` | R2 バックエンド認証 |
| `CF_R2_SECRET_ACCESS_KEY` | R2 バックエンド認証 |
| `TF_BACKEND_CONFIG` | backend.hcl の全内容（endpoint/bucket/key/region） |
| `DISCORD_WEBHOOK_URL` | 通知先 Webhook |

これらのシークレットは Terraform GitHub Provider で管理されているため、`terraform.tfvars` に値を追記して `terraform apply` で反映する。

#### 手動実行

GitHub の Actions タブから **Terraform Drift Detection** workflow を選択し、**Run workflow** で即時実行可能。

### GitHub Actions セルフホストランナーの監視

GitHub Actions のセルフホストランナー（tarakoserver）がオフラインになったことを Prometheus で自動検知しアラートを送信する仕組みを導入している。

#### 仕組み

監視サーバーの blackbox_exporter がランナーのステータス（ヘルスチェックエンドポイントなど）を定期的に監視し、5分以上疎通が失敗した場合にアラートを発火する。

設定ファイルは [prometheus_rules.yml](../iac-workspace/ansible/files/monitoring/prometheus_rules.yml) に定義されている。

```yaml
  - name: runner.rules
    rules:
      - alert: GithubRunnerOffline
        expr: probe_success{job="blackbox", instance=~".*tarakoserver.*"} == 0
        for: 5m
        labels:
          severity: warning
        annotations:
          summary: "GitHub Actions runner (tarakoserver) may be offline"
          description: "tarakoserver への疎通が5分以上失敗しています。Self-hosted runner が停止している可能性があります。"
```

アラートが発生した場合、Discord通知用Webhookを経由して管理者宛に通知が送信される。
