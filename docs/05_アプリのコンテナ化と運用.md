# 5. アプリのコンテナ化と運用

## アプリ運用の目的
メインサーバー（home_node1）で稼働させる 4 系統のアプリケーション（PostgreSQL / Minecraft / Web + Alexa / Discord Bot）を、すべて Docker コンテナとして展開し、Ansible Role で冪等にデプロイ可能な状態にする。
すべてのアプリは Phase 4 の strategy:free（並列実行）に乗っており、相互に依存しない設計となっている。

### 共通設計原則
すべての app ロールは以下のパターンを踏襲する。

1. **専用システムユーザーの作成**（postgres, minecraft, www-data 等）。コンテナを非 root UID で動かし、ホスト OS との特権分離を確保
2. **専用ディレクトリの所有権設定**（`/opt/<app>` または `/var/www/<app>`、mode 0750）
3. **設定の二層管理**: `files/` は静的ファイル（docker-compose 等）、`templates/` は Vault 変数を流し込む `.env`
4. **データの永続化**: Docker named volume または bind mount で、`docker compose down -v` 以外では消えない
5. **冪等性**: 同じ Playbook を再実行しても既存データを破壊しない（ロールバック防止）

### Docker 環境の前提（docker ロール）
本リポジトリでは Docker のインストール自体を独立した `docker` ロールに切り出し、Phase 3 で home_node1 / home_node2 に並列で適用している。アプリ系ロール（postgres, minecraft, web, ubsleepy）は Docker が導入済みであることを前提とする。

[ansible/roles/docker/tasks/main.yml](../iac-workspace/ansible/roles/docker/tasks/main.yml)（要約）
- 依存パッケージ（ca-certificates, curl, gnupg, python3-debian）を導入
- Docker 公式 GPG キーを `/etc/apt/keyrings/docker.asc` に取得
- `deb822_repository` モジュールで Docker 公式リポジトリを arm64 アーキで追加
- `docker-ce`, `docker-ce-cli`, `containerd.io`, `docker-buildx-plugin`, `docker-compose-plugin` を一括導入
- `docker` サービスを enabled + started

### 5.1 PostgreSQL コンテナ

#### 5.1.1 採用構成
| 項目 | 値 |
|---|---|
| イメージ | `postgres:15-alpine` |
| コンテナ名 | `shake_postgres` |
| データ永続化 | `./data:/var/lib/postgresql/data`（bind mount） |
| 初期化スクリプト | `./init-db:/docker-entrypoint-initdb.d`（初回起動時のみ実行） |
| 並走サービス | `adminer`（WebUI、ポート 8080） / `shake_postgres-backup`（eeshugerman/postgres-backup-s3 で R2 へ日次バックアップ） |
| 実行 UID/GID | ホストの `postgres` システムユーザーを Docker user オプションで指定 |

#### 5.1.2 初期化スクリプト（一度きり実行）
コンテナの `docker-entrypoint-initdb.d` 機構を利用し、データディレクトリが空の場合のみ初回起動時に SQL を実行する。これにより**Ansible 再実行時のロールバック（ユーザー上書き）を構造的に防止**する。

[ansible/files/postgres/init-db/01_setup_users.sh](../iac-workspace/ansible/files/postgres/init-db/01_setup_users.sh)
```bash
#!/bin/bash
set -e

psql -v ON_ERROR_STOP=1 --username "$POSTGRES_USER" --dbname "$POSTGRES_DB" <<-EOSQL
    CREATE USER ${PKDB_READER_USER} WITH PASSWORD '${PKDB_READER_PASSWORD}';
    GRANT CONNECT ON DATABASE ${POSTGRES_DB} TO ${PKDB_READER_USER};
    GRANT USAGE ON SCHEMA public TO ${PKDB_READER_USER};
    GRANT SELECT ON ALL TABLES IN SCHEMA public TO ${PKDB_READER_USER};
    ALTER DEFAULT PRIVILEGES IN SCHEMA public GRANT SELECT ON TABLES TO ${PKDB_READER_USER};

    CREATE USER ${PKDB_EDITOR_USER} WITH PASSWORD '${PKDB_EDITOR_PASSWORD}';
    GRANT CONNECT ON DATABASE ${POSTGRES_DB} TO ${PKDB_EDITOR_USER};
    GRANT USAGE ON SCHEMA public TO ${PKDB_EDITOR_USER};
    GRANT SELECT, INSERT, UPDATE, DELETE ON ALL TABLES IN SCHEMA public TO ${PKDB_EDITOR_USER};
    ALTER DEFAULT PRIVILEGES IN SCHEMA public GRANT SELECT, INSERT, UPDATE, DELETE ON TABLES TO ${PKDB_EDITOR_USER};
EOSQL
```

reader / editor の 2 ユーザーを最小権限で作成する。Web ロール（quiz_app）からは reader、編集系アプリからは editor を使い分ける。

#### 5.1.3 自動バックアップ（db-backup コンテナ）
postgres compose には `eeshugerman/postgres-backup-s3:15` を組み込み済み。

| 環境変数 | 値 | 効果 |
|---|---|---|
| `SCHEDULE` | `@daily` | 毎日 1 回 pg_dump 実行 |
| `S3_ENDPOINT` | `https://${CF_ACCOUNT_ID}.r2.cloudflarestorage.com` | Cloudflare R2 を S3 互換 API として利用 |
| `S3_BUCKET` | `shakeserver-backup` | バックアップ先バケット |
| `S3_PREFIX` | `db-backups` | バケット内のプレフィックス |

→ R2 上には `s3://shakeserver-backup/db-backups/<db>_<timestamp>.dump.gz` 形式で蓄積される。月次のリストアテストはこれを元に行う（[06_高可用性とシステム堅牢化.md](06_高可用性とシステム堅牢化.md) §6.6 参照）。

#### 5.1.4 Ansible Role
[ansible/roles/postgres/tasks/main.yml](../iac-workspace/ansible/roles/postgres/tasks/main.yml) の主要構造:
1. システムユーザー `postgres` を `nologin` シェルで作成し、`docker` グループへ追加
2. `/opt/postgres/{data,init-db}` を所有権 `postgres:postgres`、mode 0750 で作成
3. `files/postgres/` の静的ファイル一式を `/opt/postgres/` に同期
4. `templates/postgres/env.j2` を Vault 変数で展開して `.env` を生成（mode 0600）
5. `docker_compose_v2` モジュールで `state: present`, `build: always`, `remove_orphans: true` で起動

[ansible/templates/postgres/env.j2](../iac-workspace/ansible/templates/postgres/env.j2) — Vault からの機密変数注入
```env
POSTGRES_USER={{ postgres_user }}
POSTGRES_PASSWORD={{ postgres_password }}
POSTGRES_DB={{ db_name }}
PKDB_READER_USER={{ db_user_reader }}
PKDB_READER_PASSWORD={{ db_password_reader }}
PKDB_EDITOR_USER={{ db_user_editor }}
PKDB_EDITOR_PASSWORD={{ db_password_editor }}

CF_ACCOUNT_ID={{ cf_account_id }}

```

#### 5.1.6 パスワードの冪等同期（initdb 後のドリフト是正）

アプリ用ユーザー（pkdb_reader / pkdb_editor）のパスワードは、初期化スクリプト `init-db/01_setup_users.sh` が「データボリュームの初回作成時にだけ」設定する。そのため Vault 側のパスワードを後から変更しても、既存ボリュームの DB には反映されず乖離する。一方で Web ロールはデプロイのたびに `.env` を Vault 値で再生成するため、ある日のデプロイで `.env` と DB のパスワードが食い違い、認証失敗（`password authentication failed`）に至る。

これを構造的に防ぐため、postgres ロールはコンテナ起動後に、毎回 Vault 値へ `ALTER USER` してパスワードを収束させる。

iac-workspace/ansible/roles/postgres/tasks/main.yml
```yaml
- name: PostgreSQLが接続受付可能になるまで待機
  ansible.builtin.command:
    cmd: docker exec shake_postgres pg_isready -U {{ postgres_user }} -d {{ db_name }}
  register: pg_ready
  until: pg_ready.rc == 0
  retries: 15
  delay: 4
  changed_when: false

- name: アプリDBユーザーのパスワードを Vault 値へ同期（冪等収束）
  ansible.builtin.command:
    cmd: docker exec -i shake_postgres psql -U {{ postgres_user }} -d {{ db_name }} -v ON_ERROR_STOP=1
    stdin: |
      ALTER USER "{{ db_user_reader }}" WITH PASSWORD '{{ db_password_reader | replace("'", "''") }}';
      ALTER USER "{{ db_user_editor }}" WITH PASSWORD '{{ db_password_editor | replace("'", "''") }}';
  changed_when: false
  no_log: true
```

補足: pg_hba.conf はループバック（127.0.0.1/32）を `trust`、それ以外を `scram-sha-256` としている。アプリは DB_HOST にホストの LAN IP（127.0.0.1 以外）を用いるため scram 認証経路を通る。接続確認の際に 127.0.0.1 で試すと無検証で通ってしまい誤判定になるため、検証はアプリと同じ非ループバック経路で行うこと。

### 5.2 Minecraft コンテナ

#### 5.2.1 採用構成
| 項目 | 値 |
|---|---|
| イメージ | `itzg/minecraft-server:java21`（NeoForge + Pixelmon 1.21.1） |
| コンテナ名 | `shake_minecraft-pixelmon1.21.1` |
| メモリ割当 | 3 GB |
| RCON | ポート 25575 を `127.0.0.1` のみで公開（外部不可、ローカルからの管理用） |
| バックアップ | 並走する `itzg/mc-backup` コンテナが毎 24h で R2 へ自動アップロード（rclone） |
| リソースパック | URL と SHA1 を `MC_RESOURCE_PACK_URL` / `MC_RESOURCE_PACK_HASH` で指定（クライアントに自動配布） |
| MOD | `CURSEFORGE_FILES` で Pixelmon と Ore Excavation を自動取得 |

#### 5.2.2 ロールバック防止のリストアロジック
Ansible 再実行時に R2 から最新バックアップを取得してデータディレクトリを上書きしてしまうとロールバックになる。これを防ぐため、ロールは以下の条件付きロジックを採用する。

[ansible/roles/minecraft/tasks/main.yml](../iac-workspace/ansible/roles/minecraft/tasks/main.yml)（要点）
```yaml
- name: 移行先Minecraftワールドデータの存在確認
  ansible.builtin.stat:
    path: "{{ mc_dir }}/pixelmon/world"
  register: minecraft_world_stat

- name: R2からのデータリストア処理
  when: not minecraft_world_stat.stat.exists
  block:
    - name: R2から最新のバックアップファイルを特定してダウンロード
      ansible.builtin.shell: |
        LATEST=$(docker run --rm rclone/rclone lsf --sort time --reverse \
                  r2:${CF_BUCKET_NAME}/minecraft-backups/ | head -n 1)
        docker run --rm -v {{ mc_dir }}/tmp:/data \
          rclone/rclone copy r2:${CF_BUCKET_NAME}/minecraft-backups/$LATEST /data/
      args:
        executable: /bin/bash

    - name: アーカイブをpixelmonディレクトリへ展開
      ansible.builtin.unarchive:
        src: "{{ minecraft_backup_archive.files[0].path }}"
        dest: "{{ mc_dir }}/pixelmon/"
        remote_src: true
```

`world/` ディレクトリが既に存在する場合は block 全体がスキップされる。初回構築時のみ R2 からの自動移行が走る。

#### 5.2.3 RCON によるゲーム内操作の自動化
RCON（Remote Console）は Minecraft のサーバーコンソールコマンドを TCP で実行するプロトコル。本リポジトリでは `itzg/minecraft-server` 内蔵の `rcon-cli` を Ansible から `docker exec` で呼び出すことで、IaC とゲーム内操作を統合する。

##### シナリオ A: メンテナンス時の安全な再起動と告知
```yaml
- name: メンテナンス開始の全体告知
  ansible.builtin.command:
    cmd: docker exec shake_minecraft-pixelmon1.21.1 rcon-cli "say 1分後にインフラ更新のためのサーバー再起動を行います。"

- name: 1分間待機
  ansible.builtin.pause:
    minutes: 1

- name: ワールドデータの強制セーブ
  ansible.builtin.command:
    cmd: docker exec shake_minecraft-pixelmon1.21.1 rcon-cli "save-all"
```

##### シナリオ B: ホワイトリストのコード化
```yaml
vars:
  allowed_players:
    - ruruthegeek
    - Baked_sususu

tasks:
  - name: ホワイトリスト機能の有効化
    ansible.builtin.command:
      cmd: docker exec shake_minecraft-pixelmon1.21.1 rcon-cli "whitelist on"

  - name: 定義されたプレイヤーをホワイトリストに一括登録
    ansible.builtin.command:
      cmd: docker exec shake_minecraft-pixelmon1.21.1 rcon-cli "whitelist add {{ item }}"
    loop: "{{ allowed_players }}"
```

RCON でできる主な操作: kick, ban / pardon, op / deop, whitelist, save-all / save-off / save-on, time set, weather, gamerule, say, tellraw など。

### 5.3 Web + Alexa スキル（Web ロール）

#### 5.3.1 採用構成
Web ロールは複数の独立した Git リポジトリを統合管理する。`shake-web` のモノレポを
コンテンツ単位で 3 リポジトリへ分割し、サブドメインごとに配信している（→ shake-web の
`docs/repo-split/`）。単一の Nginx（`shake_web`）が `server_name` でサブドメインを振り分ける。

| リポジトリ | コンテナ | サブドメイン | 役割 |
|---|---|---|---|
| `shake-web` | `shake_web`（Nginx） + `bsquiz`（quiz_app: Node.js） | apex `ruruthegeek.dpdns.org`／`shake.` | apex シェル＋Shake ブランド（Issues/Shaketter/ToBa/Ikura）の静的フロントと API（`quiz_app`） |
| `pkhack` | `pkhack_app`（Next.js） | `pkhack.` | ポケモンクイズ（`/quiz` + クイズ API、socket.io マルチプレイ） |
| `ayahuya` | （静的、`shake_web` がボリューム配信） | `ayahuya.` | アヤフヤ大辞典・技術デモ（純静的） |
| `pokebs_alexaskill` | `alexa_skill` | （`/alexa`） | Alexa スキル用バックエンドコンテナ |

> `pkhack` / `ayahuya` は private リポジトリで、`shake-web` と同じ Deploy Key（`/var/www/.ssh/id_ed25519_deploy`）で clone する。
> 各リポジトリは独立して Git からチェックアウトし、コード変更があったときだけビルド・再起動する Smart 同期を実装している。

#### 5.3.2 Smart 同期: コミット SHA 比較によるリビルド回避
毎回 `git pull` + 全コンテナ再ビルドは非効率かつダウンタイム要因。Web ロールはローカル SHA とリモート SHA を比較し、**差分がある場合のみ pull + 再ビルド**する。

[ansible/roles/web/tasks/main.yml](../iac-workspace/ansible/roles/web/tasks/main.yml)（要点）
```yaml
- name: "SHA確認 [shake-web] ローカル"
  ansible.builtin.command:
    cmd: git rev-parse HEAD
    chdir: "{{ app_dir }}/src_tmp"
  register: web_local_sha
  failed_when: false   # 初回は src_tmp が存在しないため失敗を許容

- name: "SHA確認 [shake-web] リモート"
  ansible.builtin.command:
    cmd: git ls-remote --heads {{ web_repo_url }} main
  register: web_remote_sha
  environment:
    GIT_SSH_COMMAND: "ssh -i /var/www/.ssh/id_ed25519_deploy -o StrictHostKeyChecking=no"

- name: "更新判定 [shake-web]"
  ansible.builtin.set_fact:
    web_code_updated: >-
      {{
        web_force_rebuild | default(false) | bool or
        web_local_sha.rc != 0 or
        (web_remote_sha.stdout | trim).split('\t')[0] != (web_local_sha.stdout | trim)
      }}
```

| 関連変数 | 説明 |
|---|---|
| `web_code_updated` | shake-web の更新要否フラグ |
| `web_alexa_code_updated` | Alexa スキルの更新要否フラグ |
| `web_pkhack_code_updated` | pkhack の更新要否フラグ |
| `web_ayahuya_code_updated` | ayahuya の更新要否フラグ |
| `web_force_rebuild` | `-e web_force_rebuild=true` で強制再ビルド |

> `pkhack` / `ayahuya` のリモート SHA 確認は `shake-web` と同じく `GIT_SSH_COMMAND`（Deploy Key 指定）を伴う。
> `pkhack` は `html`/`quiz` を、`ayahuya` は `html` のみを本番ディレクトリへ rsync する。

#### 5.3.3 GitHub Deploy Key の自動セットアップ
Web ロールは `/var/www/.ssh/id_ed25519_deploy` を初回自動生成し、公開鍵を GitHub の Deploy Keys に登録するよう運用者にメッセージで促す。未登録のままだと `ansible.builtin.fail` で停止し、Playbook 実行ログに公開鍵を表示する。

```yaml
- name: GitHubへのデプロイキー登録確認（未登録時はエラーで公開鍵を表示）
  ansible.builtin.fail:
    msg: |
      ==================================================================
      GitHubへの接続に失敗しました。
      以下の公開鍵をGitHubの「Deploy keys」に登録してから再実行してください。
      {{ web_deploy_pub_key['content'] | b64decode }}
      ==================================================================
  when: "'successfully authenticated' not in web_github_ssh_check.stderr"
```

`ansible.builtin.pause` モジュールは Phase 4 の `strategy: free` 下では使えないため `fail` で代替している（→[08_CD（自動デプロイ）.md](08_CD（自動デプロイ）.md) §strategy:free 注意事項）。

#### 5.3.5 条件付き再起動による最小ダウンタイム
Web ロールは「変更があったコンポーネントだけを再起動」する設計。

    cmd: docker compose up -d --build quiz_app

- name: アプリケーションコード更新に伴うAlexaスキルコンテナの再構築と起動
  ansible.builtin.command:
    cmd: docker compose up -d --build alexa_skill
    chdir: "{{ app_dir }}"
  when: web_alexa_code_updated

- name: アプリケーションコード更新に伴う pkhack コンテナの再構築と起動
  ansible.builtin.command:
    cmd: docker compose up -d --build pkhack_app

  when: web_pkhack_code_updated

- name: 設定変更に伴うNginxコンテナの再起動
  ansible.builtin.command:
    cmd: docker compose restart web

> `ayahuya` は静的のため専用コンテナを持たず、`shake_web`（Nginx）が `./ayahuya/html` をボリュームマウントして配信する。

#### 5.3.6 Nginx 設定: サブドメインごとの server ブロック
単一の `shake_web`（Nginx）が、`conf.d/*.template` の各 `server` ブロックを `server_name` で振り分ける。
いずれも 443 と 8443 を待ち受け、`listen 443 ssl proxy_protocol` で Proxy Protocol を解釈する
（→[02_ネットワーク基盤.md](02_ネットワーク基盤.md) §2.3.4）。CF↔オリジンは SSL モード Full
（オリジン証明書のホスト名検証なし）、ブラウザ↔CF は Universal SSL が `*.ruruthegeek.dpdns.org` をカバーする。

| テンプレート | server_name | 主要マッピング |
|---|---|---|
| [ssl.conf.template](../iac-workspace/ansible/files/web/conf.d/ssl.conf.template) | apex `ruruthegeek.dpdns.org` | `/` → 静的 HTML、`/quiz` → `quiz_app:3000`、`/api/*` → `quiz_app` の `/quiz/api/*`、`/alexa` → `alexa_skill` |
| [shake.conf.template](../iac-workspace/ansible/files/web/conf.d/shake.conf.template) | `shake.` | 静的フロント（issues/shaketter/toba/ikura）＋ `/api/*` → `quiz_app` の `/quiz/api/*`（外向き名前空間） |
| [pkhack.conf.template](../iac-workspace/ansible/files/web/conf.d/pkhack.conf.template) | `pkhack.` | `/quiz` → `pkhack_app:3000`（WebSocket 対応）、`/quiz/index.html`・`/assets` 静的、`/` → `/quiz/` |
| [ayahuya.conf.template](../iac-workspace/ansible/files/web/conf.d/ayahuya.conf.template) | `ayahuya.` | `/usr/share/nginx/ayahuya` を静的配信、`/` → `/docs/words/` |

Shake ブランドの外向き API は `/api/*` に統一し、Next.js の実体パス `/quiz/api/*` へ nginx が書き換える
（後方互換で `/quiz/api/*` も有効）。`redirect.conf.template` は 80 番ポートで受けた全 HTTP リクエストを
`https://` へ 301 リダイレクトする。

> 最終的に apex は旧 URL（`/quiz`→pkhack、`/docs`→ayahuya、`/issues` 等→shake）を 301/308 で各サブドメインへ
> 転送する「シェル」へ縮小する予定（shake-web `docs/repo-split/` Phase D）。

### 5.4 Discord Bot (ubsleepy) コンテナ

#### 5.4.1 採用構成
| 項目 | 値 |
|---|---|
| イメージ | `python:3.13`（Bot コードを volume mount で実行） |
| コンテナ名 | `ubsleepy_bot` |
| 起動方式 | `pip install -r setup/requirements.txt && python main.py` |
| データ | `/opt/ubsleepy/save/`, `config.json`, `resource/pokemon_senryu.csv` を bind mount |
| 再起動ポリシー | `unless-stopped` |

#### 5.4.2 初期データ投入の冪等性
`save/` ディレクトリが存在しない場合のみ、R2 から `app_data.zip` を取得して展開する。再実行時には稼働中のセーブデータを上書きしない。

[ansible/roles/ubsleepy/tasks/main.yml](../iac-workspace/ansible/roles/ubsleepy/tasks/main.yml)（要点）
```yaml
- name: UBSLEEPYのsaveディレクトリの存在を確認
  ansible.builtin.stat:
    path: "{{ ubsleepy_dir }}/save"
  register: ubsleepy_save_dir

- name: R2からapp_data.zipをダウンロード
  ansible.builtin.shell: |
    aws s3 cp s3://{{ ubsleepy_r2_bucket }}/{{ ubsleepy_r2_object_key }} {{ ubsleepy_dir }}/{{ ubsleepy_r2_object_key }} \
      --endpoint-url https://{{ ubsleepy_r2_endpoint }}
  environment:
    AWS_ACCESS_KEY_ID: "{{ ubsleepy_r2_access_key_id }}"
    AWS_SECRET_ACCESS_KEY: "{{ ubsleepy_r2_secret_access_key }}"
  when: not ubsleepy_save_dir.stat.exists

- name: App_data.zipを展開
  ansible.builtin.unarchive:
    src: "{{ ubsleepy_dir }}/{{ ubsleepy_r2_object_key }}"
    dest: "{{ ubsleepy_dir }}"
    remote_src: true
    extra_opts:
      - -O
      - cp932    # 日本語ファイル名対応
  when: not ubsleepy_save_dir.stat.exists
```

`extra_opts: [-O, cp932]` は zip 内の日本語ファイル名を正しく展開するための unzip オプション。古い Windows 環境で作成された zip との互換性確保。

#### 5.4.3 機密トークンの注入
[ansible/templates/ubsleepy/env.j2](../iac-workspace/ansible/templates/ubsleepy/env.j2)
```text
DISCORD_TOKEN={{ ubsleepy_discord_token }}
```

Discord API トークンは Vault の `ubsleepy_discord_token` から動的注入される（→[03_機密情報管理.md](03_機密情報管理.md) §3.1）。

### 5.5 ubsleepy_backup（世代管理付き自動バックアップ + 死活監視）

#### 5.5.1 採用構成
Bot の稼働データ（save/, config.json, pokemon_senryu.csv）を毎日 R2 へ自動バックアップする。systemd timer による定期起動と、コンテナ死活監視機能を組み合わせる。

| コンポーネント | 役割 |
|---|---|
| `backup_ubsleepy_r2.sh` | tar.gz 圧縮、R2 アップロード、世代管理（最大 7 世代）、Discord 通知 |
| `ubsleepy-backup.service` | スクリプトをワンショット実行する systemd ユニット |
| `ubsleepy-backup.timer` | 毎日 0:00 に発火、`Persistent=true` でメンテ中の停止に対応 |

#### 5.5.2 死活監視と Discord アラート色分け
スクリプト実行時に `docker inspect -f '{{.State.Running}}' ubsleepy_bot` で稼働状態をチェック。失敗パターンごとに Discord に色分けで通知する。

| 状態 | 色 | 内容 |
|---|---|---|
| バックアップ成功 | 緑 (32768) | 通常完了通知 |
| コンテナ停止検知 | 橙 (16753920) | `ubsleepy_bot` が稼働していない警告 |
| tar 圧縮失敗 / R2 アップロード失敗 | 赤 (16711680) | エラー通知 |

#### 5.5.3 Docker ソケットへのアクセス権限
systemd タイマーから起動するバックアップサービスは、`docker inspect` を実行するために Docker ソケットへの書き込み権限が必要。Service 定義で `Group=docker` を明示する。

[ansible/roles/ubsleepy_backup/templates/ubsleepy-backup.service.j2](../iac-workspace/ansible/roles/ubsleepy_backup/templates/ubsleepy-backup.service.j2)
```ini
[Unit]
Description=Backup UBSLEEPY save data to Cloudflare R2
Requires=network-online.target
After=network-online.target

[Service]
Type=oneshot
ExecStart=/usr/local/bin/backup_ubsleepy_r2.sh
User=ansible_admin
Group=docker
WorkingDirectory=/opt/ubsleepy
Environment="PATH=/usr/sbin:/usr/bin:/usr/local/sbin:/usr/local/bin:/sbin:/bin"
```

#### 5.5.4 タイマー（Persistent=true による補填実行）
[ansible/roles/ubsleepy_backup/templates/ubsleepy-backup.timer.j2](../iac-workspace/ansible/roles/ubsleepy_backup/templates/ubsleepy-backup.timer.j2)
```ini
[Unit]
Description=Run UBSLEEPY R2 backup daily
Requires=ubsleepy-backup.service

[Timer]
OnCalendar=daily
Persistent=true
RandomizedDelaySec=15m
WakeSystem=true

[Install]
WantedBy=timers.target
```

`Persistent=true` により、深夜 0:00 にサーバーが停止していた場合でも、次回起動時に未実行分が自動補填される。`RandomizedDelaySec=15m` は複数サーバーが同時刻に R2 へアクセスして API レート制限を踏まないように散らすため。

#### 5.5.5 Check Mode（Dry Run）対策
Ansible の `--check` 実行時には、前段のテンプレート配置が実際には適用されないため、systemd タイマーの enable タスクがファイル不在で失敗する。これを防ぐため `when: not ansible_check_mode` を付与する。

```yaml
- name: Systemdタイマーを有効化して開始
  ansible.builtin.systemd:
    name: ubsleepy-backup.timer
    enabled: true
    state: started
  when: not ansible_check_mode
```
