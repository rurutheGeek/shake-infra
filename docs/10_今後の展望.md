## 10.2 自動化の拡張

### 10.2.1 月次リストアテストの定期実行化
[06章 §6.6](06_高可用性とシステム堅牢化.md) で扱う `test_restore.yml` は手動実行を前提としている。これを以下のいずれかで完全自動化する。

- GitHub Actions の `schedule:` で月初に自動 trigger
- ホストの systemd timer で月次起動

成功時のみ Discord に通知、失敗時は OnCall ロールにエスカレーション。

### 10.2.2 アプリリポジトリのテンプレート化
[08章 §8.2](08_CD（自動デプロイ）.md) のアプリ側 workflow（`peter-evans/repository-dispatch` 呼び出し）を Cookiecutter または GitHub Repository Template として用意し、新アプリの追加を 1 操作で可能にする。

### 10.2.3 cd_deploy.yml のアプリ追加自動化
現状、新しいアプリ（例: 新しいゲームサーバー）を CD パイプラインに組み込むには、cd_deploy.yml の `repository_dispatch.types` と `if:` 条件を手動で追記する必要がある。これを `matrix` 戦略または独立ファイル分割で簡素化する。

## 10.3 信頼性とセキュリティの強化

### 10.3.1 シークレットローテーション運用の自動化
現状、Cloudflare API トークン・Vault パスフレーズ・GitHub PAT は手動更新前提。期限切れ前に Discord へ通知し、ローテーション手順を Playbook 化する。

### 10.3.2 自宅 LAN セグメンテーション
緊急用 LAN（192.168.10.0/24）を運用 LAN と物理的に分離（VLAN 化）し、UFW での緊急アクセス経路がより堅牢になるよう改善する。

### 10.3.3 監視サーバー自体の冗長化検討
home_node2（tarakoserver）がダウンすると、Prometheus / Grafana / Alertmanager がまとめて停止し、フェイルオーバートリガーも失われる。クラウド側にメトリクスのミラーを持つ（Grafana Cloud 無料枠等）か、tarakoserver のメトリクス自体を別経路で監視する仕組みを検討する。

## 10.4 アプリケーション領域

### 10.4.1 Web アプリの段階デプロイ（カナリア）
現状、`--tags web` でのデプロイは「即時全リプレース」。トラフィックの一部を新バージョンに流すカナリアデプロイは未実装。Nginx の `split_clients` で比率指定が可能。

### 10.4.2 Minecraft ワールドの自動差分バックアップ
itzg/mc-backup は tar.gz でフル取得しているため、世代が増えるとサイズが膨らむ。差分（rsync / restic）への切替を検討。
