# 環境構築ロードマップ

## 本書の目的
第1〜8章は機能目的別に独立して記述されているため、「ゼロから shake 基盤を再現するには、どの章をどの順番で実施すればよいか」という通し手順としては読み取りにくい。本書は各章を構築の依存関係に沿って並べ直し、ナビゲーションとして機能させるためのガイドである。

各ステップでは「何をするか」「対応する章」「所要時間の目安」「完了判定」を示す。具体的なコマンドや設定内容は各章へのリンク先を参照する。

所要時間はあくまで目安であり、ハードウェアの状態やネットワーク環境により増減する。

## STEP 0: 事前準備

構築開始前に、作業端末・ハードウェア・アカウント・トークンを揃える。

- 実施内容: [事前準備マニュアル.md](事前準備マニュアル.md) のチェックリストをすべて埋める
- 完了判定: GitHub PAT、Cloudflare API トークン / R2 鍵 / Zone ID / Account ID、Tailscale Auth Key、Discord Webhook が手元に揃っている

---

## STEP 1: 作業端末と初期プロビジョニング

対応章: [01_IaC環境構築と初期プロビジョニング.md](01_IaC環境構築と初期プロビジョニング.md)

1. WSL2 / Ubuntu のインストール（§1.1）
2. Ansible / Terraform のインストール（§1.2〜1.3）
3. SSH 公開鍵の生成と各 Pi への配置（§1.4）
4. 作業用ディレクトリの作成（§1.5）
5. Terraform で Cloudflare の DNS レコードと R2 バケットを作成（§1.6）
6. ansible.cfg による性能最適化の確認（§1.7）
7. 自動操作用ユーザー ansible_admin の作成と inventory.ini の更新（§1.8）
8. common ロールの適用（自動アップデート、SWAP 無効化、ライフサイクル監視）（§1.8）
9. vars.yml / vault.yml のテンプレート配置（§1.9）

完了判定:
- `terraform apply` で R2 バケットと DNS レコードが作成された
- 全ノードに ansible_admin で SSH 接続でき、common ロールが正常適用された

---

## STEP 2: 機密情報の整備

対応章: [03_機密情報管理.md](03_機密情報管理.md)

1. vault.yml に実値を記入し `ansible-vault encrypt` で暗号化（§3.1）
2. key.pem など機密バイナリの暗号化（§3.1.4）
3. terraform.tfvars に各種トークンを記入（§3.2.6）
4. `terraform apply -var-file=...` で GitHub Actions Secrets を一括自動登録（§3.2.6）
5. pre-commit の導入と全ファイルチェック（§3.3.4）

完了判定:
- GitHub の Settings > Secrets and variables > Actions に必要な Secret が登録された
- `pre-commit run --all-files` がパスする

備考: STEP 2 は STEP 1 で R2 と inventory が整った後に行う。GitHub Secrets は STEP 8（CI/CD）で消費されるため、CD を使わない場合は §3.2 を後回しにしてもよい。

## STEP 3: ネットワーク基盤

対応章: [02_ネットワーク基盤.md](02_ネットワーク基盤.md)

1. Tailscale Auth Key を install_tailscale.yml に設定し、全ノードを Tailnet に参加させる（§2.2）
2. 実行後、inventory.ini が Tailscale IP（100.x.y.z）に自動置換されたことを確認（§2.2.5）
3. proxy ロールを適用し、大学固有の NTP/DNS、SSLH、Nginx Stream、UFW を構成（§2.3〜2.4）

完了判定:
- `tailscale status` で 3 ノードがすべてオンライン
- proxy ノードの 80/443 が ANY 開放、その他ポートが Tailscale + 緊急 LAN のみに制限されている

備考: 以降の監視・アプリ展開は Tailscale IP を前提とするため、ネットワーク基盤は監視より先に構築する。

## 補足: 一括反映と個別反映

通し構築の途中以降は、統合管理スクリプトで反映する。

```bash
cd iac-workspace
./run_playbook.sh
```

- 全体を一括適用するには「[2] デプロイ反映フェーズ → [1] 本番反映実行」を選ぶ
- 特定サービスのみ再反映するには同フェーズのサブメニュー、または `--tags <ロール名>` を使う

各ツールの日常運用・カスタマイズ手順は [ツール活用方法.md](ツール活用方法.md)、ディレクトリとファイルの役割は [ディレクトリ構造リファレンス.md](ディレクトリ構造リファレンス.md) を参照する。
