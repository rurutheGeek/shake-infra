PostgreSQLの環境構築（Dockerコンテナ化およびAnsibleによる自動展開）の手順をまとめる。

この構成では、セキュリティ確保のためにホストOS側にシステムユーザーを作成し、データの初期化およびユーザー作成をDockerの標準機能（`init-db.d`）に委ねることで、確実なリストアとロールバック防止を実現している。
また、パスワードなどの機密情報は、マニュアル4で導入した **Ansible Vault** の仕組みを利用し、安全にテンプレート（`env.j2`）へ流し込んでコンテナに渡す運用とする。

## 2. 設定ファイルの内容

### `files/postgres/init-db/01_setup_users.sh`

コンテナ初回起動時にのみ実行されるユーザー作成スクリプト。環境変数を展開して実行する。

```bash
#!/bin/bash
set -e

psql -v ON_ERROR_STOP=1 --username "$POSTGRES_USER" --dbname "$POSTGRES_DB" <<-EOSQL
    CREATE USER ${PKDB_READER_USER} WITH PASSWORD '${PKDB_READER_PASSWORD}';
    GRANT CONNECT ON DATABASE ${POSTGRES_DB} TO ${PKDB_READER_USER};
    GRANT USAGE ON SCHEMA public TO ${PKDB_READER_USER};
    GRANT SELECT ON ALL TABLES IN SCHEMA public TO ${PKDB_READER_USER};
    ALTER DEFAULT PRIVILEGES IN SCHEMA public GRANT SELECT ON TABLES TO ${PKDB_READER_USER};

    CREATE USER ${PKDB_EDITOR_USER} WITH PASSWORD '${PKDB_EDITOR_PASSWORD}';
    GRANT CONNECT ON DATABASE ${POSTGRES_DB} TO ${PKDB_EDITOR_USER};
    GRANT USAGE ON SCHEMA public TO ${PKDB_EDITOR_USER};
    GRANT SELECT, INSERT, UPDATE, DELETE ON ALL TABLES IN SCHEMA public TO ${PKDB_EDITOR_USER};
    ALTER DEFAULT PRIVILEGES IN SCHEMA public GRANT SELECT, INSERT, UPDATE, DELETE ON TABLES TO ${PKDB_EDITOR_USER};
EOSQL

```

※作成後、`chmod +x` で実行権限を付与しておく。

## 3. Ansible Role (`roles/postgres/tasks/main.yml`)

システムユーザーの作成、ディレクトリ権限の管理、テンプレートからの環境変数の生成を一括で行う。

**`roles/postgres/tasks/main.yml`**
```yaml
---
# postgres/tasks/main.yml
- name: データベース用システムユーザー(postgres)の作成
  ansible.builtin.user:
    name: postgres
    system: true
    shell: /usr/sbin/nologin
    create_home: false
    groups: docker
    append: true
  register: pg_user

- name: PostgreSQL用ディレクトリの作成
  ansible.builtin.file:
    path: "{{ item }}"
    state: directory
    owner: "{{ pg_user.uid }}"
    group: "{{ pg_user.group }}"
    mode: "0750"
  loop:
    - "{{ pg_dir }}"
    - "{{ pg_dir }}/data"
    - "{{ pg_dir }}/init-db"

- name: PostgreSQL構成ファイル群の配置
  ansible.builtin.copy:
    src: postgres/
    dest: "{{ pg_dir }}/"
    owner: "{{ pg_user.uid }}"
    group: "{{ pg_user.group }}"
    mode: "0640"

- name: PostgreSQL用 .env の生成 (Jinja2テンプレート)
  ansible.builtin.template:
    src: postgres/env.j2
    dest: "{{ pg_dir }}/.env"
    owner: "{{ pg_user.uid }}"
    group: "{{ pg_user.group }}"
    mode: "0600"

- name: PostgreSQL コンテナの起動
  community.docker.docker_compose_v2:
    project_src: "{{ pg_dir }}"
    state: present
    build: always
    remove_orphans: true
```

### 4. Ansible Roleファイル (`roles/minecraft/tasks/main.yml`)

ローカルコピーの処理を削除し、サーバーへ転送した`.env`ファイル内の環境変数を展開してR2からデータを取得する処理に変更した。

**`roles/minecraft/tasks/main.yml`**
```yaml
---
# minecraft/tasks/main.yml
- name: Minecraft用システムユーザー(minecraft)の作成
  ansible.builtin.user:
    name: minecraft
    system: true
    shell: /usr/sbin/nologin
    create_home: false
    groups: docker
    append: true
  register: mc_user

- name: Minecraft用ディレクトリの作成
  ansible.builtin.file:
    path: "{{ item }}"
    state: directory
    owner: "{{ mc_user.uid }}"
    group: "{{ mc_user.group }}"
    mode: "0750"
  loop:
    - "{{ mc_dir }}"
    - "{{ mc_dir }}/pixelmon"

- name: Minecraft構成ファイル群の同期
  ansible.builtin.copy:
    src: minecraft/
    dest: "{{ mc_dir }}/"
    owner: "{{ mc_user.uid }}"
    group: "{{ mc_user.group }}"
    mode: "0640"

- name: Minecraft用 .env の生成 (Jinja2テンプレート)
  ansible.builtin.template:
    src: minecraft/env.j2
    dest: "{{ mc_dir }}/.env"
    owner: "{{ mc_user.uid }}"
    group: "{{ mc_user.group }}"
    mode: "0600"

- name: 移行先Minecraftワールドデータの存在確認
  ansible.builtin.stat:
    path: "{{ mc_dir }}/pixelmon/world"
  register: mc_world_stat

- name: R2からのデータリストア処理
  when: not mc_world_stat.stat.exists
  block:
    - name: 一時ディレクトリの作成
      ansible.builtin.file:
        path: "{{ mc_dir }}/tmp"
        state: directory
        owner: "{{ mc_user.uid }}"
        group: "{{ mc_user.group }}"
        mode: "0750"

    - name: R2から最新のバックアップファイルを特定してダウンロード
      ansible.builtin.shell: |
        set -o pipefail
        set -a
        source {{ mc_dir }}/.env
        set +a

        LATEST=$(docker run --rm \
          -e RCLONE_CONFIG_R2_TYPE=s3 \
          -e RCLONE_CONFIG_R2_PROVIDER=Cloudflare \
          -e RCLONE_CONFIG_R2_ACCESS_KEY_ID=${CF_ACCESS_KEY} \
          -e RCLONE_CONFIG_R2_SECRET_ACCESS_KEY=${CF_SECRET_KEY} \
          -e RCLONE_CONFIG_R2_ENDPOINT=https://${CF_ACCOUNT_ID}.r2.cloudflarestorage.com \
          rclone/rclone lsf --sort time --reverse r2:${CF_BUCKET_NAME}/minecraft-backups/ | head -n 1)

        docker run --rm \
          -v {{ mc_dir }}/tmp:/data \
          -e RCLONE_CONFIG_R2_TYPE=s3 \
          -e RCLONE_CONFIG_R2_PROVIDER=Cloudflare \
          -e RCLONE_CONFIG_R2_ACCESS_KEY_ID=${CF_ACCESS_KEY} \
          -e RCLONE_CONFIG_R2_SECRET_ACCESS_KEY=${CF_SECRET_KEY} \
          -e RCLONE_CONFIG_R2_ENDPOINT=https://${CF_ACCOUNT_ID}.r2.cloudflarestorage.com \
          rclone/rclone copy r2:${CF_BUCKET_NAME}/minecraft-backups/$LATEST /data/
      args:
        executable: /bin/bash

    - name: ダウンロードしたアーカイブのファイル名を取得
      ansible.builtin.find:
        paths: "{{ mc_dir }}/tmp"
        patterns: "*.tar.gz"
      register: backup_archive

    - name: アーカイブをpixelmonディレクトリへ展開
      ansible.builtin.unarchive:
        src: "{{ backup_archive.files[0].path }}"
        dest: "{{ mc_dir }}/pixelmon/"
        remote_src: true
      when: backup_archive.matched > 0

    - name: 一時ファイルの削除
      ansible.builtin.file:
        path: "{{ mc_dir }}/tmp"
        state: absent

    - name: 展開したファイルの所有権を修正
      ansible.builtin.file:
        path: "{{ mc_dir }}/pixelmon"
        owner: "{{ mc_user.uid }}"
        group: "{{ mc_user.group }}"
        recurse: true

- name: Minecraft コンテナの起動
  community.docker.docker_compose_v2:
    project_src: "{{ mc_dir }}"
    state: present
    build: always
    remove_orphans: true
```

### rcon
RCON（Remote Console）は、Minecraftのゲームクライアントを立ち上げずに、外部からTCP通信経由でサーバーのコンソールコマンドを直接実行するためのプロトコルである。

1. RCONで実行可能な主な操作

ゲーム内でOP権限を持つプレイヤーが実行できるコマンドのほぼすべてを、インフラ側から実行可能である。

- **プレイヤー管理**: `kick`（キック）、`ban` / `pardon`（アクセス禁止/解除）、`op` / `deop`（権限付与/剥奪）、`whitelist`（ホワイトリストの操作）。
    
- **ワールド制御**: `save-all`（強制セーブ）、`save-off` / `save-on`（オートセーブの無効化/有効化）。
    
- **環境操作**: `time set`（時間の変更）、`weather`（天候の変更）、`gamerule`（ゲームルールの変更）。
    
- **コミュニケーション**: `say`（サーバー全体へのメッセージ送信）、`tellraw`（装飾されたシステムメッセージの送信）。
    

2. AnsibleとRCONの連携アーキテクチャ

現在運用中の `itzg/minecraft-server` のDockerイメージには、環境変数で指定したRCONパスワードを利用してローカルからコマンドを送れる `rcon-cli` というツールが標準で内包されている。

Ansibleの `command` または `shell` モジュールから `docker exec` を用いてこのツールを呼び出すことで、インフラ構成管理（IaC）とゲーム内操作をシームレスに統合できる。

3. 実践的な連携シナリオとPlaybook実装例

### シナリオA: メンテナンス時の安全な再起動と告知

インフラの更新や再デプロイを行う際、プレイ中のユーザーに警告を出し、データを安全にディスクへ書き出した上で処理を進行する。

```yaml
  tasks:
    - name: メンテナンス開始の全体告知
      command: docker exec shake_minecraft-pixelmon1.21.1 rcon-cli "say 1分後にインフラ更新のためのサーバー再起動を行います。"

    - name: 1分間待機
      pause:
        minutes: 1

    - name: ワールドデータの強制セーブ
      command: docker exec shake_minecraft-pixelmon1.21.1 rcon-cli "save-all"

    # この後にコンテナのダウンや設定ファイルの入れ替えタスクを記述する
```

シナリオB: プレイヤー権限・ホワイトリストのコード化 (IaC)

Ansibleの変数として許可するプレイヤーリストを定義し、Playbookを実行するだけで自動的にホワイトリストに反映させる。これにより、誰が許可されているかをGitなどのバージョン管理システムで追跡可能になる。

```yaml
  vars:
    allowed_players:
      - ruruthegeek
      - Baked_sususu

  tasks:
    - name: ホワイトリスト機能の有効化
      command: docker exec shake_minecraft-pixelmon1.21.1 rcon-cli "whitelist on"

    - name: 定義されたプレイヤーをホワイトリストに一括登録
      command: docker exec shake_minecraft-pixelmon1.21.1 rcon-cli "whitelist add {{ item }}"
      loop: "{{ allowed_players }}"
```

シナリオC: 定期的な状態の補正とイベントトリガー

CronとAnsibleを組み合わせることで、定期的にゲームルールを補正したり、インフラ側の状態（UPSのバッテリー低下など）をゲーム内に通知したりすることが可能である。


```yaml
  tasks:
    - name: インベントリ保持ルールの強制適用（OPの誤操作防止）
      command: docker exec shake_minecraft-pixelmon1.21.1 rcon-cli "gamerule keepInventory true"

    - name: デプロイ完了の演出（特定座標への雷の召喚など）
      command: docker exec shake_minecraft-pixelmon1.21.1 rcon-cli "summon lightning_bolt 0 70 0"
```

これらのタスクを既存の `roles/minecraft/tasks/main.yml` に組み込むことで、単なるコンテナの起動にとどまらない、サーバー運用に特化した自動化パイプラインが構築できる。

## Discord Bot アプリケーション（ubsleepy）の管理移行とバックアップ構成

Discord Botである ubsleepy のコンテナ化移行、および Cloudflare R2 を用いた毎日の自動バックアップ（ubsleepy_backup）の自動構成手順をまとめる。日常のAnsible再実行時に稼働中の最新セーブデータが古い初期データで上書き（ロールバック）されることを防ぐ冪等性設計、および実機デバッグによって発覚した重大な不具合の対策を組み込んでいる。

### 1. 作業用PCのディレクトリ構造

~/iac-workspace/ansible/ 配下を以下の構造で管理する。

```text

│   └── all/
│       ├── vars.yml
│       └── vault.yml
├── templates/

│       ├── ubsleepy-backup.service.j2
│       └── ubsleepy-backup.timer.j2
└── roles/
    ├── ubsleepy/
    │   └── tasks/

    └── ubsleepy_backup/
        ├── tasks/
        │   └── main.yml
        └── handlers/
            └── main.yml
```

### 2. 設定ファイルの内容と仕様

#### templates/ubsleepy/docker-compose.yml.j2
環境変数を参照し、データ永続化（save ディレクトリなど）をマウントして動かすコンテナ定義。

#### templates/ubsleepy_backup/backup_ubsleepy_r2.sh.j2
毎日自動的にセーブデータ、設定ファイル、および川柳データを圧縮し、R2 バケットへアップロードするシェルスクリプト。awscli が利用され、指定世代数（最大7世代）を超えた古いバックアップを削除する世代管理機能を備えている。

#### templates/ubsleepy_backup/ubsleepy-backup.timer.j2
毎日午前0時にバックアップスクリプトをキックする Systemd タイマー。実行予定時にシステムが停止していた場合でも、次回のシステム起動時に自動的に即時実行される（Persistent=true）堅牢な仕様。

### 3. ロール定義（roles/ubsleepy/tasks/main.yml）

初期化時にのみ初期データ（app_data.zip）のダウンロードと解凍を実行し、日常的なプレイブック再実行時には稼働中データの保護のために各処理をスキップする冪等性タスク。

```yaml
---
- name: UBSLEEPYのベースディレクトリを作成
  ansible.builtin.file:
    path: "{{ ubsleepy_dir }}"
    state: directory
    owner: "ansible_admin"
    group: "ansible_admin"
    mode: "0755"

- name: UBSLEEPYのリポジトリをクローンまたは更新
  ansible.builtin.git:
    repo: "{{ ubsleepy_repo_url }}"
    dest: "{{ ubsleepy_dir }}"
    version: main
    update: yes
    force: yes
  notify: Restart UBSLEEPY

- name: Awscliとunzipをインストール
  ansible.builtin.apt:
    name:
      - awscli
      - unzip
    state: present

- name: UBSLEEPYのsaveディレクトリの存在を確認
  ansible.builtin.stat:
    path: "{{ ubsleepy_dir }}/save"
  register: ubsleepy_save_dir

- name: R2からapp_data.zipをダウンロード
  ansible.builtin.shell: |
    aws s3 cp s3://{{ ubsleepy_r2_bucket }}/{{ ubsleepy_r2_object_key }} {{ ubsleepy_dir }}/{{ ubsleepy_r2_object_key }} \
      --endpoint-url https://{{ ubsleepy_r2_endpoint }} \
      --no-verify-ssl
  environment:
    AWS_ACCESS_KEY_ID: "{{ ubsleepy_r2_access_key_id }}"
    AWS_SECRET_ACCESS_KEY: "{{ ubsleepy_r2_secret_access_key }}"
  args:
    chdir: "{{ ubsleepy_dir }}"
  when: not ubsleepy_save_dir.stat.exists
  changed_when: false
  notify: Restart UBSLEEPY

- name: App_data.zipを展開
  ansible.builtin.unarchive:
    src: "{{ ubsleepy_dir }}/{{ ubsleepy_r2_object_key }}"
    dest: "{{ ubsleepy_dir }}"
    remote_src: yes
    extra_opts:
      - -O
      - cp932 # 日本語ファイル名対応のため
  when: not ubsleepy_save_dir.stat.exists
  notify: Restart UBSLEEPY

- name: App_data.zipをクリーンアップ
  ansible.builtin.file:
    path: "{{ ubsleepy_dir }}/{{ ubsleepy_r2_object_key }}"
    state: absent
  when: not ubsleepy_save_dir.stat.exists
  notify: Restart UBSLEEPY

- name: UBSLEEPYディレクトリ内の所有者をansible_adminに変更
  ansible.builtin.file:
    path: "{{ ubsleepy_dir }}"
    state: directory
    recurse: yes
    owner: "ansible_admin"
    group: "ansible_admin"
  when: not ubsleepy_save_dir.stat.exists
  notify: Restart UBSLEEPY

- name: Docker Composeファイルの配置
  ansible.builtin.template:
    src: ubsleepy/docker-compose.yml.j2
    dest: "{{ ubsleepy_dir }}/docker-compose.yml"
    owner: "ansible_admin"
    group: "ansible_admin"
    mode: "0644"
  notify: Restart UBSLEEPY

- name: .envファイルの生成
  ansible.builtin.template:
    src: ubsleepy/env.j2
    dest: "{{ ubsleepy_dir }}/.env"
    owner: "ansible_admin"
    group: "ansible_admin"
    mode: "0600"
  notify: Restart UBSLEEPY

- name: UBSLEEPYコンテナの起動
  community.docker.docker_compose_v2:
    project_src: "{{ ubsleepy_dir }}"
    state: present
    pull: always
```

### 5. 世代管理付き自動バックアップと死活監視（ubsleepy_backup）の構築手順

稼働するBotデータを毎日自動で Cloudflare R2 へバックアップし、コンテナの健康状態を監視して、異常や障害時には Discord に即時アラート通知するための仕組みを順番に構築します。

#### 手順 5.1: roles/ubsleepy_backup/templates/backup_ubsleepy_r2.sh.j2 の作成
作成意図と設計思想:
毎日定期実行され、セーブデータ（save/）、構成定義（config.json）、川柳データ（pokemon_senryu.csv）をまとめて tar.gz 圧縮し、Cloudflare R2 ストレージへ安全にアップロードします。また、ストレージの圧迫を防ぐために過去のバックアップ履歴を取得し、指定された世代数（最大7世代）を超えた古いバックアップを自動的に削除して世代管理を行う役割を持ちます。

デバッグおよび監視機能（Discord アラート）の最適化意図:
1. デバッグ検証時に発覚した aws s3 コマンドの R2_ENDPOINT におけるプロトコルスキーム欠落エラー（スキームがないためawscliがアップロードに失敗していた問題）を防ぐため、定義の先頭に明示的に https:// を付与するように最適化されています。
2. スクリプトの実行時に docker inspect コマンドを利用して対象 Bot コンテナの稼働ステータス（State.Running）を直接チェックする死活監視機能を統合しています。コンテナの停止や障害を検出した際、またはバックアップ処理そのものが圧縮失敗やネットワークエラーで失敗した際に、Discord Webhook URL を通じて、即座に赤や橙の色分けを伴うグラフィカルなアラート（Embed形式）を外部へ通知する仕組みを組み込み、インフラの可用性と監視性を高めています。

以下のコードを roles/ubsleepy_backup/templates/backup_ubsleepy_r2.sh.j2 に配置します。

```bash
#!/bin/bash
# R2にUBSLEEPYのセーブデータをバックアップするスクリプト

# --- 設定項目 ---
BACKUP_DIR="/opt/ubsleepy"
SAVE_DATA_PATH="${BACKUP_DIR}/save"
CONFIG_FILE_PATH="${BACKUP_DIR}/config.json"
POKEMON_SENRYU_PATH="${BACKUP_DIR}/resource/pokemon_senryu.csv"

R2_BUCKET="{{ ubsleepy_backup_r2_bucket }}"
R2_ENDPOINT="https://{{ ubsleepy_backup_r2_endpoint }}"
R2_ACCESS_KEY_ID="{{ ubsleepy_backup_r2_access_key_id }}"
R2_SECRET_ACCESS_KEY="{{ ubsleepy_backup_r2_secret_access_key }}"
R2_OBJECT_PREFIX="ubsleepy-save-backup/" # R2上のパス

MAX_BACKUPS="{{ ubsleepy_backup_max_backups }}" # 保持するバックアップの最大数
DISCORD_WEBHOOK_URL="{{ ubsleepy_backup_discord_webhook }}"

# --- Discord通知関数 ---
{% raw %}
send_discord_alert() {
    local status_type="$1" # "SUCCESS" or "ERROR" or "ALERT"
    local message="$2"
    local color=32768 # デフォルト緑 (Dec: 32768, Hex: 008000)
    
    if [ "${status_type}" = "ERROR" ]; then
        color=16711680 # 赤 (Dec: 16711680, Hex: FF0000)
    elif [ "${status_type}" = "ALERT" ]; then
        color=16753920 # 橙 (Dec: 16753920, Hex: FFA500)
    fi

    # Discord Webhookが未設定の場合はスキップ
    if [ -z "${DISCORD_WEBHOOK_URL}" ] || [ "${DISCORD_WEBHOOK_URL}" = "https://discordapp.com/api/webhooks/..." ]; then
        return 0
    fi

    local payload=$(cat <<EOF
{
  "embeds": [
    {
      "title": "UBSLEEPY バックアップ・監視アラート [\${status_type}]",
      "description": "\${message}",
      "color": \${color},
      "timestamp": "$(date -u +%Y-%m-%dT%H:%M:%SZ)"
    }
  ]
}
EOF
)

    curl -s -H "Content-Type: application/json" -X POST -d "${payload}" "${DISCORD_WEBHOOK_URL}" > /dev/null
}
{% endraw %}

# --- 環境変数設定 (awscli用) ---
export AWS_ACCESS_KEY_ID="${R2_ACCESS_KEY_ID}"
export AWS_SECRET_ACCESS_KEY="${R2_SECRET_ACCESS_KEY}"

# --- バックアップ処理 ---
TIMESTAMP=$(date +"%Y%m%d%H%M%S")
BACKUP_FILE="ubsleepy-save-${TIMESTAMP}.tar.gz"
BACKUP_PATH="/tmp/${BACKUP_FILE}"

echo "[$TIMESTAMP] ubsleepy_botコンテナの稼働状態を確認します..."
BOT_STATUS=$(docker inspect -f '{% raw %}{{.State.Running}}{% endraw %}' ubsleepy_bot 2>&1)

if [ $? -ne 0 ] || [ "${BOT_STATUS}" != "true" ]; then
    echo "[$TIMESTAMP] 警告: ubsleepy_botコンテナが停止しているか、存在しません。"
    send_discord_alert "ALERT" "Discord Botコンテナ (ubsleepy_bot) が正常に稼働していません。ステータス: ${BOT_STATUS}"
else
    echo "[$TIMESTAMP] ubsleepy_botコンテナは正常に稼働しています。"
fi

echo "[$TIMESTAMP] UBSLEEPYセーブデータのバックアップを開始します..."

# 必要なファイルとディレクトリを圧縮
tar -czvf "${BACKUP_PATH}" -C "${BACKUP_DIR}" save config.json resource/pokemon_senryu.csv

if [ $? -ne 0 ]; then
    echo "[$TIMESTAMP] エラー: バックアップファイルの作成に失敗しました。"
    send_discord_alert "ERROR" "バックアップ用アーカイブ of 圧縮作成に失敗しました。"
    exit 1
fi

echo "[$TIMESTAMP] バックアップファイル '${BACKUP_PATH}' を作成しました。"

# R2にアップロード
echo "[$TIMESTAMP] バックアップファイルをR2にアップロード中..."
aws s3 cp "${BACKUP_PATH}" "s3://${R2_BUCKET}/${R2_OBJECT_PREFIX}${BACKUP_FILE}" \
    --endpoint-url "${R2_ENDPOINT}"

if [ $? -ne 0 ]; then
    echo "[$TIMESTAMP] エラー: R2へのアップロードに失敗しました。"
    send_discord_alert "ERROR" "作成したバックアップアーカイブのCloudflare R2へのアップロードに失敗しました。"
    rm -f "${BACKUP_PATH}"
    exit 1
fi

echo "[$TIMESTAMP] バックアップファイルをR2にアップロードしました: s3://${R2_BUCKET}/${R2_OBJECT_PREFIX}${BACKUP_FILE}"

# ローカルの一時ファイルを削除
rm -f "${BACKUP_PATH}"
echo "[$TIMESTAMP] ローカルの一時ファイルを削除しました。"

# --- 世代管理 (古いバックアップの削除) ---
echo "[$TIMESTAMP] R2上の古いバックアップを削除します (最大${MAX_BACKUPS}世代保持)..."

# R2上のバックアップファイルをリストし、古いものを削除
BACKUPS_TO_DELETE=$(aws s3 ls "s3://${R2_BUCKET}/${R2_OBJECT_PREFIX}" --recursive --endpoint-url "${R2_ENDPOINT}" | awk '{print $NF}' | grep "${R2_OBJECT_PREFIX}.*\.tar\.gz" | sort | head -n -"${MAX_BACKUPS}")

if [ -n "${BACKUPS_TO_DELETE}" ]; then
    for old_backup in ${BACKUPS_TO_DELETE}; do
        echo "[$TIMESTAMP] 古いバックアップを削除中: s3://${R2_BUCKET}/${old_backup}"
        aws s3 rm "s3://${R2_BUCKET}/${old_backup}" --endpoint-url "${R2_ENDPOINT}"
    done
else
    echo "[$TIMESTAMP] 削除対象の古いバックアップはありませんでした。"
fi

echo "[$TIMESTAMP] バックアップ処理が完了しました。"
send_discord_alert "SUCCESS" "UBSLEEPYセーブデータのバックアップおよび世代管理（最大${MAX_BACKUPS}世代）が正常に完了しました。\\nファイル名: ${BACKUP_FILE}"
```

#### 手順 5.2: Systemd 自動起動ユニット（サービスとタイマー）の作成
作成意図:
バックアップスクリプトを毎日午前0時に自動的かつ確実にバックグラウンドで実行させるための Systemd 設定ファイルを構築します。

1. **roles/ubsleepy_backup/templates/ubsleepy-backup.service.j2**
   作成意図:
   バックアップバッチを root 特権ではなく、非特権の管理ユーザーである ansible_admin として安全に起動します。その際、docker inspect によるコンテナ監視コマンドを実行できるようにするため、Group=docker を付与して必要なソケットアクセス権限を最小特権の原則に基づいて安全に設定しています。

   以下のコードを roles/ubsleepy_backup/templates/ubsleepy-backup.service.j2 に配置します。

```ini
[Unit]
Description=Backup UBSLEEPY save data to Cloudflare R2
Requires=network-online.target
After=network-online.target

[Service]
Type=oneshot
ExecStart=/usr/local/bin/backup_ubsleepy_r2.sh
User=ansible_admin
Group=docker
WorkingDirectory=/opt/ubsleepy
Environment="PATH=/usr/sbin:/usr/bin:/usr/local/sbin:/usr/local/bin:/sbin:/bin"
```

2. **roles/ubsleepy_backup/templates/ubsleepy-backup.timer.j2**
   作成意図:
   毎日午前0時（daily）にサービスをキックします。特に重要な設計として、深夜の時間帯にサーバーの電源が落ちていた場合やメンテナンス中であった場合でも、次回のシステム起動時に自動的かつ即座に未実行分のバックアップ処理を補填・キックする Persistent=true 設定を有効化し、不測のデータ消失を防ぎます。

   以下のコードを roles/ubsleepy_backup/templates/ubsleepy-backup.timer.j2 に配置します。

```ini
[Unit]
Description=Run UBSLEEPY R2 backup daily
Requires=ubsleepy-backup.service

[Timer]
OnCalendar=daily
Persistent=true
RandomizedDelaySec=15m
WakeSystem=true

[Install]
WantedBy=timers.target
```

#### 手順 5.3: 自動バックアップ適用タスク（roles/ubsleepy_backup/tasks/main.yml）の作成
作成意図:
作成したスクリプトや Systemd ユニットファイルをサーバー上の適切なシステムパス（/usr/local/bin や /etc/systemd/system）に配布し、タイマーの自動有効化と起動を管理します。

設計の重要ポイント（Check Mode エラー対策）:
Ansibleの Check Mode（ドライラン）実行時には、前段のファイル配置タスクが実際には適用されない（ファイルが存在しない）ため、systemdモジュールがタイマーファイルの検出に失敗してエラーで停止します。このDry Run時の偽陽性の不具合を回避するため、タイマー起動タスクに when: not ansible_check_mode を付与し、安全なシミュレーションテストを実行可能にしています。

以下のコードを roles/ubsleepy_backup/tasks/main.yml に配置します。

```yaml
---
- name: バックアップスクリプトを/usr/local/binに配置
  ansible.builtin.template:
    src: backup_ubsleepy_r2.sh.j2
    dest: /usr/local/bin/backup_ubsleepy_r2.sh
    owner: root
    group: root
    mode: "0755"

- name: Systemdサービスファイルを配置
  ansible.builtin.template:
    src: ubsleepy-backup.service.j2
    dest: /etc/systemd/system/ubsleepy-backup.service
    owner: root
    group: root
    mode: "0644"
  notify: systemd daemon-reload

- name: Systemdタイマーファイルを配置
  ansible.builtin.template:
    src: ubsleepy-backup.timer.j2
    dest: /etc/systemd/system/ubsleepy-backup.timer
    owner: root
    group: root
    mode: "0644"
  notify: systemd daemon-reload

- name: Systemdタイマーを有効化して開始
  ansible.builtin.systemd:
    name: ubsleepy-backup.timer
    enabled: true
    state: started
  when: not ansible_check_mode
```

#### 手順 5.4: systemd ハンドラー（roles/ubsleepy_backup/handlers/main.yml）の作成
作成意図:
Systemd サービスファイルやタイマーファイルが新規に配置、または修正されたタイミングで、Ansible の notify 経由で systemd daemon-reload を自動的に発火させ、常に最新の Systemd 構成が Linux カーネル/デーモンに即座に反映されるように構成します。

以下のコードを roles/ubsleepy_backup/handlers/main.yml に配置します。

```yaml
---
- name: systemd daemon-reload
  ansible.builtin.systemd:
    daemon_reload: true
```
