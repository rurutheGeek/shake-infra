# 6. 高可用性とシステム堅牢化

## 高可用性とシステム堅牢化の目的
通常運用が安定した後、長期運用に向けて以下 4 軸の堅牢化を実施する。各軸は独立しているが、いずれも「単一障害点（SPOF）の排除」「データ喪失リスクの低減」「ハードウェア寿命の延長」を共通の狙いとする。

| 軸 | 内容 |
|---|---|
| 自動フェイルオーバー | プロキシダウン時に Cloudflare Workers で代理応答（メンテナンス画面） |
| OS の自動更新と SWAP 無効化 | unattended-upgrades による自動セキュリティパッチ、SD/SSD 摩耗防止 |
| 状態管理の冗長化 | Terraform Remote State（R2）と週次 Drift 検知 |
| 運用作業の自動化 | 安全シャットダウン Playbook と月次リストアテスト |

### 6.1 自動フェイルオーバー（Cloudflare Workers）

#### 6.1.1 採用目的とアーキテクチャ
自宅サーバー側がダウン（停電、ネットワーク断、メンテナンス中など）した場合、Cloudflare 側のエッジサーバー (Workers) で**任意のメンテナンス画面（HTTP 503）を即座に代理応答**させる。ゼロコストで実現する高可用性機構。

```text
通常時:  クライアント → Cloudflare → 大学プロキシ → 自宅メインサーバー
障害時:  クライアント → Cloudflare → Workers（メンテナンス画面）
            ▲
            │ 30秒間応答なしを Prometheus blackbox が検知
            │ → Alertmanager → failover_webhook → toggle_maintenance.sh on
            │ → Terraform で workers_route を有効化
```

#### 6.1.2 構成要素

| ファイル | 役割 |
|---|---|
| [terraform/worker_scripts/maintenance.html](../iac-workspace/terraform/worker_scripts/maintenance.html) | メンテナンス画面の HTML（ミニゲーム付き） |
| [terraform/worker_scripts/failover_worker.js](../iac-workspace/terraform/worker_scripts/failover_worker.js) | Workers ランタイムで HTML を返す JS |
| [terraform/main.tf](../iac-workspace/terraform/main.tf) | `cloudflare_workers_script` と `cloudflare_workers_route` のリソース定義 |
| [toggle_maintenance.sh](../iac-workspace/toggle_maintenance.sh) | 手動切替用。`terraform apply -var=maintenance_mode={on,off}` を実行する CLI ラッパー |
| [ansible/files/monitoring/maintenance_toggle.sh](../iac-workspace/ansible/files/monitoring/maintenance_toggle.sh) | 自動切替用（webhook が使用）。Cloudflare API で Workers Route を作成/削除。tarakoserver の /opt/monitoring に配置 |
| [ansible/roles/monitoring/templates/maintenance.env.j2](../iac-workspace/ansible/roles/monitoring/templates/maintenance.env.j2) | 上記スクリプトの CF API トークン/zone_id を Vault から注入する env テンプレート |

メンテナンス HTML の中身（ミニゲーム）は運用者の自由に編集できる領域。具体実装は IaC のスコープ外として扱う。

#### 6.1.3 Terraform リソース定義
[terraform/main.tf](../iac-workspace/terraform/main.tf)（抜粋）
```hcl
variable "maintenance_mode" {
  description = "Enable maintenance mode (routes traffic to Worker)"
  type        = bool
  default     = false
}

resource "cloudflare_workers_script" "failover_script" {
  account_id = var.cloudflare_account_id
  name       = "maintenance-failover"
  content    = replace(file("${path.module}/worker_scripts/failover_worker.js"),
                       "__MAINTENANCE_HTML_CONTENT__",
                       file("${path.module}/worker_scripts/maintenance.html"))
  module     = true
}

resource "cloudflare_workers_route" "failover_route" {
  count       = var.maintenance_mode ? 1 : 0
  zone_id     = var.cloudflare_zone_id
  pattern     = "ruruthegeek.dpdns.org/*"
  script_name = cloudflare_workers_script.failover_script.name
}
```

`count = var.maintenance_mode ? 1 : 0` がフェイルオーバーの本体。`maintenance_mode=true` の場合のみ Workers Route が作成され、トラフィックが Workers に流れる。

#### 6.1.4 手動切り替えスクリプト
[toggle_maintenance.sh](../iac-workspace/toggle_maintenance.sh)
```bash
# 使用方法
./iac-workspace/toggle_maintenance.sh on   # メンテ画面に切替
./iac-workspace/toggle_maintenance.sh off  # 通常運用に戻す
```

スクリプトは内部で `terraform apply -var="maintenance_mode=true|false" -auto-approve` を実行し、Discord にも切替状態を Webhook で通知する。

#### 6.1.5 Alertmanager + failover_webhook による自動切替
通常運用時はこれを自動化する。監視・通知側の詳細は [04_監視・ログ・通知基盤.md](04_監視・ログ・通知基盤.md) を参照。流れ:

1. Prometheus の `blackbox_exporter` がオリジンへの HTTP 疎通を 15 秒ごとに監視（`blackbox` ジョブ）。origin の :80 は HTTPS へ 301 するため、リダイレクトを追従せず 3xx を成功とみなす `http_proxy_reachable` モジュールを用いる（追従すると proxy_protocol 必須の :443 に当たり誤検知になるため。[04_監視・ログ・通知基盤.md](04_監視・ログ・通知基盤.md) §4.5 参照）。
2. 30 秒間ダウンを検知すると `ProxyDown` アラートが firing。
3. Alertmanager が `http://127.0.0.1:9000/webhook`（tarakoserver 常駐の failover_webhook.py）に POST。
4. Webhook が `/opt/monitoring/maintenance_toggle.sh on` を実行 → Cloudflare API で Workers Route を作成 → メンテ画面へ切替。
5. プロキシ復旧で `ProxyDown` が resolved になると `maintenance_toggle.sh off` が実行され、Route を削除して元に戻る。

切替スクリプトは 2 系統あり、用途で使い分ける。
- 自動（webhook が使用）: `/opt/monitoring/maintenance_toggle.sh`。Cloudflare API を直接呼んで Workers Route を作成/削除する。tarakoserver に terraform を置かずに高速切替でき、CF API トークンは Vault から `/opt/monitoring/maintenance.env`（0600・所有 ruru）へ注入する。monitoring ロールが配置する。
- 手動（任意）: `iac-workspace/toggle_maintenance.sh`。`terraform apply -var maintenance_mode=...` を実行する。terraform 環境のある作業端末から使う。

注意: かつて webhook が `/home/ruru/.../toggle_maintenance.sh`（terraform 方式）を呼ぶ実装だったが、tarakoserver には terraform もリポジトリも無く未動作だった。上記の Cloudflare API 方式へ置き換えて実動作を回復済み。

#### 6.1.6 動作テスト手順
1. 手動（terraform 方式）: `./iac-workspace/toggle_maintenance.sh on` → ドメインにアクセスしてメンテ画面表示を確認 → Discord 通知確認 → `off`。
2. 自動チェーンの E2E（推奨）: tarakoserver 上で webhook に擬似アラートを POST し、公開ドメインが 503 メンテ画面 → 200 復旧へ切り替わることを確認する。
```bash
# メンテ ON（firing）
curl -X POST -H 'Content-Type: application/json' \
  -d '{"alerts":[{"status":"firing","labels":{"alertname":"ProxyDown"}}]}' \
  http://127.0.0.1:9000/webhook
curl -s -o /dev/null -w '%{http_code}\n' https://ruruthegeek.dpdns.org/   # 503 を期待

# メンテ OFF（resolved）
curl -X POST -H 'Content-Type: application/json' \
  -d '{"alerts":[{"status":"resolved","labels":{"alertname":"ProxyDown"}}]}' \
  http://127.0.0.1:9000/webhook
curl -s -o /dev/null -w '%{http_code}\n' https://ruruthegeek.dpdns.org/   # 200 を期待
```
3. 実ダウン試験: プロキシの Nginx 停止や UFW で 80/443 を塞いで擬似ダウンを発生。約 30 秒後にメンテ画面へ切替、復旧で戻ることを確認。

### 6.2 OS の自動セキュリティアップデート

#### 6.2.1 採用目的
Raspberry Pi を VPN 経由で長期運用する場合、OS パッケージの脆弱性放置は致命的。`unattended-upgrades` パッケージで**セキュリティ系のパッチのみ自動適用**し、手動運用負荷を排除する。

#### 6.2.2 適用範囲と動作
全ノードに対して common ロールが以下を一括設定する。

| 設定ファイル | 内容 |
|---|---|
| `/etc/apt/apt.conf.d/20auto-upgrades` | アップデート確認と自動適用を有効化 |
| `/etc/apt/apt.conf.d/50unattended-upgrades` | 適用対象を `*-security` のみに限定。再起動が必要な場合は午前 2 時に自動再起動 |

[ansible/roles/common/tasks/main.yml](../iac-workspace/ansible/roles/common/tasks/main.yml)（抜粋）
```yaml
- name: セキュリティ自動アップデートツールのインストール
  ansible.builtin.apt:
    name: [unattended-upgrades, apt-listchanges]
    state: present

- name: 自動アップデートの詳細設定 (50unattended-upgrades)
  ansible.builtin.template:
    src: common/50unattended-upgrades.j2
    dest: /etc/apt/apt.conf.d/50unattended-upgrades
    mode: "0644"

- name: Unattended-upgrades サービスの有効化と起動
  ansible.builtin.systemd:
    name: unattended-upgrades
    enabled: true
    state: started
```

カーネル更新時の自動再起動は深夜 2 時に固定し、業務時間中のサービス断を回避している。

### 6.3 SWAP 完全無効化（SD/SSD 寿命延長）

#### 6.3.1 採用目的
Raspberry Pi 標準でオンになっている SWAP（メモリ不足時にストレージをメモリ代替に使う機能）は、microSD や SSD に対して**微細なランダム書き込みを頻発させ、物理セルの摩耗を加速**する。さらにディスク I/O ボトルネックは応答性能を低下させる。これらを避けるため、全ホストで SWAP を**恒久的に無効化**する。

#### 6.3.2 一括無効化処理
common ロールが以下を実施する（→[01_IaC環境構築と初期プロビジョニング.md](01_IaC環境構築と初期プロビジョニング.md) で初期から組み込み済）。

```yaml
- name: Raspberry Pi 標準の dphys-swapfile サービスの停止と無効化
  ansible.builtin.systemd:
    name: dphys-swapfile
    enabled: false
    state: stopped
  failed_when: false

- name: 現在有効なすべてのスワップの即時無効化 (swapoff)
  ansible.builtin.command: swapoff -a
  changed_when: false

- name: Fstab からのスワップ設定の完全排除
  ansible.builtin.replace:
    path: /etc/fstab
    regexp: '^([^#].*\sswap\s.*)$'
    replace: '# \1'

- name: 不要な dphys-swapfile パッケージの削除
  ansible.builtin.apt:
    name: dphys-swapfile
    state: absent
    purge: true
```

この一連のタスクにより、現在稼働中のスワップ領域が即座に解放されるだけでなく、`/etc/fstab` 内の定義が自動でコメントアウトされ、再起動後も一切スワップが生成・割り当てされない強固な IaC 制御が確立する。

### 6.4 安全なシャットダウン・再起動 Playbook

#### 6.4.1 採用目的
複数台の Raspberry Pi で稼働中のコンテナを止めずに OS を落とすと、DB の破損や Minecraft のセーブデータ消失が起きる可能性がある。これを防ぐため、**コンテナの安全停止 → ホスト OS シャットダウン** を一発で実行する Playbook を用意する。

#### 6.4.2 対象コンテナとシャットダウン順序
[ansible/playbooks/shutdown_servers.yml](../iac-workspace/ansible/playbooks/shutdown_servers.yml) は以下の順序で停止する。

**home_node1（メインサーバー）**:
1. Web 系（shake_web, bsquiz, alexa_skill）
2. Discord Bot（ubsleepy_bot）
3. Minecraft（shake_minecraft-pixelmon1.21.1 + バックアップサイドカー）
4. PostgreSQL 系（shake_postgres, shake_adminer, shake_postgres-backup）
5. cAdvisor

**home_node2（監視サーバー）**:
1. 監視スタック全体（prometheus, grafana, alertmanager, blackbox_exporter, loki, promtail）
2. GitHub Actions セルフホストランナー（github_runner）

すべてのコンテナ停止が完了した後、`shutdown -h now` を非同期実行する。

#### 6.4.3 統合管理スクリプトでの実行
```bash
./iac-workspace/run_playbook.sh
```
→「[3] サーバー電源・ライフサイクル操作」→「[1] 安全なシャットダウン実行」

対象は「全サーバー一括」または特定ホストを選択できる。シャットダウン実行時には最終確認プロンプトと Vault パスフレーズ入力が必須。

#### 6.4.4 再起動の非同期化（VPN 切断対策）
再起動操作は、Tailscale VPN の切断による Ansible の SSH 接続ハングアップを防ぐため、`reboot` モジュールに `-B 2 -P 0`（非同期 2 秒・ポーリングなし）を付与している。コマンド送信完了後に即座に制御が呼び出し側に返るため、Playbook 実行がフリーズすることはない。

### 6.5 Terraform Remote State と Drift 検知

#### 6.5.1 Remote State 採用目的
Terraform が管理するインフラの状態（Cloudflare の DNS 設定など）は、デフォルトでは作業用 PC のローカルファイル `terraform.tfstate` に保存される。**PC の故障や誤削除で状態ファイルを喪失すると、Terraform はクラウド側の現状を把握できなくなり、手動再構築が必要になる**。
このリスクを排除するため、状態ファイルをすでに運用中の Cloudflare R2（S3 互換オブジェクトストレージ）に保存（Remote State 化）する。

#### 6.5.2 backend "s3" の設定
[terraform/main.tf](../iac-workspace/terraform/main.tf) の `terraform { backend "s3" { ... } }` ブロック:
```hcl
terraform {
  backend "s3" {
    bucket                      = "shakeserver-backup"
    key                         = "terraform/state/terraform.tfstate"
    region                      = "auto"
    skip_credentials_validation = true
    skip_region_validation      = true
    skip_requesting_account_id  = true
    skip_metadata_api_check     = true
    skip_s3_checksum            = true
  }
  required_providers { ... }
}
```

R2 は S3 互換 API を提供するが、AWS S3 と異なるリージョン・認証フローを持つため、複数の `skip_*` フラグで AWS 固有のチェックをバイパスしている。

R2 の認証情報は機密のためコード内に書かず、別ファイル [local_config/terraform/backend.hcl](../iac-workspace/local_config/terraform/backend.hcl)（Git 管理外）に置く。

```hcl
access_key = "YOUR_R2_ACCESS_KEY"
secret_key = "YOUR_R2_SECRET_KEY"
endpoints  = { s3 = "https://YOUR_CLOUDFLARE_ACCOUNT_ID.r2.cloudflarestorage.com" }
```

#### 6.5.3 状態ファイルのマイグレーション（初回のみ）
```bash
cd iac-workspace/terraform
terraform init -backend-config=../local_config/terraform/backend.hcl -migrate-state
```
プロンプトに `yes` で応答すると、ローカルの `terraform.tfstate` が R2 にコピーされる。以降はすべて R2 上の状態ファイルが正となる。

#### 6.5.4 週次 Drift 検知の自動化
Terraform 管理外でクラウド側に手動変更が加わると、コードと実態が乖離（drift）する。これを早期発見するため、GitHub Actions で**毎週月曜 AM9:00 (JST) に `terraform plan -detailed-exitcode` を自動実行**する。

[.github/workflows/terraform_drift.yml](../.github/workflows/terraform_drift.yml) の主要動作:

| `terraform plan` の exit code | 解釈 | アクション |
|---|---|---|
| 0 | 差分なし | 正常終了（通知なし） |
| 2 | 差分あり | Discord に `[DRIFT DETECTED]` 通知 |
| 1 | エラー | Discord に `[DRIFT CHECK ERROR]` 通知 |

必要な GitHub Actions Secrets は [03_機密情報管理.md](03_機密情報管理.md) §3.2 で Terraform が自動登録した `CLOUDFLARE_API_TOKEN`, `CF_R2_ACCESS_KEY_ID`, `CF_R2_SECRET_ACCESS_KEY`, `TF_BACKEND_CONFIG`, `DISCORD_WEBHOOK_URL`。

#### 6.5.5 手動実行
緊急時は GitHub の Actions タブから「Terraform Drift Detection」workflow を選択し「Run workflow」で即時実行できる。

### 6.6 バックアップリストアテスト（月次）

#### 6.6.1 採用目的
バックアップは「取れている」だけでは意味がない。「確実に復元できる」ことを定期的に確認して初めて、災害復旧（DR）能力として機能する。本リポジトリでは月次で PostgreSQL のバックアップを実機外のテストコンテナにリストアし、整合性を確認するワンショット Playbook を用意する。

#### 6.6.2 ワークフロー
[ansible/playbooks/test_restore.yml](../iac-workspace/ansible/playbooks/test_restore.yml) は以下の流れで実行する。

1. `/tmp/restore_test/` 作業ディレクトリを作成
2. R2 バケット `shakeserver-backup` の `db-backups/` プレフィックスから最新の `.dump.gz` を特定（aws-cli コンテナ）
3. ダウンロードして gunzip で展開
4. `postgres:15-alpine` でテスト用一時コンテナを起動（`restore_test_postgres`）
5. `pg_isready` で待機後、`pg_restore --no-owner --no-privileges` でテストコンテナにリストア
6. `psql -c "\dt"` でテーブル一覧を取得し、出力を debug 表示
7. Discord に成功通知（`ubsleepy_backup_discord_webhook`）
8. **always ブロック**: テストコンテナと作業ディレクトリを必ず削除（成否によらず後片付け）

#### 6.6.3 バックアップ形式の前提
postgres ロールの db-backup コンテナは `eeshugerman/postgres-backup-s3:15` を使用しており、デフォルトで以下を行う。

- `pg_dump --format=custom`（カスタム形式、バイナリ）
- `gzip` で圧縮
- R2 へ `s3://shakeserver-backup/db-backups/<db>_<timestamp>.dump.gz` として保存

したがってリストア側も `pg_restore` を使う必要がある（`psql < file` では動かない）。これは以前のドキュメントとコードの不整合だったが修正済。

#### 6.6.4 必要な Vault 変数
| 変数 | 用途 |
|---|---|
| `r2_access_key_id` | R2 へのアクセス鍵 |
| `r2_secret_access_key` | R2 シークレットキー |
| `r2_endpoint` | `https://<account_id>.r2.cloudflarestorage.com` |
| `ubsleepy_backup_discord_webhook` | テスト結果の通知先 |

#### 6.6.5 実行方法
```bash
cd iac-workspace
ansible-playbook -i local_config/ansible/inventory.ini \
  -e @local_config/ansible/vars.yml \
  -e @local_config/ansible/credentials/vault.yml \
  ansible/playbooks/test_restore.yml
```

月初に手動実行することを推奨。テストの自動化（cron 化や CD ワークフロー化）は今後の展望に記載（[10_今後の展望.md](10_今後の展望.md)）。
