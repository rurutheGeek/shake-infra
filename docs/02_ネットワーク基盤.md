# 2. ネットワーク基盤（VPN・リバースプロキシ・UFW戦略）

## ネットワーク基盤の目的
3拠点に分散配置された Raspberry Pi 群を、追加コストとポート開放なしに相互接続し、外部公開トラフィックを単一ドメインに集約しつつ、最小権限のファイアウォール（UFW）でゾーニングする。
本章では、ノード配置、Tailscale メッシュ VPN、リバースプロキシ、UFW の IP 戦略の4要素を一体として扱う。各要素の採用根拠の詳細は [09_技術スタックと選定根拠.md](09_技術スタックと選定根拠.md) を参照。

### 2.1 ノード構成と物理配置

| インベントリ名 | グループ | ハードウェア | 拠点 | 役割 |
|---|---|---|---|---|
| `negitoroserver` | `proxy_node` | Raspberry Pi Zero 2 W | 大学内ネットワーク | 外部 443/80/25565 を受け、Tailscale 経由でメインサーバーへ転送するゲートウェイ |
| `shakeserver` | `home_node1` | Raspberry Pi 5 | 自宅 | アプリケーション本体（Web/PostgreSQL/Minecraft/Discord Bot）と UPS 監視 |
| `tarakoserver` | `home_node2` | Raspberry Pi 4 | 自宅 | 監視スタック（Prometheus/Grafana/Loki/Alertmanager）と GitHub Actions セルフホストランナー |

Zero 2 W を大学拠点に配置する目的は、自宅にグローバル IP を露出させずに外部公開を実現するためである。大学のグローバル IP に紐づく A レコードで受信したトラフィックを、Tailscale でメインサーバーへ引き渡す構造により、自宅ルーターのポート開放を完全に避けられる。

### 2.2 Tailscale（メッシュ VPN）の構築

#### 2.2.1 採用目的
- 物理的に離れた3ノード間を、ポート開放なしで相互接続したい
- WireGuard ベースの低レイテンシ通信
- CGNAT 帯（100.64.0.0/10）から各ノードに一意のプライベート IP を自動付与 → UFW で帯域指定がしやすい
- Auth Key により Ansible での一括登録が可能

#### 2.2.2 Tailscale Auth Key の取得
Ansible から自動でログイン処理を行うための認証キーを発行する。
1. ブラウザで Tailscale Admin Console (`https://login.tailscale.com/admin/settings/keys`) にアクセスし、自身のアカウントでログインする。
2. 左側メニューの Settings > Keys を開く。
3. Auth keys のセクションにある Generate auth key をクリックする。
4. 設定モーダルで以下の通り選択する。
    - Reusable: オン（3台のマシンで同じキーを使うため必須）
    - Ephemeral: オフ
    - Pre-approved: オン（承認待ちを省くため推奨）
5. Generate key をクリックし、表示された `tskey-auth-` から始まる文字列をコピーして控える。

#### 2.2.3 Tailscale導入Playbookの作成
作業用PCの [iac-workspace/ansible/playbooks](../iac-workspace/ansible/playbooks) ディレクトリに、Tailscaleの導入と設定を行うPlaybookを作成する。
以下のコードを [install_tailscale.yml](../iac-workspace/ansible/playbooks/install_tailscale.yml) として保存する。`tailscale_authkey` の値は、手順 2.2.2 で取得した実際のキーに置き換える。

```yaml
---
- name: Tailscaleのインストールとネットワーク参加
  hosts: all
  become: true
  vars:
    tailscale_authkey: "tskey-auth-XXXXXXXXXXXXXXXXXXXXX"
  tasks:
    - name: Tailscaleインストールスクリプトの実行
      ansible.builtin.shell: curl -fsSL https://tailscale.com/install.sh | sh
      args:
        creates: /usr/bin/tailscale

    - name: Tailnetへの参加 (Auth Keyを使用)
      ansible.builtin.command: tailscale up --authkey={{ tailscale_authkey }} --accept-routes
      register: tailscale_result
      changed_when: "'Success' in tailscale_result.stdout"
      failed_when: tailscale_result.rc != 0

    - name: Tailscale IP (IPv4) の取得
      ansible.builtin.command: tailscale ip -4
      register: ts_ip
      changed_when: false

    - name: 各ノードのTailscale IPを表示
      ansible.builtin.debug:
        msg: "{{ inventory_hostname }} の Tailscale IP は {{ ts_ip.stdout }} です"

    - name: inventory.iniのIPアドレスをTailscale IPに自動更新
      delegate_to: localhost
      become: false
      ansible.builtin.replace:
        path: "{{ inventory_file }}"
        regexp: '^({{ inventory_hostname }}\s+ansible_host=)[0-9\.]+(.*)$'
        replace: '\g<1>{{ ts_ip.stdout }}\g<2>'
```

#### 2.2.4 Playbookの実行
作業用 PC の [iac-workspace/ansible](../iac-workspace/ansible) ディレクトリで以下のコマンドを実行する。

```bash
ansible-playbook -i local_config/ansible/inventory.ini ansible/playbooks/install_tailscale.yml
```

#### 2.2.5 IPアドレスの記録
Playbookの実行が完了すると、最後に出力される `TASK [各ノードのTailscale IPを表示]` のセクションに、各 Raspberry Pi に割り当てられた `100.x.y.z` 形式の IP アドレスが表示される。inventory.ini への自動置換も同タスク内で行われる。

```text
TASK [各ノードのTailscale IPを表示]
ok: [negitoroserver] => { "msg": "negitoroserver の Tailscale IP は 100.X.X.1 です" }
ok: [shakeserver]    => { "msg": "shakeserver の Tailscale IP は 100.X.X.2 です" }
ok: [tarakoserver]   => { "msg": "tarakoserver の Tailscale IP は 100.X.X.3 です" }
```

### 2.3 リバースプロキシ（proxy ノード）の構築

#### 2.3.1 採用目的（SSLH + Nginx Stream の2段構成）
1つのグローバル IP・1つの 443 ポートに対して、HTTPS と Minecraft（25565）の両方を多重化したい。さらに、Pi 5 側で本来のクライアント IP を取得できるようにしたい。これを実現するために以下の2段プロキシを採用する。

- SSLH: プロトコル判別器。443 番に届いた通信が TLS なら Nginx へ、Minecraft プロトコルなら 25565 へ振り分ける
- Nginx stream: 443 を受けて、ホームの Pi 5（Tailscale IP）の 443 へ Proxy Protocol 付きで TCP 転送する

クライアントから見ると、`https://example.com/` でも `mc.example.com:25565` でも同一の Zero 2 W がエンドポイントとなる。Pi 5 はその裏側に隠れる。

```text
外部クライアント (203.0.113.1)
       │
       ▼
[ negitoroserver / Zero 2 W ]
  ├─ 80   (sslh)         → http   → home_node1:80    (Tailscale)
  ├─ 443  (sslh)         → tls    → 自ノード:443     (Nginx Stream)
  │                       → mc    → home_node1:25565 (Tailscale)
  └─ 443  (Nginx stream) → proxy_protocol on → home_node1:443
                                                ※クライアント実IPはProxy Protocolヘッダで通知
       │
       ▼
[ shakeserver / Pi 5 ]
  ├─ 80   (Nginx, リダイレクト用)
  ├─ 443  (Nginx, proxy_protocol を解釈し real_ip 復元)
  ├─ 8443 (Nginx, proxy_protocol なし: LAN/Tailscale から直アクセス用、メンテナンス時の代替経路)
  └─ 25565 (Minecraft 待受)
```

8443 ポートは Zero 2 W がダウンした際の緊急アクセス用として、LAN や Tailscale から直接アクセスできる経路として残している。

#### 2.3.2 大学ネットワーク固有の設定（NTP/DNS の強制上書き）
proxy_node は大学キャンパス内ネットワークに設置されるため、以下3点を Ansible が自動構成する。これらは大学側の運用ポリシー（学内 DNS/NTP の利用必須）に対応するための設定であり、家庭用ネットワークでは不要。

1. Tailscale MagicDNS の無効化  
   Tailscale が `/etc/resolv.conf` を自動で書き換えてしまうのを止め、大学指定 DNS を優先させる。
2. 時刻同期 (NTP) を大学指定サーバーに固定  
   `/etc/systemd/timesyncd.conf` の `NTP=` を `133.80.128.5 133.80.153.2` に上書き。apt パッケージ取得前に `meta: flush_handlers` で即時反映させ、時刻ずれによる証明書検証エラーを防ぐ。
3. `/etc/resolv.conf` を大学指定 DNS で固定  
   既存のシンボリックリンクを削除し、平文の resolv.conf を配置する。

[ansible/roles/proxy/tasks/main.yml](../iac-workspace/ansible/roles/proxy/tasks/main.yml)（抜粋）
```yaml
- name: TailscaleのDNS強制上書き（MagicDNS）を無効化
  ansible.builtin.command: tailscale set --accept-dns=false
  failed_when: false
  changed_when: false

- name: 学内NTPサーバーの設定
  ansible.builtin.lineinfile:
    path: /etc/systemd/timesyncd.conf
    regexp: "^#?NTP="
    line: "NTP=133.80.128.5 133.80.153.2"
  notify: Restart timesyncd

- name: 既存の /etc/resolv.conf を削除（シンボリックリンク解除）
  ansible.builtin.file:
    path: /etc/resolv.conf
    state: absent

- name: 大学指定のDNSサーバーを固定で設定
  ansible.builtin.copy:
    dest: /etc/resolv.conf
    content: |
      nameserver 133.80.128.5
      nameserver 133.80.153.2
    mode: "0644"

- name: 時刻同期（NTP再起動）をapt実行前に即時反映させる
  ansible.builtin.meta: flush_handlers
```

#### 2.3.3 SSLH と Nginx Stream の構成ファイル

[files/proxy/stream_proxy.conf](../iac-workspace/ansible/files/proxy/stream_proxy.conf)
```nginx
server {
    listen 443;
    proxy_pass 100.X.X.2:443;
    proxy_protocol on;
}
```

[files/proxy/sslh.default](../iac-workspace/ansible/files/proxy/sslh.default)
```text
RUN=yes
DAEMON=/usr/sbin/sslh
DAEMON_OPTS="--user sslh --listen 0.0.0.0:80 --http 100.X.X.2:80 --anyprot 100.X.X.2:25565 --pidfile /var/run/sslh/sslh.pid"
```

#### 2.3.4 proxy ロールのタスク全体像
ロールが行うのは大別すると4工程: ①大学固有 NTP/DNS 設定、②パッケージ導入（nginx, libnginx-mod-stream, sslh, ufw）、③UFW のルール適用（→2.4）、④nginx.conf に stream ブロック追記と設定ファイル配置。

[ansible/roles/proxy/tasks/main.yml](../iac-workspace/ansible/roles/proxy/tasks/main.yml)（パッケージ導入以降の主要部）
```yaml
- name: sslhのdebconf設定（スタンドアロンモードの事前選択）
  ansible.builtin.debconf:
    name: sslh
    question: sslh/inetd_or_standalone
    value: standalone
    vtype: select

- name: パッケージのインストール (Nginx, sslh, ufw)
  ansible.builtin.apt:
    name:
      - nginx
      - libnginx-mod-stream
      - sslh
      - ufw
    state: present
    update_cache: true
    cache_valid_time: 3600
  environment:
    DEBIAN_FRONTEND: noninteractive

- name: Nginx.confにstreamブロックを追記
  ansible.builtin.blockinfile:
    path: /etc/nginx/nginx.conf
    marker: "# {mark} ANSIBLE MANAGED BLOCK: stream configuration"
    insertafter: EOF
    content: |
      stream {
          include /etc/nginx/stream.conf.d/*.conf;
      }

- name: Nginxプロキシ設定ファイルの配置
  ansible.builtin.copy:
    src: files/proxy/stream_proxy.conf
    dest: /etc/nginx/stream.conf.d/stream_proxy.conf
    mode: "0644"
  notify: Restart Nginx
```

`sslh` パッケージはインストール時に inetd/standalone の選択を対話で求めるため、`debconf` モジュールで事前に standalone を選択させ、自動化を阻害しないようにしている。

### 2.4 UFW（ファイアウォール）の IP 戦略

ネットワーク基盤の通信を制御する UFW ルールは、本リポジトリ全体で一貫した**3階層の IP 区分**を採用している。すべてのロールはこの区分にしたがってポート開放範囲を決めている。

| 区分 | CIDR | 用途 |
|---|---|---|
| Tailscale メッシュ VPN | `100.64.0.0/10` | ノード間相互通信。Tailscale が CGNAT 帯から各ノードに割り当てる Tailscale IP の包括レンジ |
| 緊急用ローカル LAN | `192.168.10.0/24` | Tailscale ダウン時の最終ライフライン。自宅 LAN 内から直接 SSH/HTTPS にアクセスできるよう常時開放 |
| 外部インターネット | `0.0.0.0/0` | proxy_node の 80/443 のみ。それ以外のノードはすべて Tailscale 経由でしか到達不可 |

このゾーニングにより、すべての内部サービス（PostgreSQL の 5432、Grafana の 3000、cAdvisor の 8090 等）は Tailscale および緊急 LAN からのみアクセス可能となり、運用上の安全性と障害時のリーチアビリティを両立する。

#### 2.4.1 各ノードの主要 UFW ルール

##### proxy_node (negitoroserver)
| ポート | 用途 | 開放範囲 |
|---|---|---|
| 80, 443 | HTTP/HTTPS パブリック | ANY |
| 22 | SSH | Tailscale + LAN のみ |

##### home_node1 (shakeserver)
| ポート | 用途 | 開放範囲 |
|---|---|---|
| 9101 | UPS Exporter | Tailscale |
| 8090 | cAdvisor | 監視ノードの Tailscale IP（`100.123.163.119`）+ LAN |
| ※コンテナ側 5432/80/443 等は Docker のホストネットワーク経由で公開 |

##### home_node2 (tarakoserver)
| ポート | 用途 | 開放範囲 |
|---|---|---|
| 9090 | Prometheus | Tailscale + LAN |
| 3000 | Grafana | Tailscale + LAN |
| 3100 | Loki | Tailscale + LAN |

#### 2.4.2 proxy ノードでの UFW 適用例
[ansible/roles/proxy/tasks/main.yml](../iac-workspace/ansible/roles/proxy/tasks/main.yml)（抜粋）
```yaml
- name: "古いSSHポートの開放ルール (全許可) を削除"
  community.general.ufw:
    rule: allow
    port: "22"
    proto: tcp
    delete: true

- name: "SSHポートの開放 (UFW: Tailscale経由を許可)"
  community.general.ufw:
    rule: allow
    src: 100.64.0.0/10
    port: "22"
    proto: tcp

- name: "SSHポートの開放 (UFW: 緊急用ローカルLANからのアクセスを許可)"
  community.general.ufw:
    rule: allow
    src: 192.168.10.0/24
    port: "22"
    proto: tcp

- name: "HTTP/HTTPSポートの開放 (UFW)"
  community.general.ufw:
    rule: allow
    port: "{{ item }}"
    proto: tcp
  loop:
    - "80"
    - "443"

- name: "UFWの有効化"
  community.general.ufw:
    state: enabled
```

旧来の `allow 22/tcp` を削除してから新ルールを追加することで、Ansible 再実行時にも安全に再構成できる冪等性を確保している。
