# 4. 監視・ログ・通知基盤

## 監視・ログ・通知基盤の目的
インフラの異常（CPU 高騰、メモリ・ディスク枯渇、コンテナ停止、ノード喪失、UPS バッテリー低下、サービスダウンなど）を早期検知し、復旧時点まで含めて運用者に伝達するための一体の基盤を構築する。
3 軸（メトリクス・ログ・能動通知）を統合する Prometheus + Grafana + Loki + Alertmanager + Discord Webhook のスタックを採用する。各ツールの選定根拠は [09_技術スタックと選定根拠.md](09_技術スタックと選定根拠.md) を参照。

### 監視基盤の3層構造

| 層 | コンポーネント | 配置ノード | 役割 |
|---|---|---|---|
| 計測 | node_exporter / cAdvisor / ups_exporter / blackbox_exporter | 各ノード（home_node1 中心） | OS・コンテナ・UPS・HTTP 疎通のメトリクスを公開 |
| 集約・保管 | Prometheus / Loki | home_node2 (tarakoserver) | メトリクスとログを 1 か所に集約 |
| 可視化・通知 | Grafana / Alertmanager / failover_webhook / Discord Webhook | home_node2 | ダッシュボード可視化と自動アラート発火 |

### 4.1 計測層: 各ノードでメトリクスを公開する

#### 4.1.1 exporters ロール（全ノード共通）
すべてのノードに以下のエクスポーターを導入する。

| エクスポーター | ポート | 公開する情報 | 配置 |
|---|---|---|---|
| node_exporter | 9100 | CPU・メモリ・ディスク・ネットワーク・LA など OS リソース | 全 3 ノード |
| cAdvisor | 8090 (ホスト) | Docker コンテナの CPU/メモリ/IO/ネット使用率 | Docker が稼働するノード（home_node1 / home_node2） |

cAdvisor のデフォルトポート 8080 は Web アプリで競合する可能性があるため、本リポジトリでは **8090 → 8080 のポートマッピング** で公開し競合を回避する。また、UFW のソース IP 制限は監視サーバー（tarakoserver の Tailscale IP `100.123.163.119`）と緊急用 LAN `192.168.10.0/24` のみに絞る（→2.4 UFW 戦略）。

[ansible/roles/exporters/tasks/main.yml](../iac-workspace/ansible/roles/exporters/tasks/main.yml)
```yaml
---
- name: Node Exporterのインストール
  ansible.builtin.apt:
    name: prometheus-node-exporter
    state: present
    update_cache: true
    cache_valid_time: 3600

- name: サービスの有効化と起動
  ansible.builtin.systemd:
    name: prometheus-node-exporter
    enabled: true
    state: started

- name: Dockerコマンドの存在確認
  ansible.builtin.command: which docker
  register: exporters_docker_check
  failed_when: false
  changed_when: false

- name: CAdvisorコンテナの起動
  community.docker.docker_container:
    name: cadvisor
    image: gcr.io/cadvisor/cadvisor:v0.49.1
    state: started
    restart_policy: unless-stopped
    privileged: true
    devices:
      - "/dev/kmsg:/dev/kmsg"
    ports:
      - "8090:8080"
    volumes:
      - "/:/rootfs:ro"
      - "/var/run:/var/run:ro"
      - "/sys:/sys:ro"
      - "/var/lib/docker/:/var/lib/docker:ro"
      - "/dev/disk/:/dev/disk:ro"
  when: exporters_docker_check.rc == 0

- name: "UFWでポート8090の開放（監視サーバーのみ）"
  community.general.ufw:
    rule: allow
    src: 100.123.163.119
    port: "8090"
    proto: tcp
  when: exporters_docker_check.rc == 0

- name: "UFWでポート8090の開放（緊急用ローカルLAN）"
  community.general.ufw:
    rule: allow
    src: 192.168.10.0/24
    port: "8090"
    proto: tcp
  when: exporters_docker_check.rc == 0
```

`when: exporters_docker_check.rc == 0` により、Docker が無いノード（proxy_node 等）では cAdvisor のデプロイを自動スキップする。これにより exporters ロールを全ノードに一律適用できる。

#### 4.1.2 ups_exporter ロール（home_node1 専用）
Geekworm X120X UPS ボードのバッテリー情報を Prometheus メトリクスとして公開するカスタムエクスポーター。

| 構成要素 | 内容 |
|---|---|
| ハードウェア | Raspberry Pi 5 + Geekworm X120X UPS ボード |
| バッテリー測定 | I2C バス（アドレス 0x36）のレジスタ読み取り |
| AC ステータス測定 | GPIO ピン 6 の押下状態（PLD）から電源喪失を検知 |
| 実行環境 | Python 3 + uv（高速パッケージマネージャ）で venv 構築 |
| 公開ポート | 9101（Tailscale のみ UFW 開放） |
| 常駐方式 | systemd サービス（ups-exporter.service） |

権限とエラー対策:
- スクリプトは非 root ユーザー（ansible_admin）で実行するため、`i2c` と `gpio` グループへ追加する
- lgpio ライブラリが作成する通知用パイプ（`.lgd-nfy`）の書き込み権限不足を防ぐため、インストールディレクトリの所有権を実行ユーザーに再帰設定する

[ansible/roles/ups_exporter/tasks/main.yml](../iac-workspace/ansible/roles/ups_exporter/tasks/main.yml)（抜粋）
```yaml
- name: 必要なパッケージのインストール
  ansible.builtin.apt:
    name: [git, i2c-tools, ufw, curl]
    state: present
    update_cache: true

- name: Uvのインストール
  ansible.builtin.shell: |
    set -o pipefail
    curl -LsSf https://astral.sh/uv/install.sh | sh
  args:
    creates: /root/.local/bin/uv
    executable: /bin/bash

- name: 実行ユーザーを i2c と gpio グループに追加
  ansible.builtin.user:
    name: "{{ ups_exporter_run_user }}"
    groups: i2c,gpio
    append: true

- name: リポジトリのクローン
  ansible.builtin.git:
    repo: "{{ ups_exporter_repo_url }}"
    dest: "{{ ups_exporter_install_dir }}"
    force: true
    version: HEAD

- name: 監視用ディレクトリの所有権をユーザーに変更
  ansible.builtin.file:
    path: "{{ ups_exporter_install_dir }}"
    state: directory
    owner: "{{ ups_exporter_run_user }}"
    group: "{{ ups_exporter_run_user }}"
    recurse: true

- name: Uv venvによるPython仮想環境の作成
  ansible.builtin.command: uv venv {{ ups_exporter_install_dir }}/venv
  args:
    creates: "{{ ups_exporter_install_dir }}/venv"

- name: Systemdサービスファイルの配置
  ansible.builtin.copy:
    src: "{{ ups_exporter_install_dir }}/ups-exporter.service"
    dest: /etc/systemd/system/ups-exporter.service
    remote_src: true
    mode: "0644"
  notify: Restart ups-exporter
```

公開する主なメトリクス:
- `ups_battery_capacity_percent` - バッテリー残量
- `ups_external_power_status` - 外部電源（AC）の有無
- `ups_voltage_volts` - 電圧

### 4.2 集約・保管層: 監視スタックを home_node2 に展開

#### 4.2.1 monitoring ロールが起動する6コンテナ
[ansible/files/monitoring/monitoring-compose.yml](../iac-workspace/ansible/files/monitoring/monitoring-compose.yml) で定義する6コンテナをすべて `network_mode: host` で起動する。

| コンテナ | イメージ | ポート | 役割 |
|---|---|---|---|
| prometheus | prom/prometheus:latest | 9090 | メトリクス収集 |
| grafana | grafana/grafana:latest | 3000 | ダッシュボード可視化 |
| alertmanager | prom/alertmanager:latest | 9093 | アラートのルーティング・通知 |
| blackbox_exporter | prom/blackbox-exporter:latest | 9115 | HTTP 疎通監視（プロキシ生存確認） |
| loki | grafana/loki:3.0.0 | 3100 | ログ集約・保管 |
| promtail | grafana/promtail:3.0.0 | 9080（内部） | 各ノードのログ転送エージェント |

##### ホストネットワークモード採用の根拠
Docker のデフォルト（ブリッジ）モードでは、コンテナとホストの間に NAT が挟まり、特に Tailscale 経由のトラフィックがコンテナ内に到達しない（ルーティング先がコンテナ内 IP に解決されないため）。`network_mode: host` で**ホストの NIC を直接共有**することで、Tailscale の 100.x.y.z 経由通信を確実にコンテナ内部に到達させる。

#### 4.2.3 prometheus.yml のスクレイプ定義
[ansible/files/monitoring/prometheus.yml](../iac-workspace/ansible/files/monitoring/prometheus.yml)
```yaml
global:
  scrape_interval: 15s

rule_files:
  - 'prometheus_rules.yml'

alerting:
  alertmanagers:
    - static_configs:
        - targets:
            - localhost:9093

scrape_configs:
  - job_name: 'node_exporter'
    static_configs:
      - targets:
          - '100.92.253.28:9100'    # negitoroserver
          - '100.116.167.59:9100'   # shakeserver
          - 'localhost:9100'         # tarakoserver

  - job_name: 'blackbox'
    metrics_path: /probe
    params:
      module: [http_2xx]
    static_configs:
      - targets:
          - http://100.116.167.59:80
    relabel_configs:
      - source_labels: [__address__]
        target_label: __param_target
      - target_label: __address__
        replacement: 'localhost:9115'

  - job_name: 'ups_exporter'
    static_configs:
      - targets:
          - '100.116.167.59:9101'

  - job_name: 'cadvisor'
    static_configs:
      - targets:
          - '100.116.167.59:8090'
          - 'localhost:8090'
```

ターゲットは Tailscale IP で固定している。理由は (1) Tailscale 経由でしかメトリクス取得を許可しない UFW 設定と整合させるため、(2) ノード再起動でも IP が変わらないため。

#### 4.2.4 Loki / Promtail によるログ集約
Prometheus は数値メトリクスのみ扱うため、各ノードの `/var/log/` 配下と Docker コンテナログを Loki に集約する。

Promtail 設定は Jinja2 テンプレート [promtail-config.yml.j2](../iac-workspace/ansible/roles/monitoring/templates/promtail-config.yml.j2) でホスト名 (`{{ inventory_hostname }}`) を自動注入するため、ノードごとに個別設定は不要。

主なラベル:
| ラベル | 値の例 | 用途 |
|---|---|---|
| `job` | `varlogs` | `/var/log/` 配下のシステムログ全般 |
| `job` | `docker` | Docker コンテナのログ全般 |
| `compose_service` | `prometheus`, `grafana` など | 特定のコンテナに絞り込む |
| `host` | `tarakoserver` | ノードで絞り込む（複数ノード時） |

#### 4.2.5 Grafana 初期設定
1. ブラウザで Grafana `http://100.X.X.3:3000` にアクセスし、初期 ID/PW: `admin/admin` でログイン。
2. プロフィール > 言語を「日本語」に変更。
3. Data Sources で Prometheus と Loki を追加（URL は `http://localhost:9090` / `http://localhost:3100`）。
4. Node Exporter ダッシュボード ID `1860` をインポート。
5. cAdvisor ダッシュボード ID `14282`（または `179`）をインポート。
6. UPS 監視用パネルを新規作成し、PromQL で `ups_battery_capacity_percent` / `ups_external_power_status` / `ups_voltage_volts` を表示。

### 4.3 通知層: Alertmanager + Discord Webhook + ライフサイクル監視

#### 4.3.1 prometheus_rules.yml で定義済みアラート
[ansible/files/monitoring/prometheus_rules.yml](../iac-workspace/ansible/files/monitoring/prometheus_rules.yml)

| アラート | 条件 | 発火待機 | 重要度 |
|---|---|---|---|
| `ProxyDown` | blackbox の `probe_success == 0`（プロキシへの HTTP 疎通失敗） | 30 秒 | critical |
| `NodeDown` | `up{job="node_exporter"} == 0` | 1 分 | critical |
| `HighMemoryUsage` | メモリ使用率 > 90% | 5 分 | warning |
| `HighDiskUsage` | ルートパーティション使用率 > 85% | 5 分 | warning |
| `GithubRunnerOffline` | tarakoserver への blackbox probe 失敗 | 5 分 | warning |

`ProxyDown` は 30 秒という極めて短い発火待機としている。これは [06_高可用性とシステム堅牢化.md](06_高可用性とシステム堅牢化.md) で扱う**自動フェイルオーバー**のトリガーとして利用されるため、復旧待ち時間を最小化するため。

#### 4.3.2 alertmanager.yml: アラートのルーティング先
[ansible/files/monitoring/alertmanager.yml](../iac-workspace/ansible/files/monitoring/alertmanager.yml)
```yaml
global:
  resolve_timeout: 1m

route:
  receiver: 'failover_webhook'
  group_by: ['alertname']
  group_wait: 10s
  group_interval: 10s
  repeat_interval: 1h

receivers:
  - name: 'failover_webhook'
    webhook_configs:
      - url: 'http://127.0.0.1:9000/webhook'
```

すべてのアラートを単一の Webhook（`failover_webhook.py`）に集約する設計。Webhook 側でアラート名・状態（firing/resolved）に応じて分岐し、Discord 通知やフェイルオーバー切替を行う。

#### 4.3.3 failover_webhook.py の役割
Alertmanager から受信した HTTP POST を解析し、Python ロジックで以下を実行する。

| アラート名 | firing 時の動作 | resolved 時の動作 |
|---|---|---|
| `ProxyDown` | `toggle_maintenance.sh on` を実行（→06章で詳述） + Discord 通知 | `toggle_maintenance.sh off` を実行 + Discord 通知 |
| その他 | Discord に色分け Embed メッセージ送信 | Discord に解決通知 |

実装は [ansible/files/monitoring/failover_webhook.py](../iac-workspace/ansible/files/monitoring/failover_webhook.py)、systemd 常駐化は [failover-webhook.service.j2](../iac-workspace/ansible/roles/monitoring/templates/failover-webhook.service.j2) で行う。

#### 4.3.4 ライフサイクル監視（OS 起動・停止の Discord 通知）
ホスト OS の予期せぬシャットダウン（停電・電源ケーブル誤脱落など）を検知し、復旧起動時に自動で警告通知する。Ansible のトリガーに依存せず、**systemd のサービスライフサイクルに同期**させることで 100% 確実な検知を実現する。

仕組み:
- 通常停止時は `/var/lib/lifecycle/normal_shutdown` というマーカーファイルを生成
- 起動時にこのファイルがあれば「正常起動」、なければ「予期せぬ起動」と判定して Discord に通知

通知スクリプトと systemd サービスは common ロールが全ノードに配置する（→01章の common ロール）。

[ansible/templates/common/server-lifecycle-notify.sh.j2](../iac-workspace/ansible/templates/common/server-lifecycle-notify.sh.j2)
```bash
#!/bin/bash
WEBHOOK_URL="{{ discord_webhook_url }}"
HOSTNAME="$(hostname)"
MARK_FILE="/var/lib/lifecycle/normal_shutdown"

if [ -z "$WEBHOOK_URL" ] || [ "$WEBHOOK_URL" = "https://discordapp.com/api/webhooks/YOUR_DUMMY_WEBHOOK_URL" ]; then
    exit 0
fi

# (Discord 送信関数とイベント分岐は実ファイル参照)
case "$1" in
    start)
        if [ -f "$MARK_FILE" ]; then
            # 正常起動通知（緑）
            rm -f "$MARK_FILE"
        else
            # 予期せぬ起動通知（赤）
        fi
        ;;
    stop)
        mkdir -p "$(dirname "$MARK_FILE")"
        touch "$MARK_FILE"
        # 安全停止通知（橙）
        ;;
esac
```

[ansible/templates/common/server-lifecycle.service.j2](../iac-workspace/ansible/templates/common/server-lifecycle.service.j2)
```ini
[Unit]
Description=Server Lifecycle Discord Notification Service
After=network-online.target tailscaled.service
Wants=network-online.target tailscaled.service

[Service]
Type=oneshot
RemainAfterExit=yes
ExecStart=/usr/local/bin/server-lifecycle-notify.sh start
ExecStop=/usr/local/bin/server-lifecycle-notify.sh stop

[Install]
WantedBy=multi-user.target
```

`After=network-online.target tailscaled.service` により、ネットワークと Tailscale が完全に開通した後に Discord 送信が試行される。

#### 4.3.5 コンテナ死活監視（バックアップスクリプト内）
ubsleepy_backup ロールが毎日キックするバックアップスクリプト [backup_ubsleepy_r2.sh.j2](../iac-workspace/ansible/roles/ubsleepy_backup/templates/backup_ubsleepy_r2.sh.j2) は、バックアップ前に `docker inspect` で対象 Bot コンテナ（ubsleepy_bot）の `.State.Running` をチェックし、停止検知時に Discord へ橙色アラートを送信する。詳細は [05_アプリのコンテナ化と運用.md](05_アプリのコンテナ化と運用.md) を参照。

権限注意: systemd タイマー経由で起動するバックアップサービスは Docker ソケット（`/var/run/docker.sock`）の読取権限が必要なため、サービスの `Group=docker` を明示する。

### 4.5 実ユーザー経路の外形監視と Discord 通知ルート

#### 4.5.1 公開 URL(HTTPS) の外形監視（blackbox_https）
内部メトリクスとは別に、利用者と同じ経路（Cloudflare → プロキシ → Nginx → アプリ → DB）で公開エンドポイントを HTTP プローブし、実際に 200 を返すかを監視する。フェイルオーバー判定に使う `ProxyDown`（job: blackbox）と区別するため、専用のジョブ名 `blackbox_https` を用いる。これにより、この外形監視の失敗がメンテナンス自動切替を誤発火させることはない。

iac-workspace/ansible/files/monitoring/prometheus.yml
```yaml
  - job_name: 'blackbox_https'
    metrics_path: /probe
    params:
      module: [http_2xx]
    static_configs:
      - targets:
          - https://ruruthegeek.dpdns.org/quiz/api/bsquiz/pokedex
          - https://ruruthegeek.dpdns.org/
    relabel_configs:
      - source_labels: [__address__]
        target_label: __param_target
      - source_labels: [__param_target]
        target_label: instance
      - target_label: __address__
        replacement: 'localhost:9115'
```
DB 連携エンドポイント（/quiz/api/bsquiz/pokedex）を対象に含めることで、Web は表示できるが DB 認証に失敗している、といった内部障害も検知できる。ポート 80 のリダイレクトのみを見る監視では見逃す障害を捕捉する目的である。

#### 4.5.2 Web/DB のアラートルール
iac-workspace/ansible/files/monitoring/prometheus_rules.yml
```yaml
  - name: web.rules
    rules:
      - alert: WebDbEndpointDown
        expr: probe_success{job="blackbox_https", instance=~".*/quiz/api/bsquiz/pokedex"} == 0
        for: 2m
        labels:
          severity: critical
      - alert: WebSiteDown
        expr: probe_success{job="blackbox_https", instance="https://ruruthegeek.dpdns.org/"} == 0
        for: 2m
        labels:
          severity: critical
```

#### 4.5.3 Alertmanager の通知ルーティング
Alertmanager はアラートの宛先をルーティングで決定する。ProxyDown はフェイルオーバー Webhook（メンテ自動切替）へ送り、続けて Discord にも通知する。その他のアラート（WebDbEndpointDown / WebSiteDown / NodeDown 等）は Discord へ通知する。Discord の宛先 URL は Vault の discord_webhook_url を注入するため、設定はテンプレート化している（Alertmanager 0.25 以降の discord_configs を利用）。設定は amtool check-config で検証できる。

iac-workspace/ansible/roles/monitoring/templates/alertmanager.yml.j2
```yaml
route:
  receiver: 'discord'
  routes:
    - matchers:
        - 'alertname = "ProxyDown"'
      receiver: 'failover_webhook'
      continue: true
    - matchers:
        - 'alertname = "ProxyDown"'
      receiver: 'discord'

receivers:
  - name: 'failover_webhook'
    webhook_configs:
      - url: 'http://127.0.0.1:9000/webhook'
  - name: 'discord'
    discord_configs:
      - webhook_url: '{{ discord_webhook_url }}'
        send_resolved: true
```
