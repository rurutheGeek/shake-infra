# 8. CD（自動デプロイ）

## CD の目的
アプリリポジトリ（ubsleepy, shake-web など）の main ブランチに変更が merge されたタイミングで、本インフラリポジトリ上のセルフホストランナーが**自動で対象サービスを Ansible で再デプロイ**する仕組みを構築する。
人手のデプロイ作業をゼロにし、コードマージから本番反映までを Pull Request 単位で完結させる。

### 全体アーキテクチャ
```text
[ アプリリポジトリ (ubsleepy 等) ]                  [ インフラリポジトリ (shake-infra) ]
        │                                                       │
        │ PR を main にマージ                                    │
        ├─ deploy.yml workflow が発火                            │
        │  └─ peter-evans/repository-dispatch でシグナル送信     │
        │     (event-type: deploy_ubsleepy)                      │
        │       │                                                │
        │       ▼                                                │
        │     INFRA_REPO_DISPATCH_TOKEN を使って GitHub API ─────▶ shake-infra に dispatch を受信
        │                                                       │
        │                              cd_deploy.yml workflow が発火
        │                                  │
        │                                  ▼ Self-hosted runner (tarakoserver) で実行
        │                              ┌─────────────────────────────────────┐
        │                              │ 1. Checkout repo                    │
        │                              │ 2. Vault/SSH 鍵を Secrets から復元   │
        │                              │ 3. ansible-galaxy collection install │
        │                              │ 4. ansible-playbook --tags <app>    │
        │                              │ 5. Discord 通知                     │
        │                              └─────────────────────────────────────┘
        │                                  │
        │                                  ▼ Ansible が各ノードに SSH 接続して反映
        │                              [ shakeserver / tarakoserver ]
```

### この章で扱うファイル
```text
infra
├── .github/workflows
│   └── cd_deploy.yml *                             # CD ワークフロー定義
└── iac-workspace
    └── ansible
        ├── site.yml                                # Phase 4 の strategy:free による並列実行
        └── roles
            └── github_runner *
                ├── defaults/main.yml *
                ├── handlers/main.yml *
                ├── tasks/main.yml *
                └── templates/
                    ├── Dockerfile.j2 *
                    ├── docker-compose.yml.j2 *
                    └── entrypoint.sh.j2 *
```

### 利用可能になる機能
- アプリリポジトリで PR をマージすると本番反映が自動実行
- 統合管理スクリプト経由でランナー手動セットアップ可能（`./run_playbook.sh` → [2][8]）
- 全デプロイ結果が Discord に成否通知される

### 8.1 セルフホストランナー（tarakoserver）

#### 8.1.1 採用目的
GitHub の個人アカウントでは Organization runner が使えず**リポジトリ単位**でしか runner を登録できない。さらに、デプロイ対象が Tailscale VPN 配下にあるため、GitHub-hosted runner（パブリッククラウド）では到達できない。したがって、ノード内（tarakoserver）に runner を常駐させる。

#### 8.1.2 ランナー構成
| 項目 | 値 |
|---|---|
| ホスト | tarakoserver（home_node2） |
| 配置形式 | Docker コンテナ（`github_runner` ロール経由） |
| 接続方式 | GitHub API（OUT のみ。ポート開放不要） |
| ホスト OS への影響 | Docker socket と Ansible 実行のため非 root で稼働 |
| 監視 | ランナーホスト(tarakoserver)の死活は `NodeDown`（node_exporter）でカバー。ランナーサービス単体の監視には専用エクスポーターが必要なため未導入 |

#### 8.1.3 github_runner ロール
[ansible/roles/github_runner/tasks/main.yml](../iac-workspace/ansible/roles/github_runner/tasks/main.yml)（要点）
```yaml
- name: Runnerのディレクトリを作成
  become: false
  ansible.builtin.file:
    path: "{{ github_runner_dir }}"
    state: directory
    mode: "0755"

- name: Dockerfileテンプレートの配置
  become: false
  ansible.builtin.template:
    src: Dockerfile.j2
    dest: "{{ github_runner_dir }}/Dockerfile"

- name: エントリーポイントスクリプトテンプレートの配置
  become: false
  ansible.builtin.template:
    src: entrypoint.sh.j2
    dest: "{{ github_runner_dir }}/entrypoint.sh"
    mode: "0755"

- name: Docker Composeテンプレートの配置
  become: false
  ansible.builtin.template:
    src: docker-compose.yml.j2
    dest: "{{ github_runner_dir }}/docker-compose.yml"

- name: ランナーコンテナのビルドおよび起動
  become: false
  community.docker.docker_compose_v2:
    project_src: "{{ github_runner_dir }}"
    state: present
    build: always
```

すべてのタスクが `become: false` で実行される。非 root の `ansible_user` で Docker を扱う前提（`ansible_user` は `docker` グループ所属）。

[ansible/roles/github_runner/defaults/main.yml](../iac-workspace/ansible/roles/github_runner/defaults/main.yml)
```yaml
---
github_runner_version: "2.316.1"
github_runner_arch: "arm64"
github_runner_dir: "/home/{{ ansible_user }}/actions-runner"
# 以下は実行時に --extra-vars で渡すか、Vaultで定義する
# github_runner_token: ""
# github_repo_url: ""
```

#### 8.1.4 ランナーのセットアップ手順

##### ステップ 1: GitHub からワンタイムトークン取得
1. GitHub のインフラリポジトリ（shake-infra）を開く
2. Settings > Actions > Runners へ移動
3. New self-hosted runner をクリック
4. OS に Linux、Architecture に ARM64 を選択
5. 画面に表示される `./config.sh --url ... --token XXXXXXXXXXXXXXXXX` の **トークン部分のみ** をメモする

##### ステップ 2: Ansible 経由でデプロイ
```bash
./run_playbook.sh
```
→ メニューから「[2] デプロイ反映フェーズ」→「[8] GitHub Runner (CI/CD) のセットアップ」を選択
→ プロンプトに従い、インフラリポジトリ URL とトークンを入力

または手動で:
```bash
ansible-playbook -i local_config/ansible/inventory.ini \
  -e @local_config/ansible/vars.yml \
  -e @local_config/ansible/credentials/vault.yml \
  ansible/site.yml --tags github_runner \
  -e "github_runner_token=取得したトークン" \
  -e "github_repo_url=https://github.com/rurutheGeek/shake-infra"
```

#### 8.1.5 グレースフルスキップ機構
通常のフルデプロイ（`./run_playbook.sh` → [2][1]）では `github_runner_token` と `github_repo_url` が未定義のため、site.yml の `when:` 条件で自動スキップされる。

[ansible/site.yml](../iac-workspace/ansible/site.yml)（抜粋）
```yaml
- name: 監視サーバーへのGitHub Runnerのセットアップ
  ansible.builtin.include_role:
    name: github_runner
  when:
    - inventory_hostname in groups['home_node2']
    - github_repo_url is defined
    - github_runner_token is defined
  tags: github_runner
```

| 実行方法 | 動作 |
|---|---|
| `./run_playbook.sh` → `[2][1]`（フルデプロイ） | github_runner をスキップ |
| `./run_playbook.sh` → `[2][8]`（Runner セットアップ） | トークンを対話的に入力して実行 |

### 8.2 Repository Dispatch によるアプリ→インフラの連携

#### 8.2.1 採用目的
個人アカウントの制約上、アプリリポジトリの runner と インフラリポジトリの runner を共有できない。**Repository Dispatch（リポジトリ間 API シグナル）** を使い、アプリ側 workflow からインフラ側 workflow を起動する。

#### 8.2.2 アプリ側 workflow の例
アプリリポジトリ（例: ubsleepy）の `.github/workflows/deploy.yml`:
```yaml
name: Trigger Deploy

on:
  push:
    branches: [main]

jobs:
  dispatch:
    runs-on: ubuntu-latest
    steps:
      - name: Trigger Infrastructure Deployment
        uses: peter-evans/repository-dispatch@v2
        with:
          token: ${{ secrets.INFRA_REPO_DISPATCH_TOKEN }}
          repository: rurutheGeek/shake-infra
          event-type: deploy_ubsleepy
```

`event-type` はアプリごとに変える（`deploy_ubsleepy`, `deploy_shakeweb`）。これによりインフラ側で「どのアプリを反映するか」を判別できる。

#### 8.2.3 INFRA_REPO_DISPATCH_TOKEN の発行と配布
アプリ側に必要な PAT。発行手順と Terraform GitHub Provider による自動登録は [03_機密情報管理.md](03_機密情報管理.md) §3.2 を参照。

repository_dispatch エンドポイントは、対象リポジトリ（shake-infra）への Contents 書き込み権限を要求する。Fine-grained PAT を使う場合は、対象リポジトリへのアクセスに加え Repository permissions の Contents を Read and write に設定すること。権限が不足すると、dispatch 送信時に HTTP 403「Resource not accessible by personal access token」となり、デプロイが起動しない。トークンの配布自体は Terraform で管理しており、shake-web リポジトリにも同名 Secret を登録している。

### 8.3 インフラ側 CD ワークフロー

#### 8.3.1 cd_deploy.yml の動作フロー
[.github/workflows/cd_deploy.yml](../.github/workflows/cd_deploy.yml) は以下のトリガーで実行される。

```yaml
on:
  workflow_dispatch:
    inputs:
      target:
        description: 'Target application to deploy'
        type: choice
        options:
          - ubsleepy
          - shakeweb
  repository_dispatch:
    types:
      - deploy_ubsleepy
      - deploy_shakeweb
```

- `repository_dispatch`: アプリ側からの自動発火（通常運用）
- `workflow_dispatch`: GitHub UI から手動実行（緊急時の手動デプロイ）

> 上記は要点の抜粋。実際の `target` 選択肢は `ubsleepy / shakeweb / pkhack / ayahuya / monitoring` を持つ。`monitoring` は監視スタック（Prometheus / Alertmanager / blackbox など）を `--tags monitoring --limit home_node2` で tarakoserver に反映し、デプロイ後に Prometheus の blackbox ターゲットがリバースプロキシ negitoroserver を正しく向いているか（= ProxyDown フェイルセーフが機能する構成か）を検証する。アプリと違いアプリリポジトリからの自動発火は通常起きないため、主に手動実行用。

#### 8.3.2 デプロイステップ
```yaml
jobs:
  deploy:
    runs-on: self-hosted
    env:
      ANSIBLE_HOST_KEY_CHECKING: "False"
      ANSIBLE_PRIVATE_KEY_FILE: "${{ github.workspace }}/.ansible_pkey"
      ANSIBLE_VAULT_PASS_FILE: "${{ github.workspace }}/.ansible_vault_pass"
    steps:
      - name: Checkout Repo
        uses: actions/checkout@v3

      - name: Setup Ansible Credentials
        env:
          ANSIBLE_SSH_KEY: ${{ secrets.ANSIBLE_SSH_KEY }}
          ANSIBLE_VAULT_PASS: ${{ secrets.ANSIBLE_VAULT_PASS }}
        run: |
          echo "$ANSIBLE_SSH_KEY" > "${{ env.ANSIBLE_PRIVATE_KEY_FILE }}"
          chmod 600 "${{ env.ANSIBLE_PRIVATE_KEY_FILE }}"
          echo "$ANSIBLE_VAULT_PASS" > "${{ env.ANSIBLE_VAULT_PASS_FILE }}"
          chmod 600 "${{ env.ANSIBLE_VAULT_PASS_FILE }}"

      - name: Install Ansible Collections
        run: |
          cd iac-workspace/ansible
          ansible-galaxy collection install -r requirements.yml --force

      - name: Deploy ubsleepy
        if: github.event.action == 'deploy_ubsleepy' || (github.event_name == 'workflow_dispatch' && github.event.inputs.target == 'ubsleepy')
        run: |
          cd iac-workspace
          ansible-playbook -i ansible/inventory.ini ansible/site.yml --tags ubsleepy \
            --private-key="${{ env.ANSIBLE_PRIVATE_KEY_FILE }}" \
            --vault-password-file="${{ env.ANSIBLE_VAULT_PASS_FILE }}"

      - name: Deploy shake-web
        if: github.event.action == 'deploy_shakeweb' || (github.event_name == 'workflow_dispatch' && github.event.inputs.target == 'shakeweb')
        run: |
          cd iac-workspace
          ansible-playbook -i ansible/inventory.ini ansible/site.yml --tags web \
            --private-key="${{ env.ANSIBLE_PRIVATE_KEY_FILE }}" \
            --vault-password-file="${{ env.ANSIBLE_VAULT_PASS_FILE }}"

      - name: Cleanup Credentials
        if: always()
        run: |
          rm -f "${{ env.ANSIBLE_PRIVATE_KEY_FILE }}"
          rm -f "${{ env.ANSIBLE_VAULT_PASS_FILE }}"
```

#### 8.3.3 機密情報の取り扱い
ワークフロー実行時、必要な機密は GitHub Actions Secrets から `${{ env.* }}` に書き出し、`if: always()` で必ず削除する設計。

| Secret | 用途 |
|---|---|
| `ANSIBLE_SSH_KEY` | ターゲットノードへの SSH 接続用 id_rsa |
| `ANSIBLE_VAULT_PASS` | Vault 復号パスフレーズ |
| `DISCORD_WEBHOOK_URL` | デプロイ結果の通知先 |

これらは [03_機密情報管理.md](03_機密情報管理.md) §3.2 の Terraform GitHub Provider で**自動登録**されており、手動設定は不要。

#### 8.3.4 Discord 結果通知
成功時・失敗時のそれぞれで Discord に通知を送る。

```yaml
- name: Discord Notification (Success)
  if: success()
  run: |
    TARGET="${{ github.event.action }}"
    if [ "${{ github.event_name }}" = "workflow_dispatch" ]; then
      TARGET="manual-${{ github.event.inputs.target }}"
    fi
    if [ -n "${{ secrets.DISCORD_WEBHOOK_URL }}" ]; then
      curl -H "Content-Type: application/json" -X POST \
           -d '{"content": "[SUCCESS] デプロイ成功\n対象: `'"$TARGET"'`\n..."}' \
           "${{ secrets.DISCORD_WEBHOOK_URL }}"
    fi
```

失敗時は同様の構造で `[FAILED]` メッセージと PR の Actions ログ URL を含む形で送る。

### 8.4 Ansible 並列実行アーキテクチャ (strategy: free)

#### 8.4.1 採用目的
[ansible/site.yml](../iac-workspace/ansible/site.yml) のサービス展開フェーズ（Phase 4）は `strategy: free` を採用し、home_node1 と home_node2 が**互いを待たずに独立して並列実行**される。同期待ちのデッドタイムを削減し、CD のターンアラウンドタイムを短縮する。

```text
Phase 1: common          全3ノード並列
Phase 2: proxy           negitoroserver のみ
         exporters       全3ノード並列
Phase 3: docker          home_node1 / home_node2 並列
Phase 4: サービス展開    home_node1 / home_node2 を strategy:free で並列実行
         ┌─ home_node1: ups_exporter → postgres → minecraft → web → ubsleepy → ubsleepy_backup
         └─ home_node2: monitoring → github_runner
          （docker完了後、両ノードが互いを待たずに独立して進む）
```

#### 8.4.2 実装方式: include_role + when ガード
`strategy: free` では Play の `roles:` キーワードが使えない。そのため Phase 4 では `tasks:` ブロック内で `ansible.builtin.include_role` を使い、`when: inventory_hostname in groups['home_node1']` でホストガードする。

```yaml
- name: 各ノードへのサービス展開（並列実行）
  hosts: home_node1, home_node2
  become: true
  strategy: free
  tasks:
    - name: メインサーバーのWebアプリケーションの構成
      ansible.builtin.include_role:
        name: web
      when: inventory_hostname in groups['home_node1']
      tags: web

    - name: 監視サーバーへのGitHub Runnerのセットアップ
      ansible.builtin.include_role:
        name: github_runner
      when:
        - inventory_hostname in groups['home_node2']
        - github_repo_url is defined
        - github_runner_token is defined
      tags: github_runner
```

home_node2 にとって `web` ロールはホストガードで即スキップされるため、自分の monitoring/github_runner を待たずに進める。

#### 8.4.3 strategy: free 環境での制約
| 機能 | strategy: free での挙動 |
|---|---|
| `ansible.builtin.pause` | 使えない（並列実行の同期点を壊すため） |
| `ansible.builtin.fail` | 代替として使用可能（→[05章 §5.3.3](05_アプリのコンテナ化と運用.md) Web ロールでの GitHub Deploy Key 警告） |
| `serial:` | 効かない |
| `notify` ハンドラー | 各ホスト独立で実行される |

### 8.5 デプロイ後スモークテスト

デプロイが「成功」（Ansible が changed/ok で完了）しても、アプリが実際に動いているとは限らない。そこで shake-web のデプロイ直後に、実ユーザー経路（Cloudflare → プロキシ → Nginx → quiz → PostgreSQL）で DB 連携エンドポイントを叩き、200 を返さなければデプロイを失敗扱いにする。これにより「デプロイ成功なのに DB 認証ズレでページが 500」といった事象をデプロイ時点で検知し、失敗通知へつなげる。

iac-workspace/.github/workflows/cd_deploy.yml（shake-web デプロイ直後のステップ）
```yaml
      - name: スモークテスト（shake-web / DB連携エンドポイント）
        if: github.event.action == 'deploy_shakeweb' || (github.event_name == 'workflow_dispatch' && github.event.inputs.target == 'shakeweb')
        run: |
          if [ "${{ github.event.inputs.dry_run }}" = "true" ]; then
            echo "dry-run のためスモークテストをスキップ"; exit 0
          fi
          BASE="https://ruruthegeek.dpdns.org"
          ok=0
          for i in $(seq 1 12); do
            code=$(curl -s -o /dev/null -w "%{http_code}" "$BASE/quiz/api/bsquiz/pokedex" || echo 000)
            [ "$code" = "200" ] && ok=1 && break
            sleep 5
          done
          [ "$ok" = "1" ] || { echo "::error::Smoke test FAILED"; exit 1; }
```
コンテナ起動直後の遅延を考慮し、最大 12 回（約 60 秒）リトライする。dry_run（シミュレーション）時はスキップする。同等の検証はローカルでも `iac-workspace/tests/` のテストで再現できる（[07_CI（コード品質と静的検証）.md](07_CI（コード品質と静的検証）.md) §7.7）。
