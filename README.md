# shake インフラ基盤構成管理リポジトリ

本リポジトリは、複数台の Raspberry Pi を用いて構成される自宅サーバー環境（shake基盤）のインフラ構築、構成管理、および各種コンテナアプリケーションのデプロイを自動化するための IaC（Infrastructure as Code）資材一式を管理するものです。

---

## システム構成概要

本インフラ基盤は、以下の主要なコンポーネントで構成されています。

### ネットワークおよびプロビジョニング
*   **VPNネットワーク**: Tailscale を用いて各ノード（shakeserver, tarakoserver, negitoroserver）間をプライベートVPNで接続します。
*   **インフラプロビジョニング**: Terraform を用いて Cloudflare 上の DNS レコード設定およびバックアップ・データ保管用の R2 バケット（shakeserver-backup, ubsleepy-app-data）を自動構築します。

### 構成管理およびアプリケーションデプロイ
Ansible プレイブックを用いて、各ホストの設定、UFWファイアウォールの設定、および以下のDockerコンテナサービスを展開・制御します（`ansible-lint` 準拠のモダンな実装を適用済）。
*   **データベース**: PostgreSQL 15（データ初期設定スクリプト自動適用）および管理ツール Adminer。
*   **ゲームサーバー**: Minecraft Reforged（ボリュームマウントによるデータ永続化）。
*   **Discord Bot (ubsleepy)**: Bot アプリケーション本体のコンテナデプロイ。`unarchive` での日本語ファイル名（cp932）対応、および `stat` モジュールを用いた稼働データの逆流防止制御を実装。
*   **ホストOSのSWAP無効化**: マイクロSDカードおよびSSDのセル摩耗（寿命低下）を防ぎ、ランダムI/Oのボトルネックを排除するため、全ホストの SWAP（スワップ）機能を完全に無効化します。

### 監視・自動化およびバックアップ
*   **自動バックアップ**: Systemd Service および Systemd Timer を連携させ、毎日深夜帯にコンテナ停止状態を確認のうえデータを圧縮し、Cloudflare R2（S3互換）へ自動転送します。
*   **エラーアラート**: バックアップ結果（正常、圧縮エラー、アップロードエラー、コンテナ異常停止）を判別し、Discord Webhook 経由で Embed メッセージを送信します。
*   **稼働監視（Prometheus/Grafana）**: 各ノードの Node Exporter、Dockerコンテナのリソース状態（cAdvisor / 8090ポートマッピング）、および Raspberry Pi 5 / Geekworm X120X UPS ボード用カスタム Python Exporter からメトリクスを収集し可視化します。

## クイックスタート手順

### 1. Terraform の適用
`iac-workspace/terraform/` に移動し、 `terraform.tfvars` に Cloudflare 認証情報を設定したうえで実行します。
```bash
cd iac-workspace/terraform
terraform init
terraform apply
```

### 2. Ansible 機密変数の暗号化
`iac-workspace/ansible/group_vars/all/vault.yml` を作成し、Ansible Vault にて暗号化します。
```bash
ansible-vault encrypt iac-workspace/ansible/group_vars/all/vault.yml
```

### 3. Playbook の実行
本プロジェクトの統合ラッパースクリプト `run_playbook.sh` を実行し、直感的な階層型メニューから簡単に対象を選択して実行します。

1. リポジトリの `iac-workspace/` ディレクトリで以下のコマンドを実行します。
   ```bash
   ./run_playbook.sh
   ```
2. 最初に入力プロンプトが表示されるので、実行したい大項目の番号を選択します。
   - [1] テスト・検証フェーズ
   - [2] デプロイ反映フェーズ
   - [3] サーバー電源・ライフサイクル操作
3. 続いて表示されるサブメニューで、詳細なアクションを選択します。

   大項目 [1] テスト・検証フェーズ を選択した場合:
   - [1] 静的解析実行 (ansible-lint による品質確認)
   - [2] 構文チェック (Ansible標準のシンタックスチェック)
   - [3] 模擬実行 (Dry Run / Check Mode)

   大項目 [2] デプロイ反映フェーズ を選択した場合:
   - [1] 本番反映実行 (全サービスの一括展開)
   - [2] Webアプリのみデプロイ
   - [3] DB (PostgreSQL) のみデプロイ
   - [4] Minecraft のみデプロイ
   - [5] Discord Bot のみデプロイ
   - [6] 自動バックアップ のみデプロイ
   - [7] UPS監視 のみデプロイ

   大項目 [3] サーバー電源・ライフサイクル操作 を選択した場合:
   - [1] 安全なシャットダウン実行 (コンテナ停止後に電源OFF)
   - [2] 安全な再起動実行 (コンテナ停止後にOS再起動)

## サーバーの安全なシャットダウン・再起動手順

各サーバー上で稼働するコンテナ（データベース、ゲームサーバー、監視スタック）を自動的に安全停止させたうえで、ホストOSのシャットダウンまたは再起動を実行します。

本プロジェクトの統合ラッパースクリプト `run_playbook.sh` を実行し、階層型メニューから簡単に対象を選択して安全に自動実行可能です。

1. リポジトリの `iac-workspace/` ディレクトリで以下のコマンドを実行します。
   ```bash
   ./run_playbook.sh
   ```
2. メインメニューから [3] サーバー電源・ライフサイクル操作 を選択します。
3. 電源メニューから [1] 安全なシャットダウン実行 または [2] 安全な再起動実行 を選択します。
4. サブメニューで、操作対象を選択します。
   - [1] 全サーバーを一括操作
   - [2] shakeserver (メイン) のみ操作
   - [3] tarakoserver (監視) のみ操作
   - [4] negitoroserver (プロキシ) のみ操作
5. シャットダウン [1] または再起動 [2] を選択した場合は、事故防止のための最終確認プロンプト（シャットダウン時のみ `本当に対象ノードの電源を落としてよろしいですか？ (y/n): ` ）が表示され、 `y` と入力することで初めて実行されます。

これにより、手動でのSSHログインやコマンド入力を行うことなく、データの安全性を保持しながら個別にまたは一括で電源制御を自動実行できます。

### OS再起動後のサービス自動起動仕様
ホストOSが起動すると、以下の自動起動制御により全てのサービスが手動操作なしで自動復旧します。
*   **Dockerコンテナ（PostgreSQL, Minecraft, 監視スタック）:** Ansible 定義時に `restart_policy: always` が設定されているため、Docker デーモン起動時に元の稼働状態へ自動復旧します。
*   **Systemd サービス/タイマー (UPS Exporter, バックアップ timer):** OS 起動時にサービスが自動有効化（enabled）されているため、バックグラウンドで自動的に稼働を再開します。

## サーバー電源状態およびライフサイクルの Discord 自動通知仕様

インフラ基盤の信頼性向上を目的に、Ansible の common ロールを通じて、各ホストOSの電源状態（シャットダウン、再起動、正常起動、および予期せぬシャットダウンからの起動）を自動検知して Discord に通知する Systemd ライフサイクル監視サービス（server-lifecycle.service）が自動適用されます。

### 1. 通知ロジックと動作仕様
本サービスはホストOSの Systemd 起動・停止シーケンスと完全に同期しており、以下のステータスを自動検知して Discord Webhook 経由でリッチな Embed メッセージを送信します。

*   **安全なシャットダウン / 再起動（停止時）**: サーバーが正常な停止処理に入ったことを検知し、 `安全なシャットダウン・再起動`（ステータス：注意）として Discord に通知を送信したうえで、正常終了フラグをローカルディスク（ `/var/lib/lifecycle/normal_shutdown` ）に永続化します。
*   **正常起動（起動時）**: OS 起動時に上記フラグの存在を確認できた場合、 `正常起動`（ステータス：情報）として Discord に通知を送信し、起動完了後にフラグファイルを消去します。
*   **予期せぬ起動（停電・強制終了後の起動）**: OS 起動時に上記フラグファイルが確認できなかった場合、前回のシャットダウン処理が正常に完了しなかった（停電による突然のシャットダウンや強制電源断など）と自動判別し、 `予期せぬ起動`（ステータス：警告）として Discord に即時警告通知を送信します。

### 2. IaC による自動展開
本監視サービスおよび通知スクリプトは、 `run_playbook.sh` の本番反映実行（大項目2 の 小項目1）などを行うことで、全てのサーバー（shakeserver, tarakoserver, negitoroserver）に対して完全に自動でインストール、有効化、および起動処理が行われます。手動での個別設定は一切不要です。
