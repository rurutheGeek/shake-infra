# 2026年5月20日 CI/CDパイプラインと自動デプロイの導入

## 概要
今後の共同開発を見据え、アプリケーション（ubsleepy, shake-webなど）が `main` ブランチにマージされた際に自動でデプロイされる仕組み（CD）を構築するための準備を行った。

## 作業内容
- **設計の決定**: GitHubの個人アカウントの制約を回避するため、「インフラリポジトリに専用のSelf-hosted runner（司令塔）を1つ配置し、アプリリポジトリからRepository Dispatch（APIシグナル）を送ってデプロイを実行させる」方針（方針1）を採用した。
- **Runner配置先の決定**: システムの監視など運用タスクに特化しており、メインサービスへの影響が少ない `tarakoserver`（監視サーバー）をRunnerのホストとして選定した。
- **Ansible Roleの作成**: `tarakoserver` に対して自動でRunnerをインストールし、サービス化するための `github_runner` ロールを作成した。
- **Playbookの更新**: `site.yml` を更新し、`home_node2`（tarakoserver）に `github_runner` ロールを適用した。
- **ドキュメント作成**: 上記の設計方針と、Runnerの登録・実行手順、GitHub Actions側の設定例をまとめた `docs/07_CI-CDパイプラインと自動デプロイ構築手順.md` を作成した。

## 今後の手順（ユーザー作業）
1. インフラリポジトリのGitHub SettingsからRunnerの登録用トークンを取得する。
2. `iac-workspace/run_playbook.sh` を実行し、`[2] デプロイ反映フェーズ` -> `[8] GitHub Runner (CI/CD) のセットアップ` を選択して、対話的にURLとトークンを入力しRunnerをデプロイする。
3. アプリリポジトリおよびインフラリポジトリにGitHub Actionsのワークフロー（`.github/workflows/*.yml`）を設定し、自動デプロイを有効化する。
